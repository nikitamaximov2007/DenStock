# BRP-каталог: импорт дилерского прайса и учёт наличия

Layer 31. Файл-источник: дилерский прайс BRP (Excel, ~127 тысяч строк,
пример: BRP DEALER PRICE 2.10.2026.xlsx). Файл в Git НЕ коммитится.

## 1. Справочник, а не остатки

BRP-прайс: это МАСТЕР-КАТАЛОГ (справочник всех деталей BRP), а не то, что
лежит у Дениса на складе. Поэтому:

- импорт создаёт только строки справочника BrpCatalogPart;
- импорт НЕ создаёт остатков, движений, поступлений и складских карточек;
- на складе появляются только детали, которые физически есть: пользователь
  находит позицию в BRP-каталоге, добавляет её в склад и учитывает наличие
  документом (см. раздел 6).

Если бы все 127k строк стали остатками, склад показывал бы наличие деталей,
которых нет, и все отчёты/статистика врали бы.

## 2. Как запустить импорт

Проверка без записи (обязательный первый шаг):

```bash
docker compose exec web python manage.py import_brp_catalog "/app/brp.xlsx" --dry-run
```

Боевой импорт (только после бэкапа):

```bash
docker compose exec web python manage.py backup_all
docker compose exec web python manage.py import_brp_catalog "/app/brp.xlsx" --commit
```

Файл кладётся на сервер вручную (scp) и не хранится в репозитории.
Импорт идемпотентен: повторный запуск того же файла ничего не меняет
(created=0, updated=0); новый прайс обновит изменившиеся строки по Material_No.

Итог команды: строк просмотрено/данных, создано, обновлено, пропущено,
дубликаты, уникальные номера, счётчики статусов, строки с ценами и заменами.

## 3. Формат файла и поля

- первый лист; строка 1: заголовки; строка 2: примечания; данные со строки 3;
- колонки A..H: Material_No, Part_Desc, Last_Yr_Util, Status, РОЗНИЦА (USD),
  ОПТОВАЯ (USD), ЗАМЕНА НОМЕРА, ЗАМЕНА НОМЕРА; правее: легенда, игнорируется;
- Material_No хранится СТРОКОЙ (ведущие нули не теряются), пробелы обрезаются;
- обе колонки «ЗАМЕНА НОМЕРА» сохраняются отдельно и индексируются для поиска;
- дубликаты Material_No: берётся первая строка, остальные считаются в отчёте.

Статусы BRP (по легенде файла):

- OBS: снято с производства;
- USE: замена номера;
- VIN: винтажный склад, возможна отдельная доставка;
- LIQ: последние остатки у завода;
- UCP: значение не подтверждено, показывается как есть (не выдумываем).

## 4. Цены: формула БЕЗ округления

Цены прайса в долларах. Цена клиента считается из розницы BRP:

    цена_клиента_руб = розница_USD x курс x (1 + наценка_% / 100)

Текущие настройки по умолчанию: курс 105 ₽, наценка 40%.
Примеры: 100 $ -> 14700 ₽; 99.99 $ -> 14698.53 ₽.

- НИКАКОГО округления (ни до 10, ни до 50, ни до 100): копейки сохраняются;
- только Decimal, не float;
- 40%: это НАЦЕНКА (в интерфейсе «Наценка»), не «маржа»;
- курс и наценка настраиваются: страница «Настройки цен» в BRP-каталоге
  (админ/руководитель);
- при добавлении детали в склад использованные курс/наценка, исходные цены
  USD, рассчитанная и итоговая цена фиксируются в связи (BrpPartLink) и при
  смене глобальных настроек задним числом НЕ меняются;
- цену можно указать вручную: исходная USD-цена и рассчитанное значение при
  этом сохраняются (price_source = manual).

## 5. Поиск по каталогу

Раздел «BRP-каталог» в меню (группа «Каталог»). Поиск по номеру BRP, обеим
заменам номера, штрихкоду склада и описанию; фильтр по статусу. Порядок:
сначала фактический склад (с остатками и адресами), затем справочник BRP.
Основной «Поиск детали» тоже подсказывает совпадения из BRP-каталога с
пометкой «справочник, не остаток». Сканер = клавиатура: ввод номера + Enter.

Скорость на реальном объёме (126 399 строк): поиск по номеру ~3 мс, по
замене ~2 мс, по описанию ~16 мс.

## 6. Учёт наличия (первичный ввод остатков)

1. Найти позицию в «BRP-каталог».
2. «Добавить в склад»: создаёт ТОЛЬКО карточку (название из описания BRP,
   номер BRP и замены как номера для поиска, рекомендованная цена =
   рассчитанная цена клиента). Остатков не появляется.
3. «Учесть наличие»: открывает черновик документа «Инвентаризация начальных
   остатков» (это документ поступления Layer 28 от поставщика «Стартовый
   ввод») с уже подставленной деталью: ввести количество, ячейку (адрес),
   себестоимость, комментарий.
4. «Провести поступление»: только теперь появляется остаток в выбранной
   ячейке и движение прихода. Черновик склад не меняет; проведённый документ
   read-only.

Одна деталь может лежать в нескольких адресах: остаток привязан к месту.

Права: смотреть каталог могут все роли; «Добавить в склад»: администратор и
руководитель (can_manage_parts); «Учесть наличие»: роли склада
(can_manage_inventory; кладовщику нужна уже созданная карточка).

## 7. Адресное хранение

Структура: Зона -> Стеллаж -> Уровень -> Ящик/Контейнер -> Ячейка.
Полный адрес записывается в код места хранения:

- B-S01-L02-D03-C08: зона B, стеллаж 1, уровень 2, ящик 3, ячейка 8;
- A-S02-L01-K04-C02: зона A, стеллаж 2, уровень 1, контейнер 4, ячейка 2;
- D-S01-L04: зона D, стеллаж 1, уровень 4 (крупная деталь на полке).

Буквы вида: D = ящик, K = контейнер, X = коробка; полка сегмента не имеет.
Хелпер: apps/warehouse/addresses.py (compose_address, get_or_create_location).

Рекомендации: мелочь (сальники, подшипники, крепёж, датчики): в ящики/ячейки;
среднее (ремни, комплекты прокладок, шланги, пластик): в контейнеры; крупное
(вариаторы, стартеры, генераторы, помпы): на полки. Физическое размещение -
по удобству склада, а не по виду техники: совместимость хранится в карточке
(вид/марка/модель техники), потому что одна деталь подходит нескольким.

## 8. Правила безопасности

- перед боевым импортом: свежий локальный бэкап + offsite + verify_backup;
- сначала dry-run, только потом --commit;
- импорт не трогает вручную отредактированные складские карточки и их цены
  (обновляется только справочник BrpCatalogPart);
- фактические остатки меняются только проведением документа;
- зафиксированные в истории курс/наценка не меняются при смене настроек.
