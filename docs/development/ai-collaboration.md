# Безопасная совместная работа Claude Code и Codex

Протокол передачи работы между AI-агентами в DenisStock. Правила для самих
агентов лежат в корне репозитория: [AGENTS.md](../../AGENTS.md). Шаблоны
промптов: [codex-handoff-template.md](codex-handoff-template.md) и
[claude-handoff-template.md](claude-handoff-template.md).

## A. Почему нужен протокол

- Claude Code регулярно упирается в лимиты посреди задачи: работа обрывается
  на полуслове, часть файлов изменена, тесты не дописаны.
- Codex может продолжить эту работу, но без протокола второй агент не знает,
  что уже сделано: агенты расходятся в архитектуре, дублируют реализацию,
  ломают миграции или переписывают друг друга с нуля.
- Граница безопасности: git-чекпоинт. Всё, что зафиксировано коммитом на
  отдельной ветке, второй агент может прочитать, понять и продолжить.
  Всё, что осталось незакоммиченным, при передаче теряется или ломается.

## B. Правило стабильного main

- В `main` только проверенный, готовый к деплою код: тесты пройдены,
  ruff/djlint/check чистые.
- Незавершённая работа (WIP) коммитится на ветке `wip/*` или feature-ветке,
  никогда напрямую в `main`.
- Продакшен-деплой выполняется только из `origin/main` после успешных
  проверок (см. раздел H).

## C. Два разрешённых режима

### Режим 1: Takeover (передача эстафеты)

Когда использовать: Claude начал задачу и упёрся в лимит; Codex доделывает.

1. Пользователь проверяет текущее состояние:
   ```
   git status --short
   git diff --stat
   ```
2. Пользователь создаёт WIP-ветку:
   ```
   git switch -c wip/layer-XX-claude-handoff
   ```
3. Пользователь коммитит текущую работу Claude:
   ```
   git add -A
   git commit -m "WIP Layer XX from Claude before Codex handoff"
   ```
   (`.claude/`, `.env*`, дампы и xlsx игнорируются через .gitignore,
   поэтому `git add -A` безопасен.)
4. Пользователь даёт Codex промпт из
   [codex-handoff-template.md](codex-handoff-template.md).
5. Codex продолжает от WIP-коммита. НЕ начинает задачу заново,
   НЕ создаёт конкурирующую реализацию.
6. Codex дописывает работу, прогоняет проверки (раздел G) и коммитит.
7. Только после зелёных проверок ветка вливается в `main`
   (merge или fast-forward).

### Режим 2: Параллельная безопасная работа

Когда использовать: только для НЕЗАВИСИМЫХ задач.

- Claude и Codex работают на разных ветках или в разных git worktree.
- Они не редактируют одни и те же файлы, если это явно не скоординировано
  пользователем.
- Примеры безопасного разделения:
  - Claude: backend/сервисы/миграции; Codex: документация/тесты/smoke-проверки.
  - Claude: реализация фичи; Codex: браузерное QA и багфиксы на отдельной
    ветке ПОСЛЕ того, как ветка фичи готова.
- Запрещено: оба агента одновременно и независимо реализуют одну и ту же
  фичу. Это гарантированный конфликт и двойная архитектура.

## D. Worktree для параллельной работы

```
git switch main
git pull --ff-only origin main
git worktree add ../denstock-codex -b codex/layer-XX-task main
```

- Claude работает в основной папке (локально) или в `/opt/denstock`.
- Codex работает в соседней папке `../denstock-codex`.
- У каждого worktree своя ветка: физически невозможно затереть чужие
  незакоммиченные правки.
- Слияние в `main` только после прохождения тестов.
- Убрать worktree после слияния: `git worktree remove ../denstock-codex`.

## E. Безопасность WIP-коммитов

- WIP-коммиты разрешены на ветках `wip/*` (и feature-ветках).
- WIP-коммит не обязан быть идеальным: тесты могут падать, код может быть
  недописан. Его задача: сохранить текущую работу так, чтобы другой агент
  мог безопасно продолжить.
- В сообщении коммита обязательно слово WIP и номер слоя/задачи.
- Запрещено: пушить сломанный WIP в `main` (напрямую или merge'ем без
  проверок).

## F. Файл передачи (handoff)

Для каждого нетривиального слоя активный агент создаёт или обновляет:

```
docs/development/current-handoff.md
```

Содержимое:

- название задачи;
- ветка;
- текущий коммит;
- что завершено;
- что НЕ завершено;
- затронутые файлы;
- принятые допущения;
- известные риски;
- следующие точные шаги;
- уже выполненные команды;
- команды, которые ещё нужно выполнить.

Этот файл: единственная точка правды при передаче. Второй агент читает его
ПЕРЕД кодом. После завершения задачи файл можно очистить или пометить
задачу завершённой.

## G. Правило финального слияния

Перед финальным merge/push в `main` обязательны:

```
pytest
ruff check .
djlint templates --check
python manage.py check
python manage.py makemigrations --check
```

Плюс: финальный отчёт (изменённые файлы, тесты, риски, команды деплоя) и
проверка, что в коммит не попали лишние файлы: xlsx, бэкапы, дампы,
секреты, `.claude/` (см. `git status --short` до и после `git add`).

## H. Правило продакшен-деплоя

Деплой только после того, как финальный коммит оказался в `origin/main`.
Полный порядок: [production-deploy-runbook.md](../operations/production-deploy-runbook.md).
Краткая последовательность на сервере (`/opt/denstock`):

```
docker compose exec web python manage.py backup_all
docker compose exec web python manage.py ops_check
git fetch
git pull --ff-only
docker compose up -d --build
docker compose exec web python manage.py migrate
docker compose exec web python manage.py check
docker compose exec web python manage.py makemigrations --check
docker compose exec web python manage.py ops_check
docker compose exec web python manage.py backup_all
```

Бэкап до и после; `--ff-only` гарантирует, что на сервере нет локальных
расхождений с `origin/main`.
