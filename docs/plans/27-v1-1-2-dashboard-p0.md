# План v1.1.2 — Полезный dashboard (P0 из аудита)

**Статус:** ПЛАН (реализацию не начинаем без отдельного «go»). Закрывает **P0-1** из
[docs/audits/01-v1-ui-ux-audit.md](docs/audits/01-v1-ui-ux-audit.md). **Read-only:** дашборд
только читает уже существующие сервисы Слоя 21 и тривиальные счётчики; НЕ создаёт движений,
НЕ меняет `StockBalance`, НЕ трогает scanner/barcodes/расчёты/модели/миграции.

---

## 1. Проблема и цель

Сейчас [core/dashboard.html](templates/core/dashboard.html) — 4 плитки со значением «—»
(плейсхолдер Слоя 1). Первый экран после входа выглядит недоделанным. Цель: за один взгляд
показать **что на складе**, **что требует внимания**, **какие действия доступны** и **куда идти**,
без новой бизнес-логики.

## 2. Что переиспользуем (без новой логики)

| Источник | Что даёт | Период? |
|---|---|---|
| `get_stock_report()` ([reports/services.py](apps/reports/services.py:292)) | `part_types_with_stock`, `total_available`, `total_reserved`, `total_quarantine` | нет (срез) |
| `get_low_stock_report()` (там же:326) | список `LowStockRow` → счётчик и строки «требует внимания» | нет |
| `get_sales_report(period)` (там же:167) | `count`, `revenue`, `profit` (финансы — под правом) | да (месяц) |
| `resolve_period` | разбор периода (берём «месяц» по умолчанию) | — |
| Тривиальные read-счётчики | `PartType.objects.count()` (каталог), `Reservation.objects.filter(status=active).count()` | — |

Всё это — **существующие read-only сервисы/запросы** (те же, что уже использует раздел отчётов и
списки). Никаких новых агрегатов складской физики.

## 3. Состав dashboard v1

**A. KPI-плитки** (классы `.card`/`.tile` + `.num`):
- Видов деталей (каталог) — `PartType.count()`;
- Видов на складе — `StockReport.part_types_with_stock`;
- Доступно, ед. — `StockReport.total_available` (`.num`);
- Низких остатков — `len(get_low_stock_report())` (если >0 — `.pill--danger`);
- Активных резервов — `Reservation(status=active).count()`;
- **(Финансы, только под `can_view_purchase_cost`)** Продажи за месяц / Выручка — `get_sales_report(month)`.

**B. «Требует внимания»** (`.card`; пусто → `.empty`):
- низкие остатки — топ-N строк `get_low_stock_report()` (деталь · доступно/мин) + ссылка «Все остатки» (`reports_stock`);
- в карантине, ед. — `StockReport.total_quarantine` (если >0);
- *(future, не в этом слое)* незавершённые партии и истекающие резервы — требуют отдельной
  status-логики procurement/expiry; чтобы не раздувать P0, выносим во future.

**C. «Быстрые действия»** (`.btn`/`.btn--secondary`, **каждая — по capability**):
| Действие | Маршрут | Право |
|---|---|---|
| Поиск детали | `part_search` | все |
| Приёмка сканером | `scanner_receiving` | `can_manage_inventory` |
| Перемещение | `scanner_move` | `can_manage_inventory` |
| Новая продажа | `sale_create` | `can_manage_sales` |
| Новый резерв | `reservation_create` | `can_manage_reservations` |
| Добавить деталь | `part_create` | `can_manage_parts` |
| Добавить партию | `batch_create` | `can_manage_batches` |
| Отчёты | `reports_dashboard` | `can_view_reports` |

> «Печать этикеток» отдельной landing-страницы не имеет (этикетки — с карточки объекта),
> поэтому в быстрые действия v1 не выносим (отметка во future / ведём через «Детали»).

**D. «Последние операции»** — *опционально, если просто*: последние 5 `StockMovement`
(`order_by('-created_at')`, как во `movement_list`), только под `can_view_reports`/
`can_manage_inventory`. Если усложняет — **future**. Рекомендация: вынести во future, чтобы P0
остался маленьким.

## 4. Empty states (вместо «—»)

- Каталог пуст (`PartType.count()==0`) → `.empty`: «Деталей пока нет» + (под `can_manage_parts`)
  кнопка «Добавить деталь».
- Нет остатков (`total_available==0` и т.п.) → подсказка «Начните с приёмки на склад»
  (ссылка `scanner_receiving` под правом).
- Блок «Требует внимания» без проблем → `.empty`: «Всё в порядке: низких остатков нет».
- **«—» как основной контент не используем.**

## 5. Доступ / безопасность

- **Финансы** (выручка/прибыль/себестоимость) — только под `can_view_purchase_cost`; иначе
  плитки/значения **физически не выводим** (как в отчётах Слоя 21).
- **KPI и «Требует внимания»** (производные от отчётов) — под `can_view_reports`. Пользователь
  без этого права (Продавец/Мастер) видит приветствие + **только** «Быстрые действия» по своим
  правам — экран всё равно полезен, а не пустой.
- **Быстрые действия** — каждая по своей capability; запрещённые не показываем.
- Дашборд **read-only**: не создаёт `StockMovement`, не меняет `StockBalance`/документы/статусы.

## 6. UI

Используем уже добавленный фундамент (v1.1.1): `.card`, `.card__header/__title/__body`,
`.pill(--danger/--muted)`, `.empty`, `.num`, `.btn`/`.btn--secondary`, `.page-header`. **Большой
редизайн base-layout не делаем** — меняем только dashboard. Без графиков, без JS, без зависимостей.

## 7. Границы (чего НЕ делаем)

Не меняем: модели, миграции, services складской физики, расчёты отчётов; не создаём
`StockMovement`; не меняем `StockBalance`; не трогаем scanner/barcodes; без графиков/JS/
зависимостей; без редизайна остального проекта (это следующие этапы аудита).

## 9. Риски

- Низкие: view только читает существующие сервисы/счётчики; шаблон — презентация на готовых
  классах. Главная аккуратность — **гейты прав** (финансы/отчёты/действия), их закрываем тестами.

## 10. Тесты (`tests/test_dashboard.py`)

1. Дашборд открывается авторизованному (200), показывает KPI/быстрые действия.
2. Без `can_view_purchase_cost` финансовых данных в ответе **нет** (ни выручки, ни прибыли).
3. С `can_view_purchase_cost` финплитки присутствуют.
4. Быстрые действия фильтруются по capability (Продавец видит «Новая продажа», не видит «Приёмка
   сканером»; кладовщик — наоборот и т.д.).
5. На пустой базе показывается empty-state («Добавить деталь»/«Начните с приёмки»), а не «—».
6. Открытие дашборда **не создаёт** `StockMovement` (счётчик не изменился).
7. Открытие дашборда **не меняет** `StockBalance` (значения не изменились).
8. KPI берутся из существующих сервисов (низкие остатки = `len(get_low_stock_report())`) —
   проверка согласованности числа с сервисом.
9. `pytest`/`ruff`/`djlint --check`/`makemigrations --check` (изменений нет)/`manage.py check` — зелёные.
