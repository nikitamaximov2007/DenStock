# План реализации — Слой 4. Структура склада

**Статус:** на согласовании (2026-06-24) · **Код не пишется до утверждения.**

## 1. Цель слоя

Цифровая структура склада и адресное хранение: дерево мест
(`StorageLocation`) с уровнями склад → зона → … → ячейка (+ приёмка, карантин),
уникальными кодами и штрихкодами, полным адресом и флагом `storage_allowed`.
С экранами просмотра/редактирования, ролевым доступом и навигацией.
**Только структура — без остатков, деталей, партий, движений, сканера и печати.**

## 2. Django-приложение

- **`warehouse`** — единственное новое приложение слоя.

## 3. Модель StorageLocation

| Поле | Тип | Назначение |
|---|---|---|
| name | char(150) | название места |
| code | char(60) **uniq** | уникальный код (часть адреса) |
| barcode | char(80) **uniq** | штрихкод места (авто `LOC:<code>`) |
| level | enum | уровень (см. п.4) |
| parent | FK→StorageLocation ✓null | дерево; `on_delete=PROTECT` |
| storage_allowed | bool | можно ли хранить остаток здесь |
| is_active | bool | активность (деактивация вместо удаления) |
| sort_order | int | порядок отображения |
| description | text blank | описание |
| capacity | PositiveInteger ✓null | вместимость — **только справочное поле**, без логики |
| created_at / updated_at | datetime | метки |

`capacity` включаю как простое необязательное поле (на будущее), **без какой-либо
логики проверки вместимости** на этом слое.

## 4. Уровни хранения (`level`)

`WAREHOUSE` склад · `ZONE` зона · `RACK` стеллаж · `SECTION` секция ·
`SHELF` полка · `CELL` ячейка · `RECEIVING` приёмка · `QUARANTINE` карантин/брак.

- Иерархия **гибкая**: строгий порядок уровней не навязывается (склад может
  пропускать уровни — как в `01-architecture.md`). `WAREHOUSE` обычно корень
  (без родителя), остальное — с родителем; жёстко не проверяем.
- `RECEIVING` и `QUARANTINE` — обычные места, способные держать остаток
  (`storage_allowed=true`), но помеченные особым назначением.

> **Соответствие `02-data-model.md`:** прежнее `kind` + `is_receiving` заменяется
> единым `level` (приёмка = `RECEIVING`). Ссылки в модели данных приведу при
> сборке остатков (Слой 9–10).

## 5. Полный адрес места

Адрес собирается по цепочке `parent` от корня к узлу, соединением **кодов** через
` / `. Пример: `СКЛАД-1 / A / 03 / 02 / 04`.

- Свойство `full_path` (по кодам) — для адреса.
- Свойство `full_name_path` (по названиям) — для подсказок в интерфейсе.
- Реализация — проход вверх по `parent` (склад небольшой; без рекурсивных
  SQL-CTE).

## 6. Ограничения и инварианты

1. **`code` уникален** (DB unique).
2. **`barcode` уникален** (DB unique).
3. **Запрет циклов `parent`** — в `clean()` (как у категорий: нельзя сделать
   родителем себя или своего потомка).
4. **`storage_allowed` — модельный инвариант на будущее.** Сейчас остатков нет;
   добавляю метод-помощник `can_hold_stock()` (= `is_active and storage_allowed`)
   и фиксирую правило в документации. Реальный запрет «нельзя положить остаток в
   `storage_allowed=false`» включается на слоях остатков (9–12).
5. **Нельзя удалять место — только деактивировать** (`on_delete=PROTECT`,
   удаления в UI нет, только toggle).
6. **Нельзя деактивировать место с активными дочерними** местами — toggle вернёт
   понятную ошибку (сначала деактивируйте вложенные). Симметрично: **нельзя
   активировать место, если его родитель неактивен** (чтобы дерево оставалось
   согласованным).

## 7. Генерация barcode места

- При сохранении, если `barcode` пуст, проставляется **`LOC:<code>`**.
- Если администратор задал свой штрихкод — сохраняется как есть.
- **Без PDF-печати** на этом слое: только строковое поле/значение кода
  (печать этикеток — Слой 22).

## 8. Экраны

- **Индекс склада** — точка входа.
- **Дерево мест** — список с отступами (как категории), фильтр активности.
- **Создание места** (форма: name, code, level, parent, storage_allowed,
  sort_order, description, capacity; barcode — авто или вручную).
- **Редактирование места**.
- **Активация/деактивация** (toggle, с проверками п.6).
- **Карточка места** — поля, полный адрес, штрихкод, дочерние места; блок
  «Содержимое ячейки» — **заглушка**: «Остатки появятся в следующих слоях».
- **Без содержимого ячейки** (остатков ещё нет).

Форма переиспользует общий шаблон `directories/form.html`; дерево — отдельный
шаблон по образцу категорий; карточка — новый шаблон.

## 10. Новая capability

Добавляю в `apps/accounts/roles.py` возможность **`MANAGE_WAREHOUSE_STRUCTURE`**
(`manage_warehouse_structure`) — у Администратора и Руководителя; в
`ALL_CAPABILITIES`. Плюс свойство `User.can_manage_warehouse` и
миксин `ManageWarehouseMixin` в `apps/accounts/permissions.py`.

## 11. Навигация

В боковое меню добавляется пункт **«Склад»** — виден всем авторизованным; ведёт
на индекс склада. Кнопки правки внутри экранов — только при
`manage_warehouse_structure`.

## 12. Обязательные тесты

1. Создание склада (`WAREHOUSE`).
2. Создание зоны внутри склада.
3. Создание стеллажа/полки/ячейки (вложенность).
4. Полный адрес строится правильно (цепочка кодов).
5. `barcode` генерируется (`LOC:<code>`).
6. `code` уникален (дубликат отклоняется).
7. Запрет цикла `parent`.
8. Деактивация вместо удаления (после toggle место есть, `is_active=False`).
9. Пользователь без права (кладовщик) не может создавать/редактировать → 403.
10. Наблюдатель может только просматривать (правка → 403).
11. Кладовщик может просматривать (GET → 200).
12. Пункт «Склад» виден в навигации.
13. (доп.) Нельзя деактивировать место с активными дочерними.

## 13. Ручная проверка

1. Войти администратором → создать `СКЛАД-1` → зону `A` → стеллаж `03` → полку
   `02` → ячейку `04`; проверить полный адрес `СКЛАД-1 / A / 03 / 02 / 04` и
   штрихкод ячейки.
2. Создать место приёмки (`RECEIVING`) и карантина (`QUARANTINE`).
3. Попробовать сделать место потомком самого себя → понятная ошибка.
4. Попробовать деактивировать склад с активными зонами → запрет с подсказкой.
5. Войти кладовщиком → структура видна для просмотра, кнопок правки нет; прямой
   POST на изменение → «Нет доступа».
6. Войти руководителем → может править структуру.

## 14. Критерии готовности

1. `StorageLocation` создаётся/редактируется; деактивация вместо удаления.
2. Уровни (включая приёмку и карантин) и дерево работают; циклы запрещены.
3. Полный адрес и штрихцод (`LOC:<code>`) строятся; `code`/`barcode` уникальны.
4. `can_hold_stock()` и правило `storage_allowed` зафиксированы (без остатков).
5. Доступ: правка — Админ/Руководитель; просмотр — всем авторизованным; иначе 403.
6. «Склад» в навигации; индекс, дерево, карточка места.
7. Все обязательные тесты зелёные; `ruff`/`djlint` чисты; миграции с нуля.
8. Детали/партии/остатки/движения/сканер/печать **не реализованы**.
