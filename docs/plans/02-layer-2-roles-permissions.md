# План реализации — Слой 2. Пользователи, роли и права

**Статус:** на согласовании (2026-06-24) · **Код не пишется до утверждения.**

## 1. Цель слоя

Простая и надёжная ролевая модель: роли, базовые проверки доступа, управление
пользователями (только для администратора) и **навигация, зависящая от роли**
(чтобы потом скрывать финансы от кладовщика и закупочную себестоимость от
продавца/мастера). **Без складской бизнес-логики** и **без конструктора прав** —
ровно столько, сколько нужно будущим экранам.

## 2. Роли системы

| Роль | Назначение |
|---|---|
| **Администратор** | Полный доступ, управление пользователями и настройками |
| **Руководитель** | Просмотр финансов/отчётов, подтверждение списаний и корректировок |
| **Кладовщик** | Складские операции (приёмка, размещение, перемещение, инвентаризация) |
| **Продавец/Мастер** | Поиск, резерв, продажа, установка |
| **Наблюдатель** (опционально) | Просмотр без редактирования |

«Наблюдатель» включаю — он дешёвый и закрывает требование «просмотр без
редактирования». На этом слое смотреть ещё особо нечего, поэтому роль
**фундаментная**: задаём её и право `can_edit=False`, наполнение придёт со
складскими экранами.

**Важно:** один пользователь может **совмещать несколько ролей** (требование ТЗ).
Права при этом **складываются**.

## 3. Как реализуем роли — выбранный вариант

**Роли = Django Groups (M2M), возможности (capabilities) вычисляются в коде.**

- 4–5 групп по числу ролей; пользователь состоит в одной или нескольких →
  **естественное совмещение ролей** и сложение прав.
- Тонкие свойства-помощники на `User` (`is_admin`, `is_manager`,
  `is_storekeeper`, `is_seller`, `is_viewer`) и **производные возможности**
  (`can_manage_users`, `can_view_finance`, `can_view_purchase_cost`,
  `can_edit`, `can_confirm_adjustments`) — единая таблица соответствия
  «роль → возможности» в одном месте (`apps/accounts/roles.py`).

**Почему так, а не иначе:**
- *Поле `role` в `User`* — одно значение не даёт совмещать роли; пришлось бы
  делать M2M, а это и есть Groups.
- *Полные Django-Permissions на каждую модель* — преждевременно: складских
  моделей ещё нет, получился бы пустой enterprise-каркас прав.
- Groups + код-маппинг — просто, надёжно, расширяемо: когда появятся реальные
  экраны, добавляем проверки возможностей, не трогая модель пользователя.

Группы создаются **data-миграцией** (идемпотентно), чтобы существовать сразу
после `migrate`. Django-Permission'ы к группам на этом слое **не привязываем** —
группы выступают ярлыками ролей. Суперпользователь приравнивается к Администратору
(`is_superuser` обходит проверки).

## 4. Возможности (capabilities) на этом слое

Матрица «роль → возможность» (фундамент; реальный экран сейчас есть только у
`manage_users`, остальное управляет навигацией и готово к будущим слоям):

| Возможность | Админ | Руковод. | Кладовщик | Продавец/Мастер | Наблюдатель |
|---|:--:|:--:|:--:|:--:|:--:|
| `manage_users` | ✓ | — | — | — | — |
| `can_view_finance` (выручка/прибыль/стоимость остатков) | ✓ | ✓ | — | — | ✓ |
| `can_view_purchase_cost` (закупочная себестоимость) | ✓ | ✓ | — | — | ✓ |
| `can_edit` (право изменять данные) | ✓ | ✓ | ✓ | ✓ | — |
| `can_confirm_adjustments` (подтверждать списания/корректировки) | ✓ | ✓ | — | — | — |

> Нюанс «продавец видит прибыль по своей продаже» — это уровень конкретного
> экрана продаж (Слой 16), а не глобальная возможность. Здесь оставляем
> `can_view_finance=False` для продавца (скрываем глобальную статистику).

## 5. Появляющиеся страницы/экраны (только пользователи)

| Экран | Доступ | Содержимое |
|---|---|---|
| Список пользователей | Администратор | таблица: логин, ФИО, роли, активен; кнопки |
| Создание пользователя | Администратор | логин, ФИО, пароль, роли (группы), активен |
| Редактирование пользователя | Администратор | те же поля; смена ролей; сброс пароля (опц.) |
| Активация/деактивация | Администратор | переключатель `is_active` (POST) |
| Смена роли | Администратор | часть формы редактирования (выбор групп) |
| «Нет доступа» (403) | все | понятная страница при отказе |

Никаких складских данных и отчётов на этом слое.

## 6. Ограничения доступа, которые должны работать

- Обычный пользователь (не админ) **не может управлять пользователями** — экраны
  закрыты (403), пункт навигации скрыт.
- **Кладовщик не видит финансовые разделы** в навигации (`can_view_finance=False`).
- **Продавец/Мастер не видит закупочную себестоимость** (`can_view_purchase_cost=False`)
  — возможность задана; реальное скрытие поля проверится на карточке детали (Слой 5).
- **Наблюдатель не может менять данные** (`can_edit=False`) — write-вью проверяют
  возможность. На этом слое единственная запись — управление пользователями
  (и так только админ), поэтому проверка `can_edit` пока задаётся и тестируется на
  заглушке/служебном представлении.

Разрешено реализовать **базовые проверки и заглушки разделов** без настоящих
складских данных.

## 7. Навигация с учётом роли

Боковое меню строится из списка пунктов, у каждого — требуемая возможность.
Пункт показывается, только если у пользователя есть возможность.

| Пункт меню | Видимость по возможности |
|---|---|
| Главная | всем авторизованным |
| Поиск детали (заглушка) | всем |
| Поступление (заглушка) | кладовщик/админ (`can_edit` + склад) |
| Продажа (заглушка) | продавец/админ |
| Статистика (заглушка) | `can_view_finance` |
| **Пользователи** | `manage_users` |

Возможности прокидываются в шаблоны **context processor**'ом, чтобы меню и
скрытие финансов работали единообразно. Кладовщик и продавец **не увидят**
«Статистику»; только админ увидит «Пользователи».

## 8. Middleware / decorators / mixins

Без тяжёлого middleware (не усложняем). Достаточно:

- **Mixins** для классовых вью: `AdminRequiredMixin` (== `manage_users`),
  `CapabilityRequiredMixin` (проверка произвольной возможности).
- **Decorators** для функциональных вью: `@admin_required`,
  `@capability_required("...")`.
- **Context processor** `accounts.context_processors.capabilities` — отдаёт флаги
  возможностей и пункты навигации в шаблоны.
- **Обработчик 403** (`handler403`) + шаблон `403.html`.

Все проверки — точечные, на уровне вью; глобального middleware-энфорсера нет.

## 9. Обязательные тесты

1. Администратор видит управление пользователями (страница 200, пункт меню есть).
2. Не-администратор **не видит** пункт «Пользователи» в навигации.
3. Не-администратор при заходе на управление пользователями получает **403**.
4. Роль пользователя **сохраняется** (назначили группу — после перезагрузки на месте).
5. **Деактивированный** пользователь не может войти.
6. **Навигация меняется по роли**: кладовщик не видит «Статистику»; админ видит «Пользователи».
7. (доп.) Юнит-тесты маппинга возможностей: совмещение ролей складывает права;
   суперпользователь имеет все возможности.

## 10. Ручная проверка

1. Войти администратором, создать пользователей с разными ролями (в т.ч.
   совмещённые), задать ФИО и пароль.
2. Деактивировать одного — убедиться, что он не входит.
3. Войти кладовщиком — нет «Статистики» и «Пользователей».
4. Войти продавцом — нет «Статистики»; открыть URL управления пользователями
   вручную → страница «Нет доступа».
5. Войти руководителем — «Статистика» видна, «Пользователи» нет.

## 11. Критерии готовности

1. Группы ролей создаются миграцией; пользователю можно назначить одну/несколько.
2. Администратор управляет пользователями (список/создание/редактирование/
   активация/смена ролей); остальным эти экраны недоступны (403).
3. Навигация скрывает разделы по возможностям (финансы — не для кладовщика/продавца).
4. Деактивированный пользователь не входит.
5. Возможности (`manage_users`, `can_view_finance`, `can_view_purchase_cost`,
   `can_edit`, `can_confirm_adjustments`) заданы и покрыты тестами.
6. Все обязательные тесты зелёные; `ruff` и `djlint` чисты.
7. Складская бизнес-логика **не реализована**.

## 12. Файлы, которые будут созданы/изменены

**Создаются:**
- `apps/accounts/roles.py` — константы ролей и таблица «роль → возможности»
- `apps/accounts/permissions.py` — миксины и декораторы доступа
- `apps/accounts/context_processors.py` — возможности и навигация в шаблоны
- `apps/accounts/forms.py` — формы создания/редактирования пользователя
- `apps/accounts/views.py` — CRUD пользователей + переключение активности
- `apps/accounts/urls.py` — маршруты управления пользователями
- `apps/accounts/migrations/0002_roles_groups.py` — data-миграция: создание групп
- `templates/accounts/user_list.html`, `user_form.html`,
  `user_confirm_toggle.html`
- `templates/403.html`
- `tests/test_roles.py`

**Изменяются:**
- `apps/accounts/models.py` — свойства-помощники (`is_admin`, …) и возможности
- `apps/accounts/admin.py` — отображение групп/ролей
- `config/urls.py` — подключение `accounts.urls`, `handler403`
- `config/settings/base.py` — регистрация context processor
- `templates/base.html` — навигация по возможностям (вместо заглушечных пунктов)

Слой 2: пользователи, роли и права

## Границы слоя (чего НЕ делаем)

- Не реализуем склад, детали, партии, остатки, продажи, отчёты.
- Не делаем конструктор прав в интерфейсе.
- Не превращаем в корпоративную систему прав с сотней разрешений.
- Только простая ролевая модель + базовые проверки + навигация под будущие экраны.
