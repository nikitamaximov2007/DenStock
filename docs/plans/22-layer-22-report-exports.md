# План реализации — Слой 22. Экспорт отчётов (CSV)

**Статус:** УТВЕРЖДЁН (2026-06-26) · все рекомендации приняты · реализация в границах §20.

---

## 1. Цель слоя

Добавить **безопасный экспорт** уже существующих read-only отчётов Слоя 21 в файл,
который Денис откроет в Excel, сохранит или кому-то отправит. Экспорт:

- использует **те же сервисы** `apps/reports.services`, что и UI (цифры в интерфейсе
  и файле не разъезжаются);
- использует **те же права** (`VIEW_REPORTS`) и **те же правила скрытия финансов**
  (`can_view_purchase_cost`);
- **ничего не меняет** в складе, не создаёт документы/движения, не пересчитывает
  отчёты иначе, чем UI, не сохраняет файлы на сервере/в БД.

**Главная мысль:** Слой 22 — это **форматирование** уже посчитанных отчётов в файл.
Бизнес-логику он заново не считает и **не раскрывает** финансовые поля тем, у кого
нет `can_view_purchase_cost`.

### Что уже есть (переиспользуем как есть)

| Из Слоя 21 | Назначение |
|---|---|
| `resolve_period(GET) -> Period` | единый разбор периода (UI и экспорт — одинаково) |
| `get_sales_report/…/get_low_stock_report(period)` | готовые dataclass-отчёты |
| dataclass'ы `SalesReport`/`ReturnsReport`/`RepairReport`/`WriteoffReport`/`AdjustmentsReport`/`StockReport`/`LowStockRow` | источник строк экспорта |
| `VIEW_REPORTS`, `can_view_purchase_cost`, `_require_reports` | права и гейт финансов |

### Что нового в Слое 22

- `apps/reports/exporters.py` — **чистые** функции: `(report dataclass, include_costs)
  → (header, rows)`; плюс утилита `csv_response(...)`.
- Экспорт-вьюхи и маршруты `/reports/export/*.csv` в `apps/reports`.
- Кнопки «CSV» в шаблонах дашборда и остатков.
- **Без новых моделей, миграций, приложений и зависимостей.**

## 2. Форматы — **рекомендация: только CSV в Слое 22; XLSX отложить**

| Формат | Решение | Причина |
|---|---|---|
| **CSV** | ✅ делаем | модуль `csv` в **стандартной библиотеке** — ноль новых зависимостей; с BOM+`;` отлично открывается в Excel (RU) |
| **XLSX** | ⏸ **отложить** | `openpyxl` **не установлен** (как и xlsxwriter/tablib/pandas); проект держит зависимости минимальными (Django, django-environ, pytest-django). Добавлять стороннюю либу ради первого среза экспорта не оправдано |

**Обоснование (по вашему правилу §2).** «XLSX только если уже есть openpyxl или его
можно аккуратно добавить без усложнения; иначе — только CSV». openpyxl сейчас **нет**
→ XLSX вынесли бы новую зависимость и тестовую поверхность. CSV покрывает задачу
(Excel-RU открывает BOM+`;`-CSV корректно). Поэтому **Слой 22 = CSV**, XLSX —
отдельный будущий слой (тогда же добавим `openpyxl` в `dependencies`). Раздел §11/§19
описывает, как XLSX встанет позже без переделок (экспортеры уже отдают `(header,
rows)` — XLSX-врапер просто другой «рендерер» тех же строк).

## 5. Какие отчёты экспортируем (первый слой)

Экспортируем **табличные** срезы, ровно из тех же сервисов:

| Экспорт | Источник (сервис) | Тип данных |
|---|---|---|
| Продажи (сводка за период) | `get_sales_report` | 1 строка KPI |
| Возвраты (сводка) | `get_returns_report` | 1 строка KPI |
| Ремонт/выдачи (сводка) | `get_repairs_report` | 1 строка KPI |
| Списания по причине | `get_writeoffs_report` | N строк (по причине) |
| Корректировки (инвентаризация) | `get_stocktaking_report` | 1 строка KPI |
| Текущие остатки по ячейкам | `get_stock_report` | N строк (по ячейке) + итог |
| Низкие остатки | `get_low_stock_report` | N строк (деталь) |

**Что сразу:** перечисленные 7 — они покрывают содержимое дашборда и страницы
остатков. **Что отложено:** единый «dashboard summary одним файлом» (микс KPI разных
доменов в одном CSV неудобен в Excel — лучше отдельные файлы); полная детализация
«продажи по каждой детали» (сейчас сервис даёт топ-N для UI; полный per-part экспорт —
будущий слой, чтобы не вводить новый ORM-запрос в этот слой). Топ-N деталей —
дисплейная аналитика дашборда; в CSV даём агрегатные сводки.

## 6. Маршруты — **рекомендация: отдельные endpoints на отчёт**

```
GET /reports/export/sales.csv?date_from=&date_to=&preset=
GET /reports/export/returns.csv?…
GET /reports/export/repairs.csv?…
GET /reports/export/writeoffs.csv?…
GET /reports/export/stocktaking.csv?…
GET /reports/export/stock.csv
GET /reports/export/low-stock.csv
```

**Обоснование.** Отдельные endpoints **проще тестировать** (один URL → один отчёт →
предсказуемые колонки) и **безопаснее** (нет «type»-диспетчера с риском
неучтённого варианта/IDOR). Единый `?type=` экономил бы строки, но усложнял бы
проверку прав/колонок. Период-зависимые — берут `?date_from/date_to/preset`;
остатки/низкие — точечные, без периода.

## 7. Права

- Доступ к любому экспорту — под **`VIEW_REPORTS`** (тот же `_require_reports`,
  что у страниц отчётов). Без права — **403**.
- Финансовые колонки — **только** при `can_view_purchase_cost`.
- Кладовщик (`VIEW_REPORTS`, без `purchase_cost`) экспортирует **складские** данные
  **без** финансовых колонок.
- Продавец/Мастер (нет `VIEW_REPORTS`) экспорт получить **не может** (403).

---

## 8. Безопасность — финансы не утекают в файл (ключевое правило)

При `not can_view_purchase_cost` экспорт **физически НЕ пишет** денежные поля —
ни в заголовок, ни в строки:

- не включаем себестоимость (`cost`, `cost_total`, `issued_cost`, `adjust_*_cost`,
  `unit_cost_rub`), прибыль (`profit_total`), **выручку** продаж и любые закупочные
  стоимости;
- это **не CSS-скрытие**, а **отсутствие колонок** в данных (заголовок и строки
  строятся по `include_costs`).

**Реализация:** экспортер получает флаг `include_costs = user.can_view_purchase_cost`
и формирует **разный набор колонок**. Тест проверяет, что в байтах файла кладовщика
нет ни заголовков «Себестоимость»/«Прибыль»/«Выручка», ни денежных значений.

> Согласованность с UI: на дашборде денежный блок целиком под `show_costs` (включая
> выручку), поэтому экспорт без `purchase_cost` так же **не содержит выручку** — UI и
> файл показывают одно и то же.

## 9. Период

- Используем **тот же** `resolve_period(request.GET)` из Слоя 21 — `date_from`/
  `date_to`/`preset` работают одинаково в UI и экспорте.
- Кнопки «CSV» на дашборде передают **текущий период** через query-параметры
  (`?date_from=…&date_to=…`), чтобы файл соответствовал тому, что на экране.
- Отдельно «экспортировать пресет» не нужно — пресет уже разворачивается в даты.

## 10. CSV — кодировка, разделитель, числа

| Параметр | Решение | Причина |
|---|---|---|
| Кодировка | **UTF-8 с BOM** (`﻿`) | Excel на Windows корректно открывает кириллические заголовки |
| Разделитель | **`;` (точка с запятой)** | Excel в RU-локали по умолчанию ждёт `;` |
| Перевод строк | `\r\n` | дружелюбно к Excel |
| Decimal | **строка с запятой-разделителем** (`"900,00"`) | Excel-RU распознаёт как число при `;`-разделителе; **никаких float** |
| Цитирование | стандартное `csv` (авто, если в значении встретится `;`/кавычки) | надёжность |

**Обоснование чисел:** `Decimal("900.00")` → строка `"900,00"` (`str(d).replace(".",
",")`). Значения не содержат `;`, поэтому конфликта с разделителем нет. Связка
«`;`-разделитель + `,`-десятичная» — классический RU-Excel-формат (ваша рекомендация
§10) и не ломает тесты (ассертим подстроку `"900,00"`).

## 11. XLSX — как встанет позже (в этом слое НЕ делаем)

Когда добавим XLSX-слой: зависимость `openpyxl` в `dependencies`; врапер
`xlsx_response(filename, header, rows)` поверх **тех же** `(header, rows)` от
экспортеров (один лист на отчёт, строка заголовков, авто-ширина по возможности; без
стилей/графиков/сводных). Менять экспортеры не придётся — они уже отдают строки,
формат-агностично. Поэтому откладывание XLSX **не создаёт техдолга**.

## 12. Имена файлов

| Экспорт | Имя |
|---|---|
| Период-отчёты | `denstock-<report>-YYYY-MM-DD-YYYY-MM-DD.csv` (даты периода) |
| Точечные (остатки/низкие) | `denstock-<report>-YYYY-MM-DD.csv` (текущая дата) |

Примеры: `denstock-sales-2026-06-01-2026-06-30.csv`,
`denstock-stock-2026-06-29.csv`, `denstock-low-stock-2026-06-29.csv`.

**Обоснование:** префикс `denstock-` — узнаваемость/неймспейс; имя отчёта; даты —
трассируемость «за какой период/на какую дату» прямо в имени; имя — **ASCII** (без
кириллицы), чтобы не возиться с RFC 5987-кодированием `Content-Disposition`.

## 13. Сервисы экспорта (`apps/reports/exporters.py`)

**Чистые** функции «dataclass → строки» (не дублируют ORM, не считают бизнес-логику):

```python
def sales_rows(report: SalesReport, period, *, include_costs) -> tuple[list, list]: ...
def returns_rows(report: ReturnsReport, period, *, include_costs) -> tuple[list, list]: ...
def repairs_rows(report: RepairReport, period, *, include_costs) -> tuple[list, list]: ...
def writeoffs_rows(report: WriteoffReport, period, *, include_costs) -> tuple[list, list]: ...
def stocktaking_rows(report: AdjustmentsReport, period, *, include_costs) -> tuple[list, list]: ...
def stock_rows(report: StockReport, *, include_costs) -> tuple[list, list]: ...
def low_stock_rows(rows: list[LowStockRow]) -> tuple[list, list]: ...

def csv_response(filename: str, header: list, rows: list) -> HttpResponse: ...  # BOM+;
def export_filename(slug: str, period=None) -> str: ...                        # денсток-имя
```

- Возвращают `(header, rows)` — **тестируется без HTTP** (ассертим колонки/значения).
- `include_costs` управляет **наличием** денежных колонок (§8).
- Все суммы — `Decimal`→строка (запятая); `csv_response` добавляет BOM и `;`.

## 14. Вьюхи (тонкие)

Каждая экспорт-вьюха делает строго:

1. `_require_reports(request)` (VIEW_REPORTS, иначе 403);
2. `period = resolve_period(request.GET)` (для период-отчётов);
3. `report = get_<x>_report(period)` — **тот же сервис, что UI**;
4. `header, rows = <x>_rows(report, period, include_costs=request.user.can_view_purchase_cost)`;
5. `return csv_response(export_filename(...), header, rows)`.

**Вьюхи не считают отчёты вручную** и не трогают ORM-агрегаты — только сервисы +
экспортеры.

## 16. Read-only гарантия

Экспорт **не**: создаёт `StockMovement`; меняет `StockBalance`/`Sale`/`StockLot`/
статусы документов; пишет файлы на сервере; создаёт записи в БД. Гарантируется тем,
что экспортеры — чистые формат-функции над dataclass'ами, а вьюхи только читают
сервисы. Закрепляется тестом (§18): после скачивания любого экспорта число
`StockMovement` и значения `Sale`/`StockBalance` не меняются.

---

## 17. Производительность

- Экспорт берёт **уже агрегированные** dataclass'ы (Слой 21 считает их ORM-агрегацией)
  — экспортеры лишь форматируют, без новых запросов (нет N+1).
- Объёмы малы (сводки/таблицы по ячейкам/причинам/низким остаткам) → обычный
  `HttpResponse` достаточно; **streaming не делаем** (premature optimization).
- Новых индексов/миграций нет.

## 19. XLSX — почему отложено (явно)

`openpyxl` отсутствует в зависимостях; добавление сторонней либы ради первого среза
экспорта противоречит принципу минимальных зависимостей проекта и расширяет тестовую
поверхность. CSV (stdlib) полностью решает задачу выгрузки в Excel. **XLSX — отдельный
будущий слой**: тогда добавим `openpyxl` и `xlsx_response()` поверх существующих
`(header, rows)` без изменения экспортеров/сервисов.

## 21. Ручная проверка

1. Руководителем открыть `/reports/`, выбрать период → нажать «CSV» у «Продажи».
   Файл `denstock-sales-…csv` скачивается; открыть в Excel — кириллица читаема,
   разделитель `;`, числа как «900,00»; суммы совпадают с дашбордом.
2. Проверить «CSV» у Возвратов/Ремонта/Списаний/Корректировок — суммы совпадают.
3. На `/reports/stock/` скачать остатки и низкие остатки; сверить с экраном.
4. Кладовщиком скачать те же экспорты → **нет** колонок выручки/себестоимости/
   прибыли; складские колонки на месте.
5. Продавцом/Мастером дернуть `/reports/export/sales.csv` → **403**.
6. Сменить период и повторно скачать — данные в файле меняются вместе с UI.
7. Убедиться, что после серии экспортов склад/документы не изменились
   (`check_stock_balance` зелёный; числа документов те же).

## 22. Критерии готовности

1. 7 CSV-экспортов (`sales/returns/repairs/writeoffs/stocktaking/stock/low-stock`)
   работают; данные берутся из **тех же** `get_*_report`, что UI (числа совпадают).

3. CSV: UTF-8 BOM, `;`-разделитель, Decimal-строкой с запятой; имя `denstock-…`.
4. Период берётся из `resolve_period` (как UI); кнопки «CSV» передают текущий период.
5. Возвраты в экспорте — отдельно, **не** вычитаются из выручки.
6. **Read-only:** экспорт не создаёт движений и не меняет документы/остатки (тесты).
7. Границы: только CSV; нет XLSX/PDF/графиков/email/scheduled/файлов на диске/моделей.
8. Тесты зелёные (вкл. регресс); `ruff`/`djlint` чисты; `manage.py check` ок;
   `makemigrations --check` — **изменений нет** (без моделей/зависимостей).
