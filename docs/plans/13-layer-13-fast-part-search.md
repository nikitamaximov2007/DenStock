# План реализации — Слой 13. Быстрый поиск детали

**Статус:** на согласовании (2026-06-25) · **Код не пишется до утверждения.**

---

## 1. Цель слоя

Дать **быстрый read-only поиск**, чтобы во время звонка Денис ввёл название, OEM,
артикул, штрихкод, серийник или внутренний номер и **сразу** увидел по детали:

- есть ли она;
- сколько всего (физически);
- сколько доступно;
- где лежит (ячейки);
- из каких партий;
- по какой рекомендованной/минимальной цене продавать;
- **себестоимость — только для ролей с `can_view_purchase_cost`**.

Слой **только читает** каталог, экземпляры, лоты и кэш остатков. **Никаких
складских/коммерческих действий**: не создаёт движений, не меняет `StockBalance`,
не продаёт/резервирует/перемещает; scanner resolve не трогает. Это «глаза»
продавца, отвечающие на главный вопрос Дениса.

> **Соотношение с уже существующим.** В каталоге уже есть простой поиск в
> `PartTypeListView` (имя + `PartNumber.normalized_value`) — это **управленческий
> список** деталей. Слой 13 — **отдельный экран** «Быстрый поиск» с **наличием**
> (всего/доступно/где/партии/цена), которого в каталоге нет. Старый список не
> трогаем.

---

## 2. Где размещаем — `apps/core`

**Сервис поиска и экран — в `apps/core`** (рядом со сканером). Обоснование:

1. Поиск **кросс-доменный**: читает `catalog` (`PartType`/`PartNumber`/
   `PartBarcode`/совместимость), `inventory` (`PartItem`/`StockLot`/`StockBalance`),
   `warehouse` (`StorageLocation`), `procurement` (`Batch`). Он не принадлежит ни
   одному домену.
2. `core` — **интеграционный слой**, уже зависящий от всех доменов (там живёт
   сканер Слоёв 11–12). Зависимость остаётся **односторонней** (`core →
   catalog/inventory/warehouse/procurement`). Положить поиск в `catalog` нельзя
   без обратной зависимости `catalog → inventory` (наличие), что ломает
   направление зависимостей; «найти деталь» (сканер) и «найти деталь поиском» —
   один кластер в `core`.
3. Новое приложение `apps/search` — лишний слой ради одного сервиса+экрана
   (как и на Слоях 11–12 отклоняли `apps/scanner`). *(Альтернатива в открытых
   вопросах.)*

Сервис — **чистая read-only функция** `apps/core/search.py::search_parts(query,
*, user)`; экран — `core.views.search_page`. Ничего не пишет.

## 3. Что ищем (всё сводится к `PartType`)

Результат — **виды деталей** (`PartType`): главный бизнес-вопрос «есть ли такая
деталь». Запрос сопоставляется с `PartType` через поля:

| Поле | Модель | Как |
|---|---|---|
| Название | `PartType.name` | `icontains` (часть названия) |
| OEM/артикул/аналог | `PartNumber.value` / `normalized_value` | по нормализованному (`icontains`) → `part_id` |
| Заводской штрихкод | `PartBarcode.value` | `icontains` → `part_id` |
| Внутренний номер | `PartItem.internal_number` | `icontains` → `part_type_id` |
| Внутренний штрихкод | `PartItem.internal_barcode` | `icontains` → `part_type_id` |
| Серийный номер | `PartItem.serial_number` | `icontains` → `part_type_id` |
| Техника | `VehicleMake.name` / `VehicleModel.name` | через `PartCompatibility` → `part_id` (рекомендация — лёгкий join) |

**`Batch.number` и `StorageLocation.code` в поиск НЕ включаем** (рекомендация):
«найти, что в партии/ячейке» — это **навигация**, а не «найти деталь»; скан
номера партии/кода ячейки **уже** ведёт на их карточки через scanner resolve
(Слой 11). Так поиск остаётся «детале-центричным» и не превращается в фильтр-
аналитику. *(Открытый вопрос — добавить ли отдельную ветку «что в партии/ячейке».)*

Минимальная длина запроса — **2 символа** (короче — подсказка «уточните запрос»),
чтобы не сканировать весь каталог.

---

## 4. Нормализация запроса

- `q = query.strip()`.
- Для **номеров** — переиспользуем **существующую** `catalog.normalize_number`
  (убирает пробелы/`-`/`_`/`.`/`/`, верхний регистр): `nq = normalize_number(q)`;
  сравниваем с `PartNumber.normalized_value` (`icontains nq`). Это та же
  нормализация, что у сканера и каталожного списка — единый разбор номеров.
- Для **названия / серийника / штрихкода / внутреннего номера** — `icontains q`
  (регистронезависимо), чтобы ловить часть строки.

Отдельной новой нормализации не вводим.

---

## 5. Что показываем в строке результата (`PartType`)

| Поле | Источник | Видимость |
|---|---|---|
| Название, категория, производитель | `PartType` (+select_related) | все |
| Основные OEM/артикулы | `PartNumber` (is_primary / первые N) | все |
| Режим учёта (`tracking_mode`) | `PartType` | все |
| Рекомендованная / минимальная цена | `PartType.recommended_price`/`min_price` | все |
| **Всего (физически)** | агрегат наличия (§6) | все |
| **Доступно** | агрегат наличия (§6) | все |
| На приёмке (`receiving`) | первичка (§6) | все |
| Ячейки (где лежит) | агрегат наличия (§6) | все |
| Партии | агрегат наличия (§6) | все |
| **Себестоимость** | `landed`-стоимость экземпляров/лотов | **только `can_view_purchase_cost`** |

Цены продажи (рекомендованная/минимальная) — **справочные, видны всем** (это не
закупочная себестоимость). Скрывается по роли **только** закупочная
себестоимость.

---

## 6. Как считаем наличие

**Первично — `StockBalance` (read-optimized кэш), с откатом на первичку.**

- **Всего/Доступно/Ячейки/Партии** — агрегируем `StockBalance` по `part_type`:
  `physical = Σ quantity_physical`, `available = Σ quantity_available`; ячейки/
  партии — `distinct` по строкам баланса. Это ровно та задача, под которую кэш
  и создан (Слой 10) — быстрый разрез по детали.
- **На приёмке (`receiving`)** — из **первички**: `PartItem`/`StockLot` в статусе
  `receiving` (они могут ещё не попасть в кэш — кэш наполняется приёмкой/
  `rebuild_stock_balance`). Считаем отдельным показателем «из них N на приёмке».
- **Откат на первичку**: если у найденной детали **нет строк `StockBalance`**, но
  есть первичные `PartItem`/`StockLot` (кэш ещё не пересобран для неё) — считаем
  всего/доступно/ячейки **из первички**, чтобы поиск **никогда не занижал**
  наличие.

**`StockBalance` остаётся кэшем, не источником истины.** Поиск его **только
читает** и **никогда не пишет**; рассинхрон лечит команда Слоя 10
`rebuild_stock_balance`, а не поиск. *(Альтернатива «считать всегда из первички»
— проще и всегда точно, но без кэша; в открытых вопросах.)*

Семантика `available` здесь — арифметическая сводка кэша (как зафиксировано на
Слое 10), **не** «разрешение продавать» (это проверит Слой 16).

---

## 7. Как показываем `PartItem` (поштучные, разворот по детали)

Для деталей `tracking_mode=serial` — компактный список экземпляров (с лимитом,
напр. первые 20):

- внутренний номер (ссылка на `item_detail`);
- статус;
- ячейка (`current_location`);
- партия;
- серийный номер;
- **себестоимость скрыта без `can_view_purchase_cost`**.

Разворот показываем **только тем, кто видит инвентарь** (`can_manage_inventory or
is_viewer`) — у них есть доступ к `item_detail`. Продавцу/Мастеру разворот и
ссылки на экземпляры **не показываем** (см. §10), только сводку наличия.

---

## 8. Как показываем `StockLot` (количественные, разворот по детали)

Для деталей `tracking_mode=bulk` — список лотов (лимит):

- количество (`quantity`);
- статус;
- ячейка (`location`, ссылка на `lot_detail`);
- партия;
- строка партии (`batch_line`);
- **себестоимость скрыта без `can_view_purchase_cost`**.

Тоже только для инвентарь-видящих ролей; продавцу — сводка без разворота.

## 9. UX

- Отдельная страница **«Быстрый поиск»** `/search/` (`name="part_search"`).
- Крупная строка поиска, **автофокус**, поиск по **Enter** (обычная GET-форма
  `?q=…` — нативная отправка, без кастомного JS; server-rendered).
- Результаты — простая таблица/карточки: по строке на деталь со сводкой §5;
  крупно «Доступно». Для инвентарь-видящих — разворот экземпляров/лотов §7–8.
- **Быстрые ссылки**: на `part_detail` (всем), `item_detail`/`lot_detail`/
  `location_detail` (инвентарь-видящим).
- **Пустой результат** — понятное сообщение «Ничего не найдено по запросу „…“».
- **Пустой/короткий запрос** — приглашение «Введите название, номер, штрихкод или
  серийник (минимум 2 символа)».
- Глобальное поле «Поиск детали…» в топбаре (сейчас `disabled`) и пункт меню
  «Поиск детали» (сейчас заглушка) **подключаем** к `/search/` простой GET-формой.
- **Не** делаем живой HTMX-поиск/сложные фильтры — это отдельная будущая
  оптимизация (дизайн §4.2 допускает HTMX-debounce позже).

## 10. Роли

| Роль | Поиск | Сводка наличия / ячейки / цены продажи | Разворот экз./лотов | Себестоимость |
|---|---|---|---|---|
| Администратор / Руководитель | да | да | да | **да** |
| Кладовщик | да | да | да | **нет** (нет `can_view_purchase_cost`) |
| Наблюдатель | да (read-only) | да | да | да (есть `can_view_purchase_cost`) |
| Продавец/Мастер | да | **да** | **нет** (нет доступа к `item/lot_detail`) | **нет** |

- Экран — **`@login_required`**, доступен **всем аутентифицированным** (поиск —
  это не складская операция). Это первый экран, реально полезный Продавцу/Мастеру.
- **Себестоимость** (`landed_cost_rub`/`landed_unit_cost_rub`) — только при
  `can_view_purchase_cost` (контекст `show_costs`). Цены **продажи** видны всем.
- **Разворот экземпляров/лотов** со ссылками на `item_detail`/`lot_detail` —
  только `can_manage_inventory or is_viewer` (у Продавца/Мастера эти карточки
  под `InventoryViewMixin` дают 403, поэтому ссылок ему не показываем; сводка
  наличия и `part_detail` ему доступны).
- Поиск ничего не редактирует — read-only для всех ролей.

## 11. Производительность

- Простые ORM-запросы: матч `PartType` id по полям §3 (несколько `values_list`,
  объединение в множество), затем агрегаты наличия по `part_type__in`
  с `select_related`/`prefetch_related`; лимит результатов (напр. 30 деталей).
- **`pg_trgm`/GIN-индексы** (дизайн §2.13) — **будущая оптимизация**, не вводим
  сейчас: тесты идут на SQLite, `icontains` работает и там, и в Postgres.
- **Не** добавляем поисковый движок (Elastic и т. п.).

## 12. Тесты (`tests/test_search.py`)

Покрываем список заказчика (п. 12):

1. Поиск по `PartType.name` находит деталь.
2. Поиск по OEM **с нормализацией** (`«0 986-221.047»` ↔ `«0986221047»`) находит.
3. Поиск по `PartBarcode.value` находит деталь.
4. Поиск по `PartItem.internal_number` находит деталь.
5. Поиск по `serial_number` находит деталь.
6. Результат показывает **физический остаток** (всего).
7. Результат показывает **ячейку**.
8. Результат показывает **партию**.
9. Себестоимость **скрыта** без `can_view_purchase_cost` (кладовщик/продавец).
10. Руководитель **видит** себестоимость.
11. Продавец/Мастер **может** пользоваться поиском (200), но без себестоимости и
    без ссылок на экземпляры/лоты.
12. Неизвестный запрос → **пустой результат** с сообщением.
13. Поиск **не создаёт** `StockMovement` (счётчик неизменен).
14. Поиск **не меняет** `StockBalance` (счётчик/значения неизменны).
15. Поиск **не выполняет** продажи/резервы/перемещения (нет таких сайд-эффектов;
    объекты не меняют статус/ячейку после серии поисков).

Доп.: поиск требует login (аноним → редирект); откат на первичку (деталь с
первичкой, но без строк `StockBalance`, всё равно показывает наличие);
доступно/всего считаются из кэша; `receiving`-count из первички.

## 13. Ручная проверка

1. Войти продавцом → «Быстрый поиск».
2. Ввести часть названия → строки с «Доступно N», ячейками, ценой продажи; **без**
   себестоимости и без ссылок на экземпляры.
3. Ввести OEM с пробелами/дефисами → та же деталь (нормализация).
4. Ввести заводской штрихкод → деталь.
5. Ввести внутренний номер `DS-…` или серийник → деталь этого экземпляра.
6. Войти кладовщиком → видны разворот экземпляров/лотов и ячейки, **без**
   себестоимости.
7. Войти руководителем → **видна** себестоимость.
8. Ввести мусор → «Ничего не найдено».
9. Убедиться, что после серии поисков нет новых `StockMovement`, `StockBalance`
   не изменился, статусы/ячейки объектов прежние.
10. Проверить автофокус и поиск по Enter; пустой запрос → подсказка.

## 14. Критерии готовности

1. По названию/OEM(норм.)/штрихкоду/внутр.номеру/серийнику поиск находит деталь и
   показывает всего/доступно/`receiving`/ячейки/партии/цены продажи.
2. Наличие берётся из `StockBalance` (кэш) с откатом на первичку; поиск **не
   пишет** `StockBalance` и **не создаёт** движений.
3. Себестоимость скрыта без `can_view_purchase_cost`; цены продажи видны всем.
4. Все аутентифицированные могут искать; продавец/мастер — без себестоимости и без
   инвентарных ссылок; наблюдатель — read-only.
5. Пустой/неизвестный/короткий запрос — понятные сообщения; автофокус, Enter.
6. Границы: нет продаж/резервов/установок/возвратов/списаний/инвентаризации/
   аналитики/PDF; scanner resolve не изменён; поиск строго read-only.
7. Все тесты зелёные; `ruff`/`djlint` чисты; `makemigrations --check` чисто (новых
   моделей нет → миграций нет); `manage.py check` без проблем.
