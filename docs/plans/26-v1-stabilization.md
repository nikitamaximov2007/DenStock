## 1. Fresh clone check (поднять с нуля)

Прогон по чистой машине/папке (как сделал бы Денис), фиксируем результат каждого шага:

```bash
git clone <repo> denstock && cd denstock
cp .env.example .env            # заполнить SECRET_KEY и пароли
docker compose build            # образ собирается (вкл. postgresql-client)
docker compose up -d            # db + web + proxy поднимаются (healthcheck green)
# entrypoint сам: migrate + collectstatic + (опц.) createsuperuser из .env
docker compose exec web python manage.py createsuperuser   # если не задан в .env
```

Проверки после старта:
- открыть <http://localhost> → форма входа; войти администратором;
- `GET /healthz/` → `{"status":"ok","db":"ok"}` (200);
- **media upload:** загрузить фото детали (Слой 24) → фото **отображается** (Caddy `/media/*`);
- `docker compose exec web python manage.py ops_check` → все проверки `[OK]`;
- **backup:** `docker compose exec web python manage.py backup_all` → в `./backups/<ts>/`
  появились `db.dump` + `media.tar.gz` + `manifest.json` (видны на хосте);
- `docker compose down && up -d` → данные и фото **на месте** (тома сохранили).

Фиксируем найденные расхождения (если есть) как мелкие правки конфигов/доков — **без**
изменения бизнес-логики.

**Условие:** не смешивать с бизнес-логикой; только эти 2 файла; отдельный коммит.

## 4. Smoke checklist (ручной сквозной прогон)

Один проход «как пользователь», чтобы убедиться, что цепочка цела (read-only проверка, без
спец-данных). Каждый шаг — на реальном экране/команде:

| # | Действие | Где |
|---|---|---|
| 1 | Создать справочники (категория, ед., производитель, ячейка) | `/directories/`, `/warehouse/` |
| 2 | Создать партию + строку | `/batches/` |
| 3 | Финализировать себестоимость партии | карточка партии |
| 4 | Создать `PartItem` / `StockLot` из строки | карточка партии |
| 5 | Принять на склад (в ячейку) | карточка экземпляра/лота |
| 6 | Найти деталь через поиск | `/search/` |
| 7 | Напечатать этикетку | «Печать этикетки» (Слой 23) |
| 8 | Загрузить фото детали/экземпляра | карточки (Слой 24) |
| 9 | Провести продажу | `/sales/` |
| 10 | Оформить возврат на склад | `/returns/` |
| 11 | Провести списание | `/write-offs/` |
| 12 | Провести инвентаризацию/корректировку | `/stocktaking/` |
| 13 | Открыть отчёты | `/reports/` |
| 14 | Экспорт CSV | ссылки «CSV» в отчётах (Слой 22) |
| 15 | `backup_all` | `manage.py backup_all` |
| 16 | `ops_check` | `manage.py ops_check` |

Цель — «всё открывается и проводится», а не нагрузочный тест. Любой сбой — баг-фикс
отдельной задачей (не в этом плане).
