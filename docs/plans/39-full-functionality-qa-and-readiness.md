# Layer 26 - Full functionality QA and product readiness

**Статус:** аудит готовности (проверка + тесты + docs). Без новых больших фич, без
изменения бизнес-логики, моделей, миграций, URL, backup/restore, Docker.

Цель слоя: убедиться, что текущий MVP DenStock работает как цельный продукт для склада
запчастей, и усилить автоматические сценарные проверки там, где они были слабыми.

## 7. Disabled menu items: Поступление и Статистика

### 7.1 «Поступление»

- **Сейчас:** заглушка (`stub=True`, `url=None`), без route/view/template/model/service.
- **Чем должно отличаться от «Партии» и «Приёмки сканером»:** «Партии» уже закрывают приход
  товара от поставщика (номер партии, поставщик, строки, доп. расходы, себестоимость), а
  «Приёмка сканером» ставит физический товар в ячейки. То есть приходный процесс уже есть,
  просто разбит на «партию» (документ) и «приёмку» (постановка на склад).
- **Нужен ли отдельный приходный документ:** только если Денису нужна формальная приходная
  накладная как отдельная сущность. Технически это можно построить **поверх `Batch`** (без
  новой модели), сделав «Поступление» удобной обёрткой над созданием партии + приёмкой.
- **Вывод:** не делать отдельную тяжёлую сущность. Низкий приоритет. Оставить «скоро» либо
  позже сделать как UX-обёртку над Партией. Отдельная модель/миграции не оправданы.

### 7.2 «Статистика»

- **Сейчас:** заглушка, без route/view/template/model/service.
- **Чем должно отличаться от «Отчётов»:** «Отчёты» - операционные за период (продажи,
  ремонт, возвраты, списания, корректировки). «Статистика» - складская аналитика состояния,
  а не движения за период.
- **Полезные метрики для Дениса (все выводятся из существующих данных, без новых моделей):**
  - стоимость склада (сумма landed cost по остаткам);
  - топ ходовых деталей (из движений расхода);
  - мёртвый/залежавшийся склад (позиции без движений N дней);
  - низкие остатки (уже есть на dashboard, свести сюда);
  - оборачиваемость (расход за период / средний остаток);
  - стоимость и число активных резервов;
  - выручка/прибыль сводно (есть в отчётах, дать витрину).
- **Можно ли без новых моделей:** да, всё считается из `StockBalance`, `StockLot`,
  `PartItem`, `StockMovement`, `Reservation`.
- **Вывод:** это **лучший кандидат на Layer 27** - высокая ценность для владельца склада,
  read-only, без миграций и без риска для данных.

В Layer 26 ни «Поступление», ни «Статистика» не реализуются (только анализ и roadmap).

## 8. Recommended Layer 27

**Статистика (складская аналитика, read-only).**

- Новый раздел `/statistics/` (или включить существующий серый пункт).
- Метрики: стоимость склада, мёртвый склад, топ ходовых, оборачиваемость, резервы.
- Только чтение, без новых моделей и миграций (агрегации по существующим таблицам).
- Гейт `can_view_finance` (как сейчас у пункта меню).
- Переиспользовать KPI-плитки, таблицы и фильтр периода из v1.2.x.

Причина приоритета: закрывает самый ценный серый пункт, помогает Денису видеть «что лежит
мёртвым грузом» и «сколько денег на складе», при этом безопасно (read-only).

## 10. What not to do yet

- Не добавлять web restore/import/upload бэкапов (осознанное решение безопасности).
- Не вводить новые модели/миграции ради «Поступления», пока Денис явно не попросит формальную
  приходную накладную.
- Не строить тяжёлую аналитику с отдельными агрегатными таблицами: Статистику считать
  на лету по существующим данным.
- Не менять backup/restore CLI-workflow и Docker/PostgreSQL.
- Не трогать оболочку v1.2.x: desktop sidebar раскрыт, mobile hamburger drawer как есть.

## 11. Product readiness verdict

**Вердикт: продукт функционально готов к реальной ежедневной работе Дениса.**

Что уже можно использовать прямо сейчас:
- полный склад: каталог, справочники, партии, приёмка, экземпляры/лоты, ячейки, движения,
  остатки;
- быстрый поиск и сканер (главный сценарий «есть ли деталь и где она»);
- резервы, продажи, ремонт, возвраты, списания, инвентаризация с защитой остатков;
- отчёты за период и CSV;
- роли и разграничение доступа;
- локальные и offsite-бэкапы, production deploy runbook.

Целостность данных защищена инвариантами сервисного слоя и покрыта тестами (**598 passed**),
сквозной smoke подтверждает отсутствие 500 на всех страницах.

Обязательно проконтролировать перед «боевым» запуском (эксплуатация, не код):
- что offsite-бэкап реально отрабатывает по cron на VPS (проверить `offsite_status.json` и
  бакет), см. runbook и план 37 по совместимости версий PostgreSQL;
- что администратор сменил дефолтные секреты из `.env`.

Можно отложить без риска:
- «Поступление» как отдельный документ (приход уже закрыт Партией + Приёмкой);
- расширенная «Статистика» (рекомендуется как Layer 27, но не блокирует запуск).

Короткий итог: DenStock перестал быть «набором экранов» и работает как связный складской
продукт; следующий шаг ценности - витрина «Статистика», а не новая механика.
