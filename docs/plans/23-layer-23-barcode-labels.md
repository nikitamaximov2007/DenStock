# План реализации — Слой 23. Печать складских этикеток (штрихкоды)

**Статус:** УТВЕРЖДЁН (2026-06-30) · все решения зафиксированы жёстко (§ниже) · реализация строго в границах §18. **Ключевой инвариант приёмки: Слой 23 НЕ меняет `resolve_scan`, ledger (`StockMovement`/`StockBalance`) и складскую физику — это только печатное представление уже существующих кодов.**

---

## 1. Цель слоя

Дать пользователю **печатать складские этикетки со штрихкодами** из уже существующих
в системе кодов — чтобы наклеить их на экземпляр детали, ячейку склада и (где уместно)
карточку вида детали. Слой:

- **берёт уже существующие коды** (`PartItem.internal_barcode`, `StorageLocation.barcode`,
  `PartBarcode.value`), а не вводит новые складские сущности и не меняет складскую физику;
- даёт **печатный вывод** (HTML print view), пригодный для печати из браузера;
- **полностью read-only**: не создаёт движений, не меняет объекты, не пишет файлы на диск,
  не ведёт журнал печати;
- остаётся согласованным со сканером: печатаем ровно те коды, которые сегодня умеет
  распознавать `resolve_scan` (Слой 11) — этикетка и сканер «говорят на одном языке».

**Главная мысль:** Слой 23 — это **представление уже имеющихся кодов на бумаге**. Никакой
новой складской логики, никакой записи в БД, никаких новых зависимостей.

| Существующее | Где | Что даёт этикетке |
|---|---|---|
| `PartItem.internal_number` (`DS-000001`) | `apps/inventory/models.py` | человекочитаемый номер |

| `resolve_scan(code)` | `apps/core/scanner.py` | гарантия, что печатаемый код распознаётся |

### Что нового в Слое 23

- **Новое приложение `apps/labels`** — кросс-доменное (item/location/part), без моделей.
- Генератор **inline-SVG Code128** своими силами (без зависимостей, без файлов).
- Print-view'ы и маршруты `/labels/...`, печатные шаблоны + печатный CSS (`@media print`).
- Кнопки «Печать этикетки» на карточках экземпляра / ячейки / вида детали.
- Новая возможность `PRINT_LABELS` в `apps/accounts/roles.py`.
- **Без новых моделей, миграций и внешних зависимостей.**

## 3. Решение по StockLot (ключевое)

Ранее принято правило: **`StockLot` напрямую не сканируется** — лоты выбираются из UI, у
лота нет и не должно быть собственного сканируемого штрихкода. Слой 23 это правило
**сохраняет**:

- **НЕ** вводим поле `StockLot.barcode` и **НЕ** добавляем `StockLot` в `resolve_scan`;
- **НЕ** делаем для лота этикетку со сканируемым лот-кодом (это спровоцировало бы
  сканирование лота вопреки модели «сканируем ячейку, а не лот»);
- **Финальное решение:** в Слое 23 **этикетки `StockLot` нет**. Физический носитель для
  bulk-остатка — это **этикетка ячейки** (`LOC:<code>`, уже в составе слоя). Лоту
  отдельная наклейка не нужна.
- **Future-note (не в этом слое):** если когда-нибудь понадобится «полочная» наклейка для
  лота — это будет **информационная** наклейка (название детали, кол-во, партия), а
  сканируемым на ней будет **код ячейки** `LOC:<code>`, а не новый лот-код. Никаких
  изменений в сканере и складской физике это не потребует.

## 4. Формат вывода — **рекомендация: только HTML print view; PDF отложить**

| Формат | Решение | Причина |
|---|---|---|
| **HTML print view** | ✅ делаем | ноль новых зависимостей; печать из браузера (Ctrl+P), там же «Сохранить как PDF» силами браузера; просто тестируется (ассертим разметку/коды) |
| **PDF (серверный)** | ⏸ **отложить** | требует тяжёлой новой зависимости (`weasyprint`/`reportlab` — **обе отсутствуют**, проверено). Браузерный «Print → Save as PDF» закрывает потребность «сохранить файл» без зависимости |

**Обоснование (проверка зависимостей).** Текущие зависимости проекта: `Django`,
`django-environ`, `psycopg`, `gunicorn`, `whitenoise`. Баркод/PDF-библиотек нет
(`reportlab`/`weasyprint`/`python-barcode`/`qrcode`/`Pillow` — не установлены). Проект
держит зависимости минимальными (как и в Слое 22 с XLSX). Поэтому Слой 23 = **HTML-печать**,
серверный PDF — отдельный будущий слой (тогда и добавим зависимость осознанно).

## 5. Штрихкод — как рендерить (ключевое решение)

**Рекомендация: собственный генератор inline-SVG Code128** (subset B), без зависимостей и
без файлов. Человекочитаемый код печатаем **крупным текстом под штрихами** в любом случае.

Почему Code128 / SVG:

- Code128 subset B покрывает весь нужный ASCII: заглавные, цифры, `-`, `:`. Наши коды
  (`ITEM:DS-000001`, `LOC:A-01`, заводские ШК) кодируются без проблем.
- SVG — это **набор прямоугольников-штрихов в разметке**: рендерится браузером, **не
  требует шрифтов** (отказ от «barcode-font» — нельзя надёжно хранить/шарить шрифт),
  **не требует файлов** (ничего не пишем на диск), **детерминирован и тестируем**.
- QR **не берём**: требует Рида—Соломона (ECC) — это сложнее и опаснее в ручной
  реализации, чем Code128; для линейных складских кодов Code128 достаточно.

Запасной вариант (если при реализации Code128-генератор окажется рискованным): **деградируем
до крупного машиночитаемого текстового кода** без графики. Это уже полезно: камера-сканера
в проекте нет, сканер принимает **текстовые** коды (ввод/USB-сканер) — напечатанный текст
`ITEM:DS-000001` уже вводится/сканируется как текст. Но цель слоя — настоящий Code128 SVG,
т.к. он безопасно реализуем и покрывается тестами (§6).

| Вариант | Вердикт |
|---|---|
| barcode-font (CSS/HTML текст спецшрифтом) | ❌ нельзя надёжно хранить/шарить шрифт, нестабильно |
| **SVG Code128 своими силами** | ✅ **выбран**: без зависимостей/файлов, детерминирован, тестируем |
| dependency `python-barcode` | ❌ новая зависимость ради линейного ШК не оправдана |
| только текстовый код без графики | ⚠️ запасной/деградация; уже полезен (сканер принимает текст) |

## 6. Реализация графического штрихкода (Code128 SVG)

- **Стандарт:** Code128, subset B (печатный ASCII 32–126). Структура символа: `Start B
  (104)` → данные → контрольная сумма (mod 103, взвешенная) → `Stop (106)`. Каждый символ —
  11 модулей; Stop — 13 (доп. штрих). Кодовые таблицы паттернов — константа в модуле.
- **Где код:** `apps/labels/barcode.py` — чистая функция
  `code128_svg(value: str, *, module=2, height=60) -> str` (возвращает строку `<svg>…`),
  плюс `encode_code128_b(value) -> list[int]` (список кодовых значений с контрольной суммой)
  для прямого юнит-тестирования без SVG.
- **Без внешних сервисов, без записи файлов** — только строка SVG, встраиваемая в шаблон.
- **Тесты (детерминизм):** для известной строки проверяем (а) последовательность кодовых
  значений и (б) корректную контрольную сумму mod 103 — по эталону Code128; проверяем, что
  SVG валиден (начинается с `<svg`, содержит `<rect>`-штрихи, есть quiet-zone слева/справа),
  и что недопустимый (непечатный) символ обрабатывается предсказуемо (исключение/замена).
- **Подпись:** под штрихами — человекочитаемое значение крупным моноширинным текстом.

## 7. Размещение кода — **новое приложение `apps/labels`**

Слой кросс-доменный (PartItem ← inventory, StorageLocation ← warehouse, PartType ←
catalog), поэтому логичнее отдельное приложение, а не «прописка» в один из доменов
(как `apps/reports` — самостоятельное приложение для сквозной задачи).

```
apps/labels/
  __init__.py
  apps.py            # AppConfig
  barcode.py         # генератор Code128 SVG (без зависимостей)
  views.py           # тонкие print-view'ы (read-only)
  urls.py            # /labels/...
  context.py         # (опц.) сборка данных этикетки из объекта
templates/labels/
  base_label.html    # каркас листа этикеток + @media print + @page
  item_label.html
  location_label.html
  part_label.html
```

- Подключение: `path("labels/", include("apps.labels.urls"))` в `config/urls.py`;
  `"apps.labels"` в `INSTALLED_APPS`.
- **Без моделей** → приложению не нужна папка `migrations` с моделями (см. §8).

## 9. Права

- Новая возможность **`PRINT_LABELS`** в `apps/accounts/roles.py` (строка-капабилити,
  как `VIEW_REPORTS`). Доступ к любой печати — под `PRINT_LABELS`, иначе **403**.
- Добавляется в `ALL_CAPABILITIES` и в `ROLE_CAPABILITIES` для ролей ниже. Это **чисто
  кодовое** изменение карты возможностей (возможности вычисляются в коде из членства в
  группах) — **новых Group и миграций не требует**.

**Роли (рекомендация):**

| Роль | `PRINT_LABELS` | Обоснование |
|---|---|---|
| Администратор | ✅ | полный доступ (superuser получает все возможности автоматически) |
| Руководитель | ✅ | управляет складом/каталогом, печать этикеток уместна |
| Кладовщик | ✅ | **ключевой потребитель**: приёмка/размещение → клеит этикетки на экземпляры и ячейки |
| Продавец/Мастер | ❌ | не ведёт приёмку/размещение; маркировка остатка — складское действие, не его роль |
| Наблюдатель | ❌ | read-only финансовый просмотр; операционной печатью не занимается |

**Обоснование по Продавцу/Мастеру (по запросу §9).** В текущей карте ролей приёмка и
размещение — у Кладовщика; Продавец/Мастер делает резервы/продажи/выдачу в ремонт и
закупочных деталей склада не маркирует. Печать складских этикеток — продолжение
приёмки/размещения, поэтому Продавцу/Мастеру её не выдаём (при необходимости — отдельным
обоснованным изменением карты ролей в будущем). Наблюдателю операционные действия не даём
по тому же принципу, что и остальные `MANAGE_*`.

## 11. Печать нескольких этикеток (batch print)

**Рекомендация:** в Слое 23 — одиночные этикетки + **простая** печать выбранных экземпляров.

- Core: одиночные `/labels/item|location|part/<pk>/`.
- Опционально (если просто): `/labels/items/?ids=1,2,3` — печать нескольких **PartItem**
  на одном листе (по плитке-этикетке на объект). `ids` — недоверенные: каждый pk
  re-fetch'ится из БД, несуществующие молча отбрасываются, право проверяется один раз.
- **Future (не в этом слое):** массовая печать «все этикетки партии» / «все экземпляры в
  ячейке» — отложить, чтобы не раздувать слой. Если `?ids=` окажется нетривиальным —
  тоже выносим во future, оставив только одиночные этикетки.

## 13. Данные на этикетке (состав)

**PartItem**
- ШК (Code128 SVG) от `ITEM:DS-000001`;
- `DS-000001`; название детали; основной номер/OEM (если есть); № партии; ячейка
  (`current_location.full_path` или `—`); серийный номер (если есть).

**StorageLocation**
- ШК (Code128 SVG) от `LOC:<code>`;
- `<code>`; полный адрес (`full_path`); уровень (`get_level_display`);
  назначение (`get_purpose_display`) / активность.

**PartType**
- название; основной номер детали (`numbers.filter(is_primary=True)` → иначе первый OEM);
- производитель (`manufacturer`) и категория (`category`) — если не раздувает компоновку;
- если есть `PartBarcode` — ШК (Code128 SVG) от `PartBarcode.value` + само значение;
- если `PartBarcode` нет — ШК **не рисуем** (у вида нет собственного системного кода;
  OEM-номер неоднозначен для сканера, §5/§14), печатаем только текстовые подписи.

## 14. Согласованность со сканером (read-model)

Печатаем только то, что распознаёт `resolve_scan` (Слой 11), чтобы этикетка реально
сканировалась:
- `ITEM:DS-…` → шаг 1 (PartItem.internal_barcode);
- `LOC:<code>` → шаг 1 (StorageLocation.barcode);
- заводской ШК → шаг 5 (PartBarcode.value, глобально уникален).

Сканер **не меняем** (он остаётся чистым резолвером). PartType без `PartBarcode` не получает
графический ШК именно потому, что у вида нет однозначно-сканируемого кода.

---

## 15. Безопасность

- Без `PRINT_LABELS` → **403** (декоратор/гард во всех label-view'ах).
- Объект читается сервером **по pk** (`get_object_or_404`); несуществующий объект → **404**.
- Query/hidden-параметры **недоверенные**: в batch-печати `ids` только перечисляют pk;
  каждый re-fetch'ится, ничего из параметров в данные этикетки не подставляется напрямую.
- Печать **ничего не выполняет** (read-only, §16) — нет действий, которые мог бы
  спровоцировать подменённый параметр.

---

## 16. Read-only гарантия

Печать этикеток **не**: создаёт `StockMovement`; меняет `StockBalance`/`PartItem`/
`StorageLocation`/`PartType`/`StockLot`; создаёт документы; меняет статусы; пишет файлы на
диск; ведёт журнал печати. Гарантируется тем, что view'ы только читают объект и рендерят
шаблон, а генератор ШК — чистая функция. Закрепляется тестами (§17): счётчики
`StockMovement` и значения объектов после серии печатей не меняются.

## Решения (утверждены 2026-06-30)

Все границы зафиксированы заказчиком жёстко. Открытых вопросов нет.

1. **Формат:** только HTML print view; PDF на Слое 23 **не реализуем** (нет
   `reportlab`/`weasyprint`, тяжёлые зависимости не добавляем; браузерная печать и
   «Сохранить как PDF» закрывают потребность) — ✅ принято.
2. **Штрихкод:** собственный Code128 SVG (subset B) — **только если** простой, локальный,
   детерминированный, без внешних зависимостей/сервисов/файлов и покрыт тестами
   (checksum mod 103 + базовый SVG); поддержка только нужных проекту ASCII-кодов
   (`ITEM:DS-…`, `LOC:…`, `DS-…`, OEM/ШК в рамках Code128 B). Под ШК **обязательно** крупный
   machine-readable текст. **Если Code128 раздувает слой или становится рискованным —
   откатываемся к текстовой этикетке без графики, а графический ШК выносим в future** — ✅ принято.
3. **StockLot:** поле `StockLot.barcode` **не добавляем**; этикетку лота **не делаем**;
   `resolve_scan` **не меняем**; правило «StockLot напрямую не сканируем» сохраняем. Носитель
   bulk-остатка — этикетка ячейки `StorageLocation`. Bulk label — отдельный будущий слой/решение — ✅ принято.
4. **Размещение:** новое `apps/labels` (кросс-доменное: PartItem/StorageLocation/PartType);
   **без моделей, без миграций**, без print jobs, без сохранения файлов — ✅ принято.
5. **Право:** новая capability `PRINT_LABELS` → Администратор, Руководитель, Кладовщик.
   Продавец/Мастер — без печати; Наблюдатель — без печати — ✅ принято.
6. **Что печатаем:** PartItem — ✅; StorageLocation — ✅; PartType/PartBarcode — ✅;
   StockLot — ❌ (см. п.3). Состав полей — §13 — ✅ принято.
7. **Batch print:** одиночные этикетки + **опц.** `?ids=1,2,3` для PartItem (если не
   раздувает). Массовая печать по партии/складу/ячейке, очередь печати, print jobs — **не
