# План реализации — Слой 14. Перемещение деталей и лотов

**Статус:** на согласовании (2026-06-25) · **Код не пишется до утверждения.**

---

## 1. Цель слоя

Дать **рабочий, обнаруживаемый сценарий перемещения** уже принятых
(`available`/`quarantine`) `PartItem` и `StockLot` между ячейками склада:
отсканировал экземпляр (или выбрал лот) → отсканировал новую ячейку →
подтвердил → система **создала `StockMovement` и обновила `StockBalance`**.

**Главный архитектурный контроль (как в Слое 12):** все изменения идут **только
через сервисы Слоя 10** (`move_part_item` / `move_stock_lot`). **View не создаёт
`StockMovement` напрямую**, не вызывает `_refresh_balance`, не пишет в
`StockBalance` и не дублирует ledger-логику. View — оркестратор:
*резолв → проверка → вызов сервиса*. Закрепляется тестом-моком (см. §11.18).

### Что уже есть (переиспользуем, НЕ переписываем)

| Уже реализовано | Где | Роль в Слое 14 |
|---|---|---|
| `move_part_item(item, to_location, *, by, comment="")` | `apps/inventory/services.py` (Слой 10) | единственный путь перемещения экземпляра |
| `move_stock_lot(lot, to_location, *, by, comment="")` | `apps/inventory/services.py` (Слой 10) | единственный путь перемещения лота |
| Перемещение через **обычный UI** (выпадающий список ячеек): `item_move`/`lot_move` + `templates/inventory/move_form.html` | `apps/inventory` (Слой 10) | остаётся как форма-fallback; **не удаляем** |
| Приёмка сканером, которая для `available`-объекта делает `move_*` | `core.scanner_receiving` (Слой 12) | **не трогаем**; Слой 14 — отдельный экран движения |
| Чистый `resolve_scan(code)` | `apps/core/scanner.py` (Слой 11) | резолв сканов на экране перемещения |

### Что нового в Слое 14

**Один новый экран** — «Перемещение сканером» (`/scanner/move/`,
`name="scanner_move"`) в `apps/core`, архитектурный близнец экрана приёмки Слоя
12, но заточенный под перемещение: скан экземпляра / выбор размещённого лота →
скан целевой ячейки → подтверждение → `move_*`. Плюс пункт навигации и ссылки из
карточек. **Новых моделей и сервисов нет → миграций нет.**

**Чего слой НЕ делает:** продажи, резервы, установки, возвраты, списания,
инвентаризация, аналитика, PDF-этикетки; не вводит barcode для `StockLot`; не
делает частичное перемещение лота; не меняет `resolve_scan` (остаётся чистым); не
меняет модели/сервисы Слоёв 8–10.

## 3. Как используем scanner resolve (Слой 11)

Каждый скан проходит через **чистый** `resolve_scan(code)` — он **не выполняет
действий**, только возвращает тип и объект. Экран перемещения решает, годится ли
распознанное для текущего шага.

| Скан | Шаг «объект» | Шаг «ячейка» |
|---|---|---|
| `ITEM:DS-…` / `DS-…` → `PartItem` | ✅ берём экземпляр | ❌ «ожидается ячейка» |
| серийник → один `PartItem` | ✅ берём экземпляр | ❌ «ожидается ячейка» |
| `LOC:…` / код ячейки → `StorageLocation` | ❌ «сначала отсканируйте деталь» | ✅ берём целевую ячейку |
| `PartBarcode`/`OEM` → `PartType` | ⚠️ «это вид детали, а не конкретный экземпляр; отсканируйте экземпляр (ITEM:/DS-…) или серийник» — **не перемещаем** | ❌ «ожидается ячейка» |
| `П-…` → `Batch` | ⚠️ «это партия, а не экземпляр» | ❌ «ожидается ячейка» |
| `ambiguous` (серийник у нескольких) | показываем `candidates`, оператор **выбирает** (никакого авто-выбора) | то же |
| `unknown` | ошибка + запись `UnresolvedScan` (переиспользуем `core.views._record_unresolved`) | то же |

- **`unknown`** журналируется так же, как в endpoint/приёмке — через общий
  `_record_unresolved` (не дубль логики).
- На любом «не том» результате (`unknown` / `ambiguous` / `part_type` / `batch` /
  неверный тип для шага) **сервис не вызывается** → движения и баланс не меняются
  (тест §11.8–11.9).

## 4. Что делаем со `StockLot` (нет своего barcode)

`StockLot` **не имеет** штрихкода (решение Слоёв 9/11 подтверждаем — barcode для
лота **не вводим**). Поэтому:

- **лот выбирается из UI** — список **размещённых** лотов (`available`/
  `quarantine`) с указанием детали, количества, текущей ячейки и кнопкой
  «Переместить»; клик кладёт лот в состояние операции и переводит на шаг «скан
  ячейки»;
- затем **сканируется новая ячейка**;
- **перемещается лот целиком** (`move_stock_lot`); количество движения =
  `lot.quantity`;
- **частичное перемещение лота не реализуем** — оно потребовало бы split лота
  (новый лот + перенос количества + проверки остатка) и усложнило бы слой; это
  отдельный будущий слой. На шаге подтверждения это явно обозначено («лот целиком»).

Подбор лота сканом `PartType`/`BatchLine` **не вводим** (одна строка партии может
иметь несколько лотов в разных ячейках → неоднозначность). Скан `PartType` на шаге
выбора лота как фильтр списка — вынесено в открытые вопросы (не усложнять).

## 5. Сервисы — переиспользуем Слой 10, без дублирования

Используем **существующие** сервисы (сигнатуры — `apps/inventory/services.py`):

```python
move_part_item(item, to_location, *, by=None, comment="")  # перемещение размещённого экземпляра
move_stock_lot(lot,  to_location, *, by=None, comment="")  # перемещение целого лота
```

Оба уже: проверяют `can_hold_stock()` целевой ячейки; требуют физический статус
(`receiving`/`available`/`quarantine`); отклоняют перемещение «в ту же ячейку»
(`from == to`); для лота проверяют отсутствие конфликтующего лота той же строки в
целевой ячейке; пишут `StockMovement` (`_record_movement`) и обновляют **оба**
баланса (`_refresh_balance` для from и to); работают под `select_for_update` в
`@transaction.atomic`.

- **View вызывает только эти сервисы.** Никаких `StockMovement.objects.create`,
  `_refresh_balance`, записи в `StockBalance` из вьюхи. View ловит `InventoryError`
  и показывает сообщение.
- **Новой логики в `services.py` не требуется** — все нужные операции уже есть.
  Если по ходу что-то понадобится — добавляем **в сервис**, а не во вьюху (но на
  этом слое необходимости нет).

## 6. Правила

| Правило | Кто гарантирует |
|---|---|
| Перемещать можно `available`/`quarantine` | сервис (`status in *_PHYSICAL_STATUSES`) + **гард экрана** (см. ниже) |
| `receiving`-объект идёт через **приёмку**, не через перемещение | гард экрана: статус `receiving` → сообщение «объект ещё не размещён — используйте Приёмку», **без действия** |
| `sold`/`installed`/`written_off`/`returned`/`depleted` перемещать нельзя | сервис отклоняет (`InventoryError`); экран показывает понятный текст |
| Целевая ячейка должна `can_hold_stock()` (active + storage_allowed) | сервис (`move_*`) → `InventoryError` |
| Движение ссылается на `from_location` и `to_location` | `_record_movement` (Слой 10): `move_item`/`move_lot` всегда с обеими ячейками |
| `StockBalance` обновляется (источник и приёмник) | сервис (`_refresh_balance(from)` и `_refresh_balance(to)`) |
| Перемещение в ту же ячейку **не создаёт лишнее движение** | сервис бросает «уже в этой ячейке»; экран показывает это как **нейтральный no-op** |

### Гард экрана для `receiving` (политика workflow, не ledger)

`move_part_item` технически допускает перемещение `receiving`-экземпляра, у
которого есть `current_location`. Но **дедик-экран перемещения** трактует
`receiving` как «ещё не принят» и **не перемещает** его, а отсылает на приёмку.
Это **гард уровня экрана** (проверка статуса перед выбором сервиса), а **не**
изменение сервиса и **не** ledger-логика. Так receive- и move-сценарии остаются
разделены. Форма-fallback Слоя 10 поведение сервиса не меняет (открытый вопрос —
выравнивать ли её гард; рекомендация: не трогать).

### Перемещение в ту же ячейку — обоснование выбора

**Выбор: no-op с понятным сообщением, без движения** (рекомендация).

- Сервис уже бросает `InventoryError("…уже в этой ячейке.")` → **движение не
  создаётся** (инвариант живёт в сервисе — единый источник истины).
- Экран ловит это и показывает **нейтральный** статус-блок «Объект уже в этой
  ячейке — перемещение не требуется» (не «красная» ошибка).
- Почему не «жёсткая ошибка»: оператор не сделал ничего деструктивного; повтор
  скана той же ячейки — обычная ситуация. Почему не «тихий no-op без сообщения»:
  оператору нужна обратная связь, что состояние не изменилось.

Итог удовлетворяет требованию «не создаёт лишнее движение» (тест §11.14) и даёт
понятное поведение, не дублируя проверку из сервиса во вьюхе.

## 7. UI — экран «Перемещение сканером»

Отдельный экран `/scanner/move/` (`name="scanner_move"`), крупное поле скана.
**Состояние — в скрытых полях формы (stateless, без session)**: шаг выводится из
заполненности `object_kind`/`object_id`/`location_id` (как в Слое 12).

**Шаги:**

1. **«Сканируйте деталь»** — объект не выбран. Поле скана активно; ниже — список
   **размещённых** лотов (`available`/`quarantine`) с кнопкой «Переместить».
2. **«Сканируйте ячейку»** — экземпляр отсканирован / лот выбран; ячейка ещё нет.
3. **«Подтвердите»** — объект и ячейка есть; сводка: что, **откуда → куда**, «лот
   целиком» для лота, и кнопка **«Подтвердить»**.

- **Статус-блоки** `partials/_status_block.html`: `success` (после подтверждения,
  через `messages`+PRG), `error` (InventoryError/неверный тип/unknown), `info`
  (no-op «уже в этой ячейке»; ambiguous — список вариантов).
- **Кнопка «Сбросить»** — очищает операцию (возврат к шагу 1).
- **История последних перемещений текущего пользователя** — последние ~10
  движений `move_item`/`move_lot` с `created_by=request.user` из `StockMovement`:
  дата, тип, деталь, **откуда → куда**. Суммы — только при `can_view_purchase_cost`.
- **No-JS fallback:** экран полностью server-rendered; поле скана — обычный
  `<input>` в `<form>`, Enter отправляет нативно; каждый скан — POST, сервер
  резолвит и перерисовывает шаг. Автофокус — существующим приёмом (как в Слое 12).

**Поток (stateless, скрытые поля):**

```
GET                  → шаг 1 (пустая форма); опц. ?item=<pk> / ?lot=<pk> — преселект → шаг 2
POST scan            → resolve(code): нет объекта → ждём PartItem (гард статуса) → шаг 2
                                       есть объект, нет ячейки → ждём StorageLocation → шаг 3
POST select_lot      → object_kind=stock_lot, object_id=<lot>, шаг 2
POST select_candidate→ выбран PartItem из ambiguous, шаг 2
POST confirm         → проверка (тип/статус/ячейка) → move_* ; успех → messages + PRG (reset); ошибка/no-op → сообщение
POST reset           → пустая форма
```

Скрытые `object_id`/`location_id` **недоверенные**: вьюха всегда перечитывает
объект/ячейку из БД (`_load_operation`) и доверяет проверкам сервиса — подмена id
ничего не ломает (тест §11.19).

**Ссылки из карточек.** В `item_detail`/`lot_detail` добавляем ссылку
«Переместить сканером» → `scanner_move?item=<pk>` / `scanner_move?lot=<pk>`
(deep-link в шаг 2). Существующая форма-fallback («Переместить» → выпадающий
список) **остаётся**. Точная компоновка ссылок — мелкий открытый вопрос.

## 8. Для `PartItem`

1. Скан экземпляра (`ITEM:DS-…` / `DS-…` / серийник) → `resolve_scan` → `PartItem`.
2. **Гард статуса** при выборе объекта:
   - `available` / `quarantine` → берём, шаг «скан ячейки»;
   - `receiving` → сообщение «ещё не размещён — используйте Приёмку», без действия;
   - терминальный/прочий → «нельзя переместить в текущем статусе», без действия.
3. Скан целевой ячейки (`LOC:…` / код) → `resolve_scan` → `StorageLocation`.
4. Подтвердить → `move_part_item(item, ячейка, by=request.user)`:
   - `current_location` меняется **только через сервис**;
   - создаётся `StockMovement` типа `move_item`, `quantity = 1`, с
     `from_location` (прежняя) и `to_location` (новая);
   - обновляются **оба** баланса (источник и приёмник);
   - та же ячейка → no-op (§6), без дубля движения.

---

## 9. Для `StockLot`

1. Выбрать лот из списка размещённых (`available`/`quarantine`) — **без скана** (у
   лота нет barcode).
2. Скан целевой ячейки (`LOC:…` / код) → `resolve_scan` → `StorageLocation`.
3. Подтвердить → `move_stock_lot(lot, ячейка, by=request.user)`:
   - `location` меняется **только через сервис**;
   - создаётся `StockMovement` типа `move_lot`, `quantity = lot.quantity` (**лот
     целиком**), с `from_location`/`to_location`;
   - обновляются **оба** баланса;
   - конфликт (в целевой ячейке уже есть лот этой строки) → `InventoryError`
     «слияние лотов — будущий слой», без движения;
   - та же ячейка → no-op (§6).
- **Частичное перемещение лота не реализуем** на этом слое (см. §4).

## 11. Тесты (`tests/test_inventory_movement.py`)

Покрываем список заказчика (п. 11):

1. Страница перемещения требует login (аноним → редирект).
2. Кладовщик может открыть перемещение (200).

    Приёмку»), движения нет.

    целиком (`move_stock_lot`), `quantity` движения = `lot.quantity`.
16. `StockLot.location` меняется; создаётся `StockMovement` типа `move_lot`;
    `StockBalance` обновлён для обеих ячеек.
17. Частичное перемещение `StockLot` **не реализовано** — на экране нет поля
    количества/частичного переноса (проверяем отсутствие).
18. **View не пишет `StockMovement` напрямую, а вызывает сервис** —
    `patch("apps.core.views.move_part_item")`: при подтверждении сервис вызван
    **ровно один раз**, и при замоканном сервисе вьюха **сама не создаёт** ни
    одного `StockMovement` (счётчик неизменен). Контроль архитектуры.
19. Hidden-поля перепроверяются сервером: подмена `object_id`/`location_id` на
    несуществующий/негодный → ошибка, без движения (доверяем БД/сервису, не hidden).
20. Себестоимость скрыта без `can_view_purchase_cost` (кладовщик не видит сумм в
    истории; админ — видит).

Доп.: скан ячейки на шаге «деталь» → «сначала отсканируйте деталь», без действия;
скан `PartType` (заводской ШК/OEM) → «это вид детали…», без действия.

## 12. Ручная проверка

1. Войти кладовщиком → «Перемещение».
2. Отсканировать `available`-экземпляр (`DS-…`) → шаг «сканируйте ячейку».
3. Отсканировать другую годную ячейку → шаг «подтвердите» со сводкой «откуда → куда».
4. Подтвердить → «Успешно»; в карточке экземпляра — новая ячейка; в «Движениях» —
   `move_item` с обеими ячейками; в «Остатках» — строка старой ячейки убыла, новой
   появилась; в истории экрана — запись.
5. Отсканировать ту же деталь + **ту же** ячейку → «уже в этой ячейке», движение
   не задвоилось.
6. Отсканировать `receiving`-экземпляр → «используйте Приёмку», без движения.
7. Выбрать размещённый лот из списка → отсканировать другую ячейку → подтвердить →
   лот целиком в новой ячейке, `move_lot`, баланс обеих ячеек обновлён.
8. Отсканировать ячейку со `storage_allowed=false` → ошибка, без движения.
9. Отсканировать мусор → «не распознан», запись в «Нераспознанные», без движения.
10. Войти кладовщиком → суммы в истории не видны; админом → видны.
11. Войти продавцом → пункта «Перемещение» нет; прямой URL `/scanner/move/` → 403.
12. В «Движениях» убедиться, что все `move_*` имеют автора (`created_by`) —
    доказательство, что прошли через сервис.

## 13. Критерии готовности

1. Скан экземпляра / выбор лота + скан ячейки + подтверждение перемещает объект
   **только через сервисы Слоя 10**; во вьюхе нет `StockMovement.objects.create` и
   записи в `StockBalance` (доказано тестом-моком §11.18).
2. `PartItem`: `move_part_item`; `current_location`, движение `move_item`
   (from/to, qty=1), оба баланса — корректны.
3. `StockLot`: выбор из UI + скан ячейки → `move_stock_lot` целиком; barcode для
   лота **не введён**; частичное перемещение **не реализовано**.
4. Правила §6 соблюдены: `available`/`quarantine` перемещаются; `receiving` → на
   приёмку; терминальные/`depleted` отклонены; негодная ячейка отклонена; «та же
   ячейка» — no-op без лишнего движения.
5. `unknown`/`ambiguous`/неверный тип скана **не вызывают** действий (нет движений)
   и понятно сообщаются; `unknown` журналируется в `UnresolvedScan`.
6. Права: экран — `MANAGE_INVENTORY`; продавец/мастер/наблюдатель не входят; суммы
   скрыты без `can_view_purchase_cost`.
7. Границы соблюдены: нет продаж/резервов/установок/возвратов/списаний/
   инвентаризации/аналитики/PDF; barcode лота не добавлен; частичное перемещение
   лота не сделано; `resolve_scan` остался чистым; ledger-логика Слоя 10 не
   изменена и не продублирована.
8. Все тесты зелёные; `ruff`/`djlint` чисты; `manage.py check` без проблем;
   `makemigrations --check` чисто (**новых моделей нет → миграций нет**).
