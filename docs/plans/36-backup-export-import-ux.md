# План v1.1.8 — Backup export/import UX, manual/automatic, safe restore wizard

**Статус:** ПЛАН (реализацию не начинаем без «go»). Только проектирование. В этом шаге код/
модели/миграции/restore из web **не делаем**. Опирается на существующий Backup UI (v1.1.7,
`/operations/backups/`) и `apps/operations/backup.py`.

**Ключевая мысль:** «Импорт бэкапа» = **destructive restore** (откат системы). Одной кнопки
«Импорт» быть не должно — нужен **safe restore wizard** с подтверждением фразой и обязательным
аварийным бэкапом текущего состояния перед восстановлением. Реализуется **поэтапно** (A → B → C).

---

## 1. Термины для UI (понятные владельцу)

| Термин в UI | Что реально делает | Тех. смысл |
|---|---|---|
| **Экспорт бэкапа** / «Создать ручной бэкап» | сохранить текущее состояние в локальный бэкап | `backup_all(trigger="manual")` |
| **Импорт бэкапа** | перевести систему на состояние выбранного бэкапа | **destructive `restore_db` + `restore_media`** |
| **Ручной бэкап** | создан человеком через UI | `type=manual` |
| **Автоматический бэкап** | создан планировщиком (cron) | `type=automatic` |
| **Аварийный бэкап перед восстановлением** | авто-снимок текущего состояния до restore | `type=pre_restore` |
| **Загруженный бэкап** | залит файлом в staging | `type=uploaded` |

**Почему «Импорт» = restore/откат.** «Импорт» звучит безобидно (как загрузка справочника), но по
сути **заменяет** текущую БД и media состоянием из бэкапа: всё, что появилось **после** даты
бэкапа, исчезает. Поэтому в UI рядом с «Импорт» всегда — явное «= восстановление/откат системы».

---

## 2. Разделение manual / automatic (без миграций)

Текущий `manifest.json` (`backup.backup_all`): `created_at`, `engine`, `db_file`, `media_file`,
`version`, `git_commit`. **Поля типа нет.**

**Предложение (Этап A, без миграций — это файл-манифест, не БД):**
- Добавить в `backup_all(*, trigger="manual", ...)` параметр и записать в manifest поле
  **`type`** (или `trigger`): `manual` | `automatic` | `pre_restore` | `uploaded`.
- CLI: `backup_all --trigger automatic` (cron), по умолчанию `manual`.
- UI показывает **читабельный бейдж**: «Ручной» / «Автоматический» / «Аварийный» / «Загруженный».
- **Обратная совместимость:** старые manifest без `type` → показывать как **«Legacy / неизвестный
  тип»** (не падать, не ломать список).

> Это единственное изменение backup-логики во всём слое — маленькое, аддитивное, без миграций.
> Всё остальное — UI и host-скрипты.

---

## 3. Экспорт бэкапа (Этап A)

- **Синяя primary-кнопка** «Экспорт бэкапа» (или «Создать ручной бэкап») — переименование текущей
  «Создать полный бэкап».
- Внутри — существующий `backup_all` (не дублируем backup-код); manifest получает `type=manual`.
- После создания — success-message; бэкап доступен к скачиванию.
- UI честно пишет: **локальный экспорт не заменяет offsite**.

---

## 4. Автоматические бэкапы (Этап B)

- **Host-level** scheduled script/cron: `docker compose exec -T web python manage.py backup_all
  --trigger automatic --keep-last N`.
- UI показывает их **отдельно/бейджем «Автоматический»**.
- **Retention** для automatic (daily/weekly/monthly — host-скрипт + `--keep-last`).
- Automatic-бэкап **уходит offsite** (Этап B/следующий слой; сам offsite-провайдер — отдельно).
- По возможности **ручной бэкап не мешает retention автоматических** (напр. retention считает
  только `type=automatic`; ручные не удаляются авто-ротацией). — уточнить при реализации.

---

## 5. Импорт бэкапа / restore wizard (Этап C — проектируем, НЕ реализуем)

Пошаговый мастер (несколько экранов/подтверждений, не одна кнопка):

1. **Выбор источника**: существующий backup-run **или** загрузка archive в staging (§9).
2. **Проверка manifest** (валидный/битый/отсутствует).
3. **Показать содержимое**: дата, engine, `db_file`, `media_file`, version, git_commit, **тип
   бэкапа**, размеры файлов.
4. **Красное предупреждение**: «После восстановления система вернётся к состоянию на дату
   бэкапа. Все данные, созданные после этого момента, будут утеряны.»
5. **Ручное подтверждение фразой**: пользователь вводит **`ВОССТАНОВИТЬ`** (точное совпадение).
6. **Авто emergency backup** текущего состояния: `backup_all(trigger="pre_restore")`.
7. **Restore разрешён только после успешного** emergency backup (нет emergency → restore
   заблокирован).
8. **Restore только если есть и db, и media** в выбранном бэкапе.
9. **После restore** — предложить/запустить `ops_check` и показать чек «войдите, проверьте фото/
   остатки».
10. **Что теряется (объяснение простым языком):** если бэкап от понедельника, а сегодня пятница —
    все детали/продажи/движения/фото/пользователи/изменения **со вторника по пятницу** будут
    удалены; система станет такой, какой была **в понедельник**.

---

## 6. Что именно теряется при импорте (для UI, простым языком)

Показать владельцу прямо в мастере: «Импорт бэкапа — это **откат во времени**. Пример: бэкап от
понедельника, сегодня пятница → всё, что вы сделали со вторника по пятницу (новые детали,
продажи, движения, фото, пользователи), **исчезнет**. Система вернётся к понедельнику. Отменить
это можно будет только из аварийного бэкапа, который создаётся автоматически перед восстановлением.»

---

## 7. Почему restore из web опасен (зафиксировать риски)

- Можно **затереть актуальную БД**.
- Можно **оборвать активные сессии** пользователей.
- Restore может **упасть посередине** (частично восстановленное состояние).
- **media может перезаписаться**.
- Пользователь может **выбрать не тот backup**.
- Злоумышленник с admin-доступом может **разрушить систему** одним действием.

Вывод: restore через web допустим **только** как многошаговый мастер с барьерами (§8), и только
**после** отдельного согласования. По умолчанию — restore остаётся CLI (`--yes`).

---

## 8. Минимальные условия, без которых web-restore нельзя (must-have)

- Только **admin/superuser** (`is_admin`).
- **POST-only** + **CSRF**.
- **Confirm phrase** `ВОССТАНОВИТЬ` (точное совпадение).
- **Emergency backup** (`pre_restore`) до restore; **нет emergency → restore запрещён**.
- **Manifest validation** (валидный, есть db+media).
- **File allowlist** (`manifest.json`, `db.dump`, `db.sqlite3`, `media.tar.gz`).
- **Path traversal protection** (как в v1.1.7: резолв пути + `parent == run_dir`/`BACKUP_ROOT`).
- **Никакого выполнения** загруженных файлов; распаковка tar — с data-фильтром (без traversal).
- **Audit/лог** (кто/когда запустил restore) — хотя бы server log.
- **Понятная обработка ошибок** (restore упал → внятное сообщение + указатель на emergency backup).
- **`ops_check` после restore**.

---

## 9. Upload / import backup (staging)

- Загрузка **только в staging** (не поверх рабочих данных).
- Разрешённые: архив (или набор файлов `db.*` + `media.tar.gz` + `manifest.json`, если поддержим).
- **Size limit** (крупные архивы — честно предупреждать/ограничивать).
- **Validate manifest** после загрузки.
- **Не распаковывать** опасные path-traversal-записи (tar data-фильтр).
- **Restore НЕ запускается сразу после upload** — только через мастер (§5).
- Загруженный бэкап получает `type=uploaded`.
- Backup-файлы (и staging) **не коммитятся** в Git.

## 10. Этапы будущей реализации

**Этап A — безопасно и просто (первый к реализации):**
- Переименование кнопки → «Экспорт бэкапа» / «Создать ручной бэкап», синяя primary.
- Поле `type` в manifest (`backup_all(trigger=...)`); бейджи manual/automatic/pre_restore/uploaded/legacy.
- **Без restore.**

**Этап B — scheduled automatic backup:**
- Host cron/systemd → `backup_all --trigger automatic`; retention; `offsite_status`; offsite sync.

**Этап C — import/restore wizard:**
- Многошаговый мастер (§5), confirm phrase, emergency backup, restore, `ops_check`, тесты.

> Этап A не зависит от B/C; C делаем **только** после отдельного «go» и с полным набором барьеров.

---

## 11. Тесты будущей реализации

**Этап A:** manual backup → `type=manual`; старый manifest без `type` → «legacy»; кнопка экспорта
имеет текст и primary-стиль; бейджи manual/automatic отображаются; **restore в UI отсутствует**.

**Этап B:** host-скрипт создаёт `type=automatic`; status-файл **без секретов**; offsite «не
настроено» → не ошибка.

**Этап C:** restore-wizard требует confirm phrase `ВОССТАНОВИТЬ`; **без emergency backup restore
не запускается**; невалидный/битый бэкап не восстанавливается; **path traversal запрещён**;
обычный admin-flow показывает большое предупреждение; restore **недоступен не-admin**; после
restore вызывается `ops_check` (или показывается инструкция); загруженный архив с traversal-
записями **не распаковывается**.

---

## 12. Что НЕ делать сейчас

- Не делать restore **одной кнопкой**; не делать import **без** emergency backup; не делать import
  **без** confirm phrase.
- Не давать restore **продавцу/кладовщику/наблюдателю**.
- Не хранить backups/offsite-credentials в **Git**.
- Не делать automatic offsite **без отдельной настройки**.
- Не делать «импорт» как **простую загрузку файла** без проверки manifest/типа/traversal.
- Не менять модели/миграции; не добавлять Celery/Redis/S3/шифрование в этом слое.

---

## 13. Критерии приёмки будущих этапов

**Этап A принят, если:** кнопка экспорта понятная, синяя, с текстом; manual-бэкапы помечены
`manual`; automatic/legacy могут отображаться; restore в UI **не появился**.

**Этап B принят, если:** scheduled backup настраивается; automatic помечены `automatic`; offsite-
статус виден; секреты не в Git.

**Этап C принят, если:** import/restore-wizard **невозможно запустить случайно**; есть большое
предупреждение; есть confirm phrase; создаётся emergency backup **до** restore; restore работает
**только** для admin/superuser; restore **не** делает path traversal; после restore есть
`ops_check`.
