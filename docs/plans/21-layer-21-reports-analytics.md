# План реализации — Слой 21. Отчёты и базовая аналитика (read-only)

**Статус:** УТВЕРЖДЁН (2026-06-26) · все рекомендации приняты · реализация в границах §21.

---

## 1. Цель слоя

Сделать **read-only отчёты** по складу и продажам за выбранный период, чтобы Денис
видел: продажи, выручку, себестоимость, валовую прибыль, движение склада (ремонт/
возврат/списание/инвентаризация) и остатки (включая низкие).

**Слой ничего не меняет:** только читает уже созданные документы (`Sale`,
`RepairOrder`, `StockReturn`, `WriteOffDocument`, `InventoryCountDocument`),
движения (`StockMovement`) и кэш остатков (`StockBalance`). **Не создаёт
`StockMovement`, не меняет `StockBalance`/итоги документов, не делает refund/
проводок.**

**Главная мысль:** скучные, но **правильные** таблицы и KPI. Графики, экспорт
(CSV/Excel), PDF, прогнозы — **будущие слои**, когда цифры уже проверены.

### Что уже есть (читаем; ничего не пишем)

| Источник | Поля для агрегации |
|---|---|
| `Sale` (Слой 16) | `status`, `sold_at`, `revenue_total`, `cost_total`, `profit_total` |
| `SaleLine` | `part_type`, `quantity`, `total_price`, `total_cost_rub`, `profit_rub` |
| `RepairOrder`/`RepairIssueLine` (Слой 17) | `status`, `completed_at`, `cost_total`; `part_type`, `quantity`, `total_cost_rub` |
| `StockReturn`/`StockReturnLine` (Слой 18) | `status`, `completed_at`, `cost_total`; `quantity`, `total_cost_rub` |
| `WriteOffDocument`/`WriteOffLine` (Слой 19) | `status`, `completed_at`, `reason`, `cost_total`; `part_type`, `quantity`, `total_cost_rub` |
| `InventoryCountDocument`/`InventoryCountLine` (Слой 20) | `status`, `completed_at`; `adjustment` → `StockMovement` |
| `StockMovement` | `movement_type`, `quantity`, `total_cost_rub`, `created_at` (**проиндексирован**) |
| `StockBalance` | `part_type`, `location`, `quantity_available/reserved/quarantine` (индекс `(part_type, location)`) |
| `PartType.min_stock_level` | порог низкого остатка |
| `can_view_purchase_cost` | флаг показа себестоимости/прибыли |

### Что нового в Слое 21

- Новое приложение **`apps/reports`** — **без моделей** (read-only): сервисы-
  агрегаторы, вьюхи, шаблоны. **Миграций нет** (приложение моделей не вводит).
- Возможность **`VIEW_REPORTS`**, раздел `/reports/` (дашборд по периоду + остатки).

**Чего слой НЕ делает:** экспорт CSV/Excel, PDF, графики, бухгалтерский P&L,
денежный refund, пересчёт итогов `Sale`, изменение статусов/остатков, создание
`StockMovement`, прогноз закупок/автозаказы, BI-система.

## 4. Главное ограничение — read-only

Слой **только читает**. Запрещено: создавать `Sale`/любые документы; создавать
`StockMovement`; менять `StockBalance`; пересчитывать документы задним числом;
менять финансовые итоги `Sale`; refund; бухгалтерские проводки. Гарантируется тем,
что `apps/reports` **не импортирует сервисы-мутаторы** и не имеет своих моделей;
вьюхи делают только `GET` и рендер. Закрепляется тестом (§20): после открытия любого
отчёта число `StockMovement`/состояние `Sale`/`StockLot`/`StockBalance` не меняется.

## 5. Период и фильтрация

- Параметры: `date_from`, `date_to` (даты, не datetime); быстрые пресеты:
  **сегодня / 7 дней / 30 дней / текущий месяц**. По умолчанию — **последние 30 дней**.
- Таймзона проекта (`USE_TZ`): период `[date_from 00:00, date_to 23:59:59.999999]`
  в локальной TZ через `timezone.make_aware`; фильтр по datetime-полю документа.
- **По какому полю фильтруем (только `completed`/проведённые документы):**

| Документ | Поле периода | Фильтр статуса |
|---|---|---|
| Продажи | **`sold_at`** | `status = completed` |
| Ремонт/выдачи | **`completed_at`** | `status = completed` |
| Возвраты | **`completed_at`** | `status = completed` |
| Списания | **`completed_at`** | `status = completed` |
| Инвентаризация | **`completed_at`** | `status = completed` |

**Обоснование:** считаем по **моменту проведения** (когда событие реально
произошло для бизнеса), а не по `created_at` черновика. Только `completed` —
черновики/отменённые в отчёты не попадают.

- **Остатки (текущие/низкие)** — **точечный** срез (на «сейчас»), период к ним не
  применяется (это состояние, а не поток).

## 6. Продажи

Источник: `Sale` (только `completed`, `sold_at` в периоде) и их `SaleLine`.

| Метрика | Расчёт |
|---|---|
| Выручка | `Sum(Sale.revenue_total)` |
| Себестоимость | `Sum(Sale.cost_total)` *(под `can_view_purchase_cost`)* |
| Валовая прибыль | `Sum(Sale.profit_total)` *(под `can_view_purchase_cost`)* |
| Число продаж | `Count(Sale)` |
| Число строк | `Count(SaleLine)` |
| Кол-во по детали | `SaleLine.values("part_type").annotate(qty=Sum("quantity"))` |
| Топ по выручке | `SaleLine.values("part_type").annotate(rev=Sum("total_price")).order_by("-rev")[:N]` |
| Топ по количеству | `…annotate(qty=Sum("quantity")).order_by("-qty")[:N]` |

Всё через **ORM-агрегацию** (`Sum`/`Count`), без N+1; `part_type` подтягивается
`values()`+именами или `select_related`.

---

## 7. Возвраты — критическое правило (не искажать выручку)

Слой 18 **не меняет** финансовые итоги `Sale` и **не делает денежный refund**.
Поэтому в отчётах:

- показываем **gross sales** (валовую выручку) как есть;
- **возвраты на склад** показываем **отдельным блоком**: `returned_quantity`,
  `returned_cost` (`Sum(StockReturn.cost_total)` по проведённым в периоде);
- **НЕ вычитаем** возвраты из выручки автоматически — денежного возврата нет, и
  такое вычитание создало бы **ложную финансовую картину**.
- В блоке возвратов — пояснение: «возврат на склад ≠ возврат денег; на выручку не
  влияет (финансовое сторно — будущий слой)».

## 8. Ремонт / установки

Источник: `RepairOrder` (`completed`, `completed_at` в периоде) и `RepairIssueLine`.

| Метрика | Расчёт |
|---|---|
| Число заказов | `Count(RepairOrder)` |
| Себестоимость выданного | `Sum(RepairOrder.cost_total)` *(под `can_view_purchase_cost`)* |
| Топ деталей в ремонт | `RepairIssueLine.values("part_type").annotate(qty=Sum("quantity"))` |

**Без выручки/прибыли** — Слой 17 не содержит цены работ (это складская выдача, не
коммерческий документ). Показываем себестоимость выданного и натуральные объёмы.

---

## 9. Списания

Источник: `WriteOffDocument` (`completed`, `completed_at` в периоде) и `WriteOffLine`.

| Метрика | Расчёт |
|---|---|
| Число списаний | `Count(WriteOffDocument)` |
| Себестоимость потерь | `Sum(WriteOffDocument.cost_total)` *(под `can_view_purchase_cost`)* |
| По причине | `WriteOffDocument.values("reason").annotate(c=Count, cost=Sum("cost_total"))` |
| Топ деталей по списаниям | `WriteOffLine.values("part_type").annotate(qty=Sum("quantity"), cost=Sum("total_cost_rub"))` |

Это **потери по себестоимости**, а не бухгалтерская аналитика.

---

## 10. Инвентаризация (корректировки)

Источник: `InventoryCountDocument` (`completed`, `completed_at` в периоде), их строки
и связанные движения `InventoryCountLine.adjustment` (`StockMovement` ADJUST_*).

| Метрика | Расчёт |
|---|---|
| Число инвентаризаций | `Count(InventoryCountDocument)` |
| Σ ADJUST_IN (кол-во/₽) | по `adjustment.movement_type == ADJUST_IN` (`Sum(quantity)`, `Sum(total_cost_rub)`) |
| Σ ADJUST_OUT (кол-во/₽) | по `ADJUST_OUT` аналогично |
| Последние корректировки | список строк с `difference` и движением |

**Не смешиваем** с списаниями: корректировки — это `ADJUST_*` через
`document_type="inventory_count"`, списания — `WRITE_OFF_*`. Разные потоки, разные
блоки отчёта.

---

## 11. Текущие остатки

Источник: **`StockBalance`** (read-optimized кэш; **только читаем**).

| Метрика | Расчёт |
|---|---|
| Деталей с остатком | `StockBalance.filter(quantity_physical>0).values("part_type").distinct().count()` |
| Σ доступно | `Sum(quantity_available)` |
| Σ зарезервировано | `Sum(quantity_reserved)` |
| Σ карантин | `Sum(quantity_quarantine)` |
| По ячейкам | `values("location").annotate(...)` |

При сомнении в кэше — есть `check_stock_balance` (отдельная команда); отчёт его не
вызывает, но мы документируем, что источник истины — первичка, а `StockBalance` —
кэш (его не меняем).

---

## 12. Низкие остатки

- Порог — `PartType.min_stock_level` (есть в каталоге; `default 0`).
- Доступное по детали — `Sum(StockBalance.quantity_available)` группировкой по
  `part_type`.
- Показываем `PartType`, текущее `available`, `min_stock_level` для строк, где
  `min_stock_level > 0` и `available < min_stock_level`.
- **Никаких автозаказов/партий** — только индикатор.

## 13–14. Права — **рекомендация (финальное решение)**

Новая возможность **`VIEW_REPORTS`**. Доступ к `/reports/` — под ней. **Денежные**
блоки (выручка/себестоимость/прибыль продаж, себестоимость списаний/ремонта/
возвратов/корректировок) дополнительно гейтятся **`can_view_purchase_cost`**.

| Роль | `/reports/` (VIEW_REPORTS) | Видит денежные блоки (purchase_cost) | Что фактически видит |
|---|---|---|---|
| Администратор | ✅ | ✅ | всё |
| Руководитель | ✅ | ✅ | всё |
| **Кладовщик** | ✅ | **нет** | складские отчёты **без финансов** (остатки, низкие, объёмы, число документов) |
| Наблюдатель | ✅ | ✅ (роль уже имеет `purchase_cost`) | всё (read-only наблюдатель) |
| Продавец/Мастер | ❌ | — | раздела отчётов **нет** на Слое 21 |

**Обоснование (элегантно, без раздувания).** Одна новая capability `VIEW_REPORTS`
открывает раздел; **существующий** флаг `can_view_purchase_cost` отделяет финансы.
Кладовщик получает `VIEW_REPORTS`, но без `purchase_cost` → видит **складские**
отчёты (остатки/низкие/объёмы/счётчики), а денежные суммы скрыты — ровно «складские
отчёты без финансов продаж». Продавцу/Мастеру `VIEW_REPORTS` **не даём** (общий
раздел отчётов им на этом слое не нужен). Наблюдатель уже финансово-смотрящая роль —
видит всё.

- `roles.py`: `VIEW_REPORTS` для `ADMIN`/`MANAGER`/`STOREKEEPER`/`VIEWER`.
- `accounts/models.py`: `can_view_reports`. Без миграции (возможности — код).
- Доступ к вьюхам — `login_required` + `can_view_reports`; денежные поля в шаблоне —
  под `show_costs = can_view_purchase_cost`.

> Примечание: «выручка» в денежном блоке тоже под `can_view_purchase_cost` — это
> простой единый флаг «видеть деньги»; отдельную «видеть выручку без себестоимости»
> градацию не вводим (не усложняем слой).

## 17. Производительность

- Только **ORM-агрегация** (`Sum`/`Count`/`values().annotate()`); без N+1;
  `select_related`/`values` для деталей.
- **Индексы — не добавляем в этом слое** (premature optimization): объёмы одной
  точки (один магазин запчастей) малы; `StockMovement.created_at` уже индексирован,
  `StockBalance` имеет индекс `(part_type, location)`.
- **Если** в будущем вырастут объёмы — кандидаты на индексы: `Sale.sold_at`,
  `RepairOrder.completed_at`, `*.completed_at`, `*.status` (отдельным слоем-миграцией,
  по факту замеров). Сейчас — **без миграций** (reports вообще без моделей).

## 18. Сервисы `apps/reports/services.py`

Вьюхи только вызывают сервисы и рендерят; вся агрегация — в сервисах (тестируемо).

```python
get_sales_report(date_from, date_to)        -> SalesReport
get_repair_report(date_from, date_to)       -> RepairReport
get_returns_report(date_from, date_to)      -> ReturnsReport
get_writeoff_report(date_from, date_to)     -> WriteoffReport
get_adjustments_report(date_from, date_to)  -> AdjustmentsReport
get_stock_report()                          -> StockReport          # точечный
get_low_stock_report()                      -> list[LowStockRow]    # точечный
get_dashboard_report(date_from, date_to)    -> DashboardReport      # сборка периодных
resolve_period(request_GET) -> (date_from, date_to, preset)         # парсинг/валидация
```

- Сервисы считают **полные** данные (включая денежные); **показ** денежных полей
  решает шаблон по `show_costs` (как во всех слоях). Это упрощает сервисы и тесты.
- Чистые функции от `(date_from, date_to)` — без побочных эффектов, без записи.

## 19. Формат данных

- Каждый отчёт — простая **dataclass** (`@dataclass`) с полями-числами и списками
  строк (`list[dict]`/маленькие dataclass для строк топов).
- **HTML-агностично:** структуры удобны и для шаблона, и для тестов (ассертим поля,
  не парся HTML, кроме тестов на скрытие денег).
- Денежные значения — `Decimal`, округление `money()`.

## 20. Тесты (`tests/test_reports.py`)

1. Продажи считаются только `completed`; черновик/отменённая — не попадают.
2. Продажа **вне периода** (по `sold_at`) — не попадает.
3. `revenue/cost/profit` суммируются корректно (по нескольким продажам).
4. Топ деталей по выручке/количеству — правильный порядок.
5. Прибыль/себестоимость **скрыты** в HTML без `can_view_purchase_cost`.
6. **Возвраты** показаны отдельным блоком и **не уменьшают** выручку.
7. Ремонт считается по `completed_at`; вне периода — не попадает.
8. Списания группируются по `reason`; суммы по причине верны.
9. Корректировки инвентаризации **отделены** от списаний (ADJUST_* vs WRITE_OFF_*).
10. Низкие остатки: попадают `available < min_stock_level` (и `min_stock_level>0`).
11. Текущие остатки: суммы available/reserved/quarantine из `StockBalance`.
12. **Read-only:** открытие отчётов не создаёт `StockMovement`.
13. **Read-only:** не меняет `Sale`/`StockLot`/`StockBalance` (счётчики/значения до=после).
14. Пользователь без `VIEW_REPORTS` (Продавец) → `/reports/` **403**.
15. Пользователь с `VIEW_REPORTS` (Кладовщик) → 200, но денежных блоков не видит.
16. Руководитель → 200 и видит деньги.
17. Вьюха использует сервисы (`get_dashboard_report` и т.п.) — мок/вызов.

    нормализуется; невалидный ввод не роняет страницу (200).

## 22. Ручная проверка

1. Руководителем открыть `/reports/` → период «30 дней»; убедиться: число продаж,

3. Создать тестовую продажу/списание/возврат/инвентаризацию (через их слои) →

## 23. Критерии готовности

1. `/reports/` (дашборд по периоду) и `/reports/stock/` (остатки) работают; данные
   считаются сервисами `apps/reports`, вьюхи только рендерят.
2. Продажи/ремонт/возвраты/списания/инвентаризация фильтруются по правильному полю
   времени и только `completed`; вне периода не попадают.

4. Корректировки инвентаризации отделены от списаний.

   (Админ/Руководитель/Кладовщик/Наблюдатель; Продавец/Мастер — нет).
7. **Read-only:** отчёты не создают движений и не меняют документы/остатки
   (тесты §20.12–13).
8. Границы соблюдены: нет экспорта/PDF/графиков/проводок/refund.
9. Тесты зелёные (вкл. регресс прошлых слоёв); `ruff`/`djlint` чисты;
   `manage.py check` ок; `makemigrations --check` — **изменений нет** (reports без
   моделей; capability — код, без миграции).
