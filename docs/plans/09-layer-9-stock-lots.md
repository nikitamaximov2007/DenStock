# План реализации — Слой 9. Количественный учёт (StockLot)

**Статус:** на согласовании (2026-06-24) · **Код не пишется до утверждения.**

## 1. Цель слоя

Научиться создавать **количественные складские лоты** (`StockLot`) для деталей с
`tracking_mode=bulk` из уже **финансово закрытых** строк партии
(`batch.cost_finalized=true`): количество в конкретной ячейке с замороженной
landed-себестоимостью за единицу.

Это **ещё не ledger**: `StockMovement` (Слой 10) и read-оптимизированный
`StockBalance` (Слой 10) **не создаются**; доступность к продаже, продажи и
резервы — не реализуются. `StockLot` хранит **текущее** количество лота; полный
журнал движений и баланс остаются отдельным следующим слоем.

## 2. Где размещаем модель — `apps/inventory`

**Решение: `StockLot` в существующем `apps/inventory`** (рядом с `PartItem`).

*Обоснование.* `StockLot` — количественный «двойник» `PartItem`: оба суть
физический складской остаток, оба создаются из закрытой `BatchLine`, оба будут
питать единые `StockMovement`/`StockBalance` (Слой 10) и резолв сканера (Слой 11).
Держать поштучный и количественный учёт в одном домене `inventory` правильно;
новое приложение плодить незачем.

## 3. Модель `StockLot`

| Поле | Тип | Назначение / правила |
|---|---|---|
| part_type | FK→catalog.PartType (PROTECT) | только `tracking_mode=bulk` (п.5) |
| batch | FK→procurement.Batch (PROTECT) | = `batch_line.batch`, ставит сервис |
| batch_line | FK→procurement.BatchLine (PROTECT), related_name=`lots` | источник лота и себестоимости |
| location | FK→warehouse.StorageLocation (PROTECT), **обязателен** | `storage_allowed=true` и `is_active=true` (п.5) |
| quantity | Decimal(12,3) | текущее количество; DB `CHECK ≥ 0`; на создание `> 0` (п.5) |
| initial_quantity | Decimal(12,3) | «как принято»; на Слое 9 = quantity (п.5) |
| landed_unit_cost_rub | Decimal(12,2), editable=False | копия `batch_line.landed_unit_cost_rub` |
| status | CharField(20), choices | статусы — п.4, default `receiving` |
| note | CharField(255), blank | примечание |
| created_at | DateTimeField(auto_now_add) | |
| updated_at | DateTimeField(auto_now) | |

**Ограничения Meta:**
- `CheckConstraint(quantity >= 0)` и `CheckConstraint(initial_quantity >= 0)`.
- `UniqueConstraint(batch_line, location)` — один лот на пару «строка × ячейка»;
  разные ячейки → разные лоты (разбиение строки). Реализует замысел
  `uniq(part, batch, location)` из `02-data-model.md`, но точнее: учитывает
  случай двух строк одной детали в одной партии.

> **`location` обязателен** (в отличие от `PartItem.current_location`). Лот по
> смыслу — «количество детали в ячейке»; без ячейки это просто нераспределённый
> остаток строки, который уже виден через прогресс распределения (п.7).

> **`initial_quantity` — включаем.** На Слое 9 движений нет, поэтому при
> создании/правке `initial_quantity = quantity`. Поле становится осмысленным со
> Слоя 10 (расход через движения замораживает `initial_quantity`, а `quantity`
> уменьшается). Заводим сейчас, чтобы не делать потом миграцию с backfill.
> *(Если предпочитаете минимализм — можно отложить до Слоя 10; см. открытые вопросы.)*

## 4. Статусы `StockLot`

Нужный минимум на этом слое — те же осмысленные состояния, что у `PartItem`:

- **`receiving`** (default) — лот заведён на приёмке, ещё не подтверждён.
- **`available`** — подтверждён в ячейке.
- **`quarantine`** — отложен.
- **`written_off`** — в choices как зарезервированный статус; **выставляется
  Слоем 19** (списание) через движения, **на Слое 9 переходов в него нет**.

**`depleted` — не вводим на этом слое.** «Исчерпан» — следствие расхода до нуля,
а расход появляется только с движениями (Слой 10). На Слое 9 `quantity > 0`
всегда. `depleted` логично добавить вместе с ledger на Слое 10.

**Разрешённые ручные переходы (Слой 9, `MANAGE_INVENTORY`):**
`receiving → available`, `receiving → quarantine`, `available ↔ quarantine`.

## 5. Правила (инварианты)

1. **Только bulk.** `StockLot` создаётся лишь для `PartType.tracking_mode = bulk`
   (поштучные — это `PartItem`, Слой 8).
2. **Только из закрытой партии.** Запрет, если `batch_line.batch.cost_finalized=false`.
3. **Себестоимость заморожена.** `landed_unit_cost_rub = batch_line.landed_unit_cost_rub`.
4. **Ячейка.** `location.storage_allowed=true` и `location.is_active=true`
   (через `can_hold_stock()`), иначе ошибка.
5. **Лимит по строке.** `Σ quantity лотов по batch_line ≤ batch_line.quantity`
   (проверка под `select_for_update` на строке; при правке — исключая сам лот).
6. **Неотрицательность.** DB `CHECK quantity >= 0` (на будущее — расход до нуля);
   **создание/правка на этом слое требует `quantity > 0`** (нулевые лоты не заводим).
7. **Тип количества — `Decimal(12,3)`** (как у `BatchLine.quantity`).
   *Обоснование:* единицы измерения детали (`PartType.unit`) бывают и штучные
   (шт/компл — целые), и непрерывные (**м/кг/л — дробные**). Integer исключил бы
   метры/килограммы/литры, поэтому только Decimal. **Точность — 3 знака**
   (совпадает с `BatchLine.quantity`, сходимость лимита п.5 точная). Округления
   на этом слое нет: количество вводит человек и хранится как есть; для штучных
   единиц значение просто целое (`5.000`), UI покажет без лишних нулей.

## 6. Граница слоя (важно)

- `StockLot` хранит **текущее** количество лота, но **`StockMovement` не
  создаётся** (Слой 10).
- **`StockBalance` не создаётся** (Слой 10).
- **Доступность к продаже не реализуется.**
- **Продажи и резервы не реализуются.**

То есть лот существует как количественный остаток в ячейке, но любое изменение
ещё **не** проходит через журнал движений — это Слой 10.

## 7. Связь с `BatchLine`

- **Прогресс распределения:** «распределено N из {quantity}» = `Σ lots.quantity`
  по строке против `batch_line.quantity`.
- Можно создать **один лот на всю строку** (на оставшееся количество).
- Можно создать **несколько лотов из одной строки в разные ячейки** — пока сумма
  не превышает `batch_line.quantity`.
- Когда распределено = `quantity`, кнопки создания скрываем.

## 8. Экраны

- **Список лотов** (`lot_list`): таблица (деталь, ячейка, количество, статус,
  партия; себестоимость — при праве). Фильтры: статус, деталь, партия, ячейка.
- **Карточка лота** (`lot_detail`): все поля, источник (партия/строка),
  себестоимость (при праве).
- **Создание из строки** (`lot_create` от `batch_line`): ячейка, количество,
  примечание.
- **Быстрое создание остатка** (`lot_create_remaining`): один лот на всё
  **оставшееся** количество строки в выбранную ячейку.
- **Редактирование** (`lot_edit`): `location`, `quantity`, `status`, `note` —
  **пока нет движений** (на этом слое всегда; со Слоя 10 правка количества
  блокируется, если по лоту есть движения).
- **На карточке партии/строки**: блок «Количественные лоты» со списком и
  прогрессом распределения N/quantity; кнопки создания при `MANAGE_INVENTORY` и
  `cost_finalized`.

## 9. Права

| Роль | Доступ к лотам |
|---|---|
| Администратор | управление + себестоимость |
| Руководитель | управление + себестоимость |
| Кладовщик | **создание/правка** (приёмка); **без сумм** |
| Продавец/Мастер | раздел **не виден** (наличие для продаж — слои 13/16) |
| Наблюдатель | только просмотр (с суммами) |

- Управление под **`MANAGE_INVENTORY`** (та же возможность, что у `PartItem`,
  Слой 8) — новых прав не вводим.
- Просмотр раздела — `can_manage_inventory or is_viewer` (как у экземпляров).
- **Себестоимость скрыта** без `can_view_purchase_cost`.

## 10. Обязательные тесты (`tests/test_stock_lots.py`)

1. `StockLot` создаётся для `bulk` PartType.
2. Нельзя создать `StockLot` для `serial` PartType — отказ.
3. Нельзя создать из партии без `cost_finalized` — отказ.
4. `landed_unit_cost_rub` копируется из `batch_line.landed_unit_cost_rub`.
5. `location` должен быть `storage_allowed=true` и `is_active=true` (иначе отказ).
6. `quantity` не может быть отрицательным или нулевым (отказ при `≤ 0`).
7. Нельзя создать лоты суммарно больше `batch_line.quantity`.
8. Можно разбить `batch_line` на несколько `StockLot` в **разные ячейки**.
9. Прогресс распределения по `batch_line` считается верно.
10. Себестоимость скрыта без `can_view_purchase_cost`.
11. Права ролей работают (кладовщик создаёт; наблюдатель только смотрит).
12. Продавец/Мастер раздел не видит (403 на список; нет в навигации).
13. `StockMovement`/`StockBalance` **не создаются** (приложений/таблиц нет —
    проверяем по сути, как в обновлённом тесте Слоя 8).

## 11. Ручная проверка

1. Войти администратором → партия с **bulk**-строкой → зафиксировать себестоимость.
2. Создать лот на часть количества в ячейку (только `storage_allowed`); создать
   второй лот на остаток в другую ячейку. Прогресс «распределено N из quantity».
3. Попробовать превысить `quantity` → отказ; нулевое/отрицательное количество →
   отказ; ячейка с `storage_allowed=false` → отказ.
4. Войти кладовщиком → может создавать/править, **сумм не видит**.
5. Войти продавцом → раздела лотов нет.
6. Убедиться, что **движений/баланса не появилось**.

## 12. Критерии готовности

1. Создаются количественные лоты из закрытой партии для `bulk`; для `serial` и
   незакрытой партии — отказ.
2. `landed_unit_cost_rub` заморожен из `batch_line`.
3. `location` — только `storage_allowed=true` и `is_active=true`.
4. `Σ quantity` по строке ≤ `batch_line.quantity`; разбиение по ячейкам работает;
   прогресс корректен.
5. `quantity > 0` на создании; DB `CHECK ≥ 0`.
6. Права: управление — `MANAGE_INVENTORY`; суммы — `can_view_purchase_cost`;
   продавец раздел не видит.
7. `StockMovement`/`StockBalance`/сканер/продажи/резервы/аналитика/PDF — **нет**;
   логика `PartItem` не меняется (кроме общих ссылок при необходимости).
8. Все тесты зелёные; `ruff`/`djlint` чисты; миграции с нуля;
   `makemigrations --check` чисто.

## Границы слоя (чего НЕ делаем)

- Не реализуем `StockMovement` и `StockBalance`.
- Не реализуем единый резолв сканера.
- Не реализуем продажи, резервы, установки, возвраты.
- Не реализуем аналитику и PDF-этикетки.
- Не меняем логику `PartItem` (только общие ссылки/навигация при необходимости).
- Лот — это **текущий количественный остаток в ячейке**, но **не ledger**:
  журнал движений и баланс остаются Слоем 10.
