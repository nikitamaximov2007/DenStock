# План v1.1.7 — Backup UI + scheduled offsite backups

**Статус:** ПЛАН (реализацию не начинаем без «go»). Только проектирование. В этом шаге код/
модели/миграции/views/templates/urls **не создаём**; существующие backup/restore-команды **не
меняем**; складскую логику не трогаем; файлы бэкапов не коммитим.

**Ключевая мысль:** ручной backup через UI **не спасает**, если сервер уже умер — кнопку «Создать
бэкап» будет негде нажать. Настоящий disaster recovery = **(1)** автоматические бэкапы **до**
аварии, **(2)** копия **вне** основного сервера (offsite), **(3)** понятное восстановление на
новом сервере. UI — это удобство и контроль, **а не единственная защита**.

---

## 1. Какие аварии закрываем

| Сценарий | Локальный backup (`backups/`) | UI backup | Нужен offsite? | Как восстанавливаться |
|---|---|---|---|---|
| Приложение/контейнер сломался, VPS жив | ✅ помогает | ✅ (сайт жив) | желательно | пересобрать контейнер; данные в томах целы; при нужде — `restore_*` из локального `backups/` |
| Повреждён `pgdata`/media-том | ✅ если backup свежий | ⚠️ если сайт ещё поднимается | **да** | `restore_db --yes` + `restore_media --yes` из последнего backup |
| **VPS полностью умер** | ❌ погиб вместе с диском | ❌ нажать негде | **обязателен** | новый VPS → скачать offsite backup → restore (§8) |
| Провайдер недоступен | ❌ | ❌ | **обязателен** | развернуть у другого провайдера из offsite |
| Случайное удаление данных | ✅ (откат к вчерашнему) | ✅ | желательно | `restore_db`/`restore_media` из подходящей точки |
| Обновление сломало проект | ✅ (backup перед обновлением) | ✅ | желательно | откат кода + `restore_db` при нужде |
| Переезд на новый хост | ✅ как перенос | частично | **да** (удобнее через offsite) | §8 restore workflow |

**Вывод:** три сценария (**смерть VPS / недоступность провайдера / переезд**) **невозможно**
закрыть без offsite. Именно они и есть настоящий disaster recovery.

---

## 2. Почему локального backup недостаточно

- `backups/` лежит **на том же диске/VPS**. Умер диск/VPS — умер и backup.
- **UI-кнопка** работает только пока **сайт и сервер живы**; в аварии её нет.
- Настоящий DR = **scheduled backup** (создаётся сам, до аварии) **+ offsite copy** (живёт
  отдельно от сервера).
- Без offsite при смерти сервера **восстанавливать может быть нечего**.

## 3. Что уже есть в проекте (факт из кода, Слой 25)

`apps/operations/`:
- Команды: **`backup_db`**, **`backup_media`**, **`backup_all`**, **`restore_db`**,
  **`restore_media`**, **`ops_check`** (`apps/operations/management/commands/*`).
- `backup.py`: `backup_root()` (= `settings.BACKUP_ROOT`, дефолт `BASE_DIR/backups`),
  `new_run_dir()` (`backups/<timestamp>/`), `backup_all(root, keep_last, ...)` — пишет
  `db.dump`|`db.sqlite3` + `media.tar.gz` + **`manifest.json`**; `restore_db/restore_media`
  (**требуют `--yes`**); `prune_old_runs()` (retention через `--keep-last`).
- **manifest.json** (реальные поля): `created_at`, `engine`, `db_file`, `media_file`,
  `version`, `git_commit`. **Секретов в манифесте нет** (пароль БД идёт через `PGPASSWORD`).
- `checks.py`: `ops_check` (БД, писабельность `MEDIA_ROOT`/`BACKUP_ROOT`, доступность
  `pg_dump`/`pg_restore`).
- **Docker**: `./backups:/app/backups` — bind-mount → **бэкапы уже видны на хосте**;
  `media` — именованный том; Caddy отдаёт `/media/*`.
- `.gitignore`: `/backups/`, `*.dump`, `*.sql.gz` — бэкапы не коммитятся.
- README (Слой 25) описывает CLI backup/restore/ops_check.

> План опирается только на эти реальные команды/функции. Новых backup-функций не выдумываем — UI
> будет **тонкой обёрткой** над существующим `apps/operations`.

## 4. Backup UI (будущий раздел)

Новый пункт левого меню **«Бэкапы»** (или «Операции»), только для owner/admin (§10).

Блоки страницы `templates/operations/backups.html`:
- **Список локальных бэкапов** — каталоги `backups/<timestamp>/` (дата, размер, наличие
  db/media/manifest).
- **Кнопка «Создать полный бэкап»** — POST → вызывает существующий `backup.backup_all()`
  (та же логика, что CLI).
- **«Скачать»** — отдать `db.*`/`media.tar.gz`/`manifest.json` **только** из backup-каталога
  (защита от path traversal, §11).
- **«Проверить manifest»** — прочитать/показать `manifest.json` (дата, engine, версия, commit,
  файлы) **без** восстановления.
- **Статус последнего локального backup** (дата/размер/manifest).
- **Статус последнего offsite backup/sync** (читается из status-файла, который пишет хостовый
  offsite-скрипт, §7 — Django сам offsite не делает).
- **Предупреждение**: «Локальный backup не заменяет offsite. При смерти сервера нужен offsite».
- **Инструкция восстановления на новом сервере** (текст + CLI-команды, §8).

> UI = удобство/контроль. Он **не** является единственной защитой от смерти сервера.

## 5. Scheduled backups (на уровне хоста, не Django)

Планировщик — **на хосте**, чтобы backup был **независим от веб-интерфейса**:
- **cron** или **systemd timer** на VPS, команда:
  `docker compose exec -T web python manage.py backup_all --keep-last N`
- Расписание — напр. **ежедневно 03:00**.
- **Retention**: daily/weekly/monthly (базово — `--keep-last` уже есть; расширенная политика
  weekly/monthly — хостовый скрипт, ротация каталогов).
- **Логирование** результата (stdout/stderr в лог-файл на хосте).
- **Проверка, что backup реально создан** (существует свежий `backups/<ts>/manifest.json`).
- **Alert/status**, если backup не создался (простой: писать status-файл + опц. уведомление —
  провайдер-специфично, future).

> Планировщик и offsite — **host-level**, не в Django и не в вебе. Так backup продолжает
> работать, даже если веб-приложение лежит.

## 6. Offsite storage — критерии выбора (провайдера сейчас НЕ реализуем)

| Вариант | Плюсы | Минусы |
|---|---|---|
| Второй VPS (rsync/scp) | просто, дёшево, свой контроль | нужен второй сервер, ручная настройка |
| S3-совместимое (generic) | стандарт, retention/versioning | нужен доступ/ключи, стоимость |
| Cloudflare R2 | без egress-платы, дёшево | доступность из РФ уточнить |
| Yandex Object Storage | доступно из РФ, S3-совместимо | привязка к провайдеру |
| rclone → Google Drive/Dropbox/Яндекс.Диск | дёшево/бесплатно на старте, гибко | лимиты API, надёжность варьируется |
| Ручное скачивание | нулевая настройка | **не** DR (человек может забыть) — только временно |

**Критерии выбора (зафиксировать перед реализацией):** цена; простота настройки; надёжность;
доступность из РФ/Европы; скорость восстановления; поддержка retention; безопасность секретов
(ключи только на сервере). **Конкретный провайдер — отдельное решение/слой.**

## 7. Offsite sync workflow (host-level скрипт после backup_all)

Последовательность (скрипт на хосте, **не** Django):
1. Найти созданный `backups/<ts>/`.
2. Проверить `manifest.json` (файлы на месте, дата свежая).
3. Отправить архив во внешнее хранилище (rsync/rclone/s3 — по выбору §6).
4. Проверить, что файл **появился** во внешнем хранилище.
5. Записать **status/log** (напр. `backups/offsite_status.json`: время, назначение, ok/fail) —
   этот файл читает UI (§4) для «статус offsite».
6. Применить **retention** offsite (daily/weekly/monthly).
7. **Не удалять последний рабочий backup** (ни локально, ни offsite).
8. **Ошибки логировать отдельно** (и, опц., алертить).

**Безопасность:**
- Ключи offsite — **только** в env/secret-файлах сервера, **не в Git**.
- **Не** включать `.env`/секреты в архив (в бэкапе — БД + media; секреты хранятся отдельно).
- **Шифрование архива** — отдельный будущий слой (GPG/age), не в этом.

## 8. Restore workflow на новом сервере (инструкция)

Базовый сценарий (полная потеря сервера, есть offsite):
1. Поднять новый VPS; установить Docker + Git.
2. `git clone` проекта.
3. Создать `.env` (секреты — заново, из безопасного места; не из backup).
4. `docker compose up -d --build`.
5. Скачать **последний backup** из offsite в `./backups/<ts>/`.
6. `docker compose exec web python manage.py restore_db <...>/db.dump --yes`.
7. `docker compose exec web python manage.py restore_media <...>/media.tar.gz --yes`.
8. `docker compose exec web python manage.py ops_check`.
9. Проверить: вход в систему, фото (`/media/`), справочники/остатки/движения.

Три подсценария:
- **Старый сервер жив, сайт сломан:** починить контейнер/откатить код; restore из локального
  `backups/` при нужде.
- **Старый сервер полностью умер:** только новый сервер + offsite (шаги 1–9).
- **Есть только offsite backup:** тот же путь (1–9); проверить целостность manifest перед restore.

## 9. Web restore safety (первый UI-слой — БЕЗ destructive restore)

**Через UI МОЖНО:** загрузить backup-архив в staging; **проверить manifest**; показать
содержимое/дату/размер/версию; **показать CLI-команды** восстановления.

**Через UI НЕЛЬЗЯ (в первом слое):** кнопка «затереть текущую БД»; запуск `restore_db`/
`restore_media`; перезапись media; восстановление поверх работающей системы.

**Почему:** web-restore может затереть текущую БД, оборвать активные сессии, упасть на середине,
перезаписать media или стать разрушительной операцией при неправильном доступе. Restore остаётся
**осознанной CLI-операцией под `--yes`** (уже так в коде).

## 10. Права доступа

- Раздел — **только owner/admin**. В `apps/accounts/roles.py` **нет** capability для backup/
  operations (есть `MANAGE_USERS`, `is_admin`, `is_superuser`, и др.).
- **Рекомендация для реализации (без миграций):** либо гейтить первый слой по **`is_admin`/
  `is_superuser`**, либо ввести **`MANAGE_BACKUPS`** как **код-онли** capability (по образцу
  `PRINT_LABELS`/`MANAGE_IMAGES` — строка в `roles.py`, без миграции). В **этом** плане роли и
  миграции **не меняем** — только фиксируем как пункт будущей реализации.
- **Продавцу/Кладовщику/Наблюдателю доступа НЕ давать.**

## 11. Техническая реализация будущего UI-слоя (проект)

- **Views** — в `apps/operations` (новые): список/создание/скачивание/валидация manifest.
- **Template** — `templates/operations/backups.html`; **route** в urls; **пункт левого меню**
  (через `apps/accounts/context_processors.py`, под правом §10).
- **Выдача файлов** — только из `backup.backup_root()`; **защита от path traversal** (резолв
  пути + проверка `is_relative_to(backup_root)`, как в `apps/operations`/tools уже сделано).
- **CSRF**; **POST-only** для создания backup; **upload staging** только для проверки (не restore);
  **validate manifest** без восстановления.
- **Status-cards**: последний локальный backup (из `backups/`), последний offsite (из
  `offsite_status.json`).
- **Без Celery/Redis** в первом слое.
- **Долгие операции честно помечать как потенциально медленные.** Риск: `backup_all` из web —
  **синхронный** запрос может долго висеть/таймаутить (особенно media). Варианты первого слоя:
  (а) manual POST c явным предупреждением «может занять время»; (б) тонкий wrapper, который просто
  запускает `backup_all` и показывает результат. **Async/очередь — future** (тогда и Celery/RQ).

## 12. Тесты будущей реализации

- Раздел backup доступен **только owner/admin**; обычный пользователь **не видит** и получает 403.
- GET списка показывает backup-файлы из `backups/`.
- POST create вызывает **существующую** `backup.backup_all()` (не дублирует логику).
- Download отдаёт **только** файлы из backup-каталога; **path traversal запрещён** (403/404).
- Upload-validate **не** вызывает restore.
- `restore_db`/`restore_media` **не вызываются** из web ни при каких действиях UI.
- Валидация manifest работает (парсит `manifest.json`, показывает поля).
- Backup-файлы **не попадают в Git** (в `.gitignore`).
- **Складская/бизнес-логика не затронута** (no StockMovement/StockBalance changes).

## 14. Критерии приёмки будущей реализации

1. Manual backup можно **создать и скачать** из UI.
2. UI **явно** показывает: локальный backup **не заменяет** offsite.
3. Есть **статус последнего локального** backup.
4. Есть **место/план для статуса последнего offsite** sync (читается из status-файла).
5. **Automatic backup** можно настроить на сервере (cron/systemd, host-level).
6. **Offsite workflow** описан (§7) и выполним host-скриптом.
7. **Restore на новый сервер** описан (§8) и воспроизводим по инструкции.
8. UI **не может** стереть текущую БД (нет web-restore).
9. Backup-файлы **не попадают в Git**.
10. Тесты зелёные; **без миграций**, если возможно (capability — код-онли или `is_admin`).
11. **Нет** изменений stock/inventory/business-логики.
