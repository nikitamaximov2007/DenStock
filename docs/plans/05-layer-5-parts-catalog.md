# План реализации — Слой 5. Каталог видов деталей

**Статус:** на согласовании (2026-06-24) · **Код не пишется до утверждения.**

## 1. Цель слоя

Создать **каталог видов деталей** (`PartType`) с номерами, заводскими
штрихкодами и совместимостью с техникой — справочную основу, по которой потом
будут приходоваться партии. **Это карточка вида детали, а не остаток.**

## 2. Что такое карточка вида детали

`PartType` описывает **вид/модель детали** («Топливный насос Yamaha»), а не
конкретный физический экземпляр и не количество на складе.

- Физических экземпляров (`PartItem`), партий, остатков и себестоимости здесь
  **нет** — они появятся на слоях 6–12.
- Одна карточка → много экземпляров и партий в будущем.

## 3. Модели (в приложении `catalog`)

- **`PartType`** — карточка вида детали.
- **`PartNumber`** — номера: OEM, артикул, аналог, внутренний справочный.
- **`PartBarcode`** — заводские штрихкоды (несколько на деталь, штрихкод уникален).
- **`PartCompatibility`** — связь детали с моделью техники (+ опц. годы, комментарий).

> **`PartImage` (фото) — откладывается.** Требует Pillow и загрузки файлов; в
> отображении карточки (п.10) фото нет. Добавлю отдельным небольшим слоем перед
> поиском (слой 13), где нужны миниатюры. Поле `02-data-model.md` это допускает.

## 4. Поля `PartType`

| Поле | Тип | Примечание |
|---|---|---|
| name | char(200) | `idx` (поиск по названию) |
| category | FK→catalog.Category | `on_delete=PROTECT` |
| manufacturer | FK→catalog.Manufacturer ✓null | `on_delete=PROTECT` |
| unit | FK→catalog.Unit | `on_delete=PROTECT` |
| tracking_mode | enum | `SERIAL` поштучный / `BULK` количественный |
| description | text blank | |
| recommended_price | decimal(12,2) ✓null | рекомендуемая цена продажи |
| min_price | decimal(12,2) ✓null | минимальная цена продажи |
| min_stock_level | decimal(12,3) | минимальный остаток, по умолч. 0 |
| is_active | bool | деактивация вместо удаления |
| created_at / updated_at | datetime | |

## 5. Важное по ценам

- `recommended_price` и `min_price` хранятся **в карточке детали**.
- **Закупочная себестоимость в `PartType` не хранится** — она появится в партиях
  (`BatchLine`) и остатках (`StockLot`/`PartItem`) на следующих слоях.
- **Прибыль на этом слое не считается.**

## 6. Хранение номеров (`PartNumber`)

| Поле | Тип | Примечание |
|---|---|---|
| part | FK→PartType | `related_name="numbers"` |
| value | char(100) | `idx` (поиск по номеру) |
| kind | enum | `OEM` / `ARTICLE` (артикул) / `ANALOG` (аналог) / `INTERNAL_REF` (внутренний справочный) |
| is_primary | bool | пометка основного OEM (для отображения/поиска) |
| note | char(255) blank | |

- Номер **не глобально уникален**: одинаковые детали могут иметь один OEM
  (ТЗ 5.1). Уникальность не навязываем; используем индекс для поиска.
- Основной OEM помечается `is_primary=true` (вместо отдельного поля
  `primary_oem` в `PartType` — нормализованнее; отмечу соответствие в `02`).

## 7. Хранение заводских штрихкодов (`PartBarcode`)

| Поле | Тип | Примечание |
|---|---|---|
| part | FK→PartType | `related_name="barcodes"` |
| value | char(100) **uniq** | заводской штрихкод уникален |
| note | char(255) blank | |

Одна деталь — **несколько** заводских штрихкодов; каждый штрихкод уникален.

## 8. Хранение совместимости (`PartCompatibility`)

| Поле | Тип | Примечание |
|---|---|---|
| part | FK→PartType | `related_name="compatibilities"` |
| vehicle_model | FK→catalog.VehicleModel | `on_delete=PROTECT` |
| year_from / year_to | int ✓null | опциональные годы применимости |
| note | char(255) blank | комментарий по совместимости |

`unique_together(part, vehicle_model, year_from, year_to)`. Одна деталь подходит
к нескольким моделям техники.

## 9. Экраны

- **Список деталей** — поиск (по названию и номеру) + фильтры (категория,
  производитель, режим учёта, активность).
- **Создание детали** — основные поля `PartType`.
- **Редактирование детали** — основные поля.
- **Карточка детали** — просмотр (см. п.10).
- **Активация/деактивация** (toggle).
- **Номера / штрихкоды / совместимость** — управляются на карточке детали
  отдельными мини-формами «добавить»/«удалить» (без сложных вложенных
  конструкторов и формсетов на одной странице).

## 10. Что показывает карточка детали

- название; категория; производитель; единица; режим учёта;
- номера (OEM/артикул/аналог/внутренний);
- заводские штрихкоды;
- совместимость (техника/модель/годы);
- рекомендованная цена; минимальная цена; минимальный остаток;
- блок **«Содержимое/остатки»** — заглушка: «Остатки появятся в следующих слоях».

## 11. Права доступа

| Возможность | Админ | Руковод. | Кладовщик | Продавец/Мастер | Наблюдатель |
|---|:--:|:--:|:--:|:--:|:--:|
| `manage_parts_catalog` (правка каталога) | ✓ | ✓ | — | — | — |
| просмотр каталога | ✓ | ✓ | ✓ | ✓ | ✓ |

- Создание/редактирование/деактивация и правка номеров/штрихкодов/совместимости —
  только `manage_parts_catalog` (Администратор, Руководитель). Иначе 403.
- Просмотр — любому авторизованному.

## 12. Capability — отдельная или существующая?

Добавляю **отдельную** `MANAGE_PARTS_CATALOG` (а не повторно `MANAGE_DIRECTORIES`).

*Почему:* каталог деталей — отдельная зона ответственности. По ТЗ (раздел 11) на
следующих слоях кладовщику понадобится **создавать карточку детали при приёмке**;
отдельное право позволит выдать это, не открывая ему редактирование остальных
справочников. Сейчас право у Админа и Руководителя; расширим точечно позже.

Добавляется в `roles.py` (+ `ALL_CAPABILITIES`, у Админа и Руководителя),
свойство `User.can_manage_parts`, миксин `ManagePartsMixin`.

## 13. Навигация

В боковое меню — пункт **«Детали»** (виден всем авторизованным), ведёт на список
деталей. Кнопки правки — только при `manage_parts_catalog`.

## 14. Обязательные тесты

1. Создание детали.
2. Создание детали с OEM-номером.
3. Создание детали с аналогом.
4. Создание детали с заводским штрихкодом.
5. Уникальность заводского штрихкода (дубль отклонён).
6. Связь детали с моделью техники.
7. `tracking_mode` сохраняется.
8. `recommended_price` и `min_price` сохраняются.
9. Себестоимость в `PartType` отсутствует (нет поля закупочной цены).
10. Деактивация вместо удаления.
11. Пользователь без права не может создавать/редактировать → 403.
12. Кладовщик может просматривать (GET → 200).
13. Наблюдатель может только просматривать (правка → 403).
14. Поиск по названию.
15. Поиск по OEM/артикулу.
16. Пункт «Детали» виден в навигации.

## 15. Ручная проверка

1. Войти администратором → создать деталь (категория, производитель, единица,
   режим учёта, рекомендованная/минимальная цена, минимальный остаток).
2. На карточке добавить OEM-номер, артикул, аналог, заводской штрихкод,
   совместимость с моделью техники.
3. Найти деталь по названию и по OEM.
4. Деактивировать деталь → исчезает из «только активные».
5. Войти кладовщиком → каталог виден для просмотра, кнопок правки нет; прямой
   POST на изменение → «Нет доступа».
6. Войти руководителем → может править каталог.

## 16. Критерии готовности

1. `PartType` с номерами, штрихкодами и совместимостью создаётся/редактируется.
2. Заводской штрихкод уникален; номера — нет (по ТЗ).
3. Цены (`recommended_price`, `min_price`) в карточке; себестоимости нет.
4. Режим учёта сохраняется; минимальный остаток есть.
5. Доступ: правка — Админ/Руководитель; просмотр — всем; иначе 403.
6. Поиск по названию и номеру работает; «Детали» в навигации.
7. Все обязательные тесты зелёные; `ruff`/`djlint` чисты; миграции с нуля;
   `makemigrations --check` чисто.
8. `PartItem`/партии/остатки/себестоимость/продажи/сканер **не реализованы**.

## Границы слоя (чего НЕ делаем)

- Не реализуем физические экземпляры `PartItem`.
- Не реализуем партии `Batch`/`BatchLine`.
- Не реализуем закупочную себестоимость.
- Не реализуем остатки `StockLot`/`StockMovement`/`StockBalance`.
- Не реализуем продажи, резервы, установку, списание, аналитику.
- Не реализуем scanner resolve и печать.
- Только каталог видов деталей, по которому потом приходуются партии.
