# План реализации — Слой 7. Landed cost и фиксация себестоимости

**Статус:** на согласовании (2026-06-24) · **Код не пишется до утверждения.**

## 1. Цель слоя

Рассчитать **полную себестоимость (landed cost)** строк партии: распределить
доставку/таможню/комиссии/прочее по строкам, перевести `accepted → cost_calculated`,
зафиксировать `cost_finalized=true` и заморозить себестоимость. **Физический
остаток на складе при этом не создаётся** — это слои 8–12.

## 2. Что добавляем к `Batch` и `BatchLine`

**`BatchLine` (новые поля):**
| Поле | Тип | Назначение |
|---|---|---|
| allocated_overhead_rub | decimal(14,2) | доля доп. расходов, отнесённая на строку |
| landed_unit_cost_rub | decimal(12,2) | себестоимость за единицу (товар + доля расходов) |
| landed_total_cost_rub | decimal(14,2) | себестоимость строки всего |

**`Batch` (новые поля):**
| Поле | Тип | Назначение |
|---|---|---|
| cost_allocation_method | enum | `by_value` (по стоимости, по умолч.) / `by_quantity` (по количеству) |
| cost_finalized_at | datetime ✓null | момент фиксации |
| cost_finalized_by | FK→User ✓null | кто зафиксировал |

> `landed_unit_cost_rub` соответствует одноимённому полю из `02-data-model.md`;
> `allocated_overhead_rub` — оттуда же. Базовая цена без накладных — это
> `unit_cost_rub`/`total_cost_rub` из Слоя 6.

## 3. Как считается landed cost

- **База строки в ₽** = `total_cost_rub` (количество × базовая закупочная цена,
  посчитано на Слое 6).
- **Накладные партии в ₽** = `total_extra_cost` (доставка + таможня + комиссии +
  прочее). *Доп. расходы партии вводятся в рублях* (доставка/таможня обычно
  оплачиваются локально); при необходимости валютные расходы добавим позже.
- **Распределение** накладных по строкам (метод — п.4).
- `landed_total_cost_rub = total_cost_rub + allocated_overhead_rub`.
- `landed_unit_cost_rub = round(landed_total_cost_rub / quantity, 2)`.

**Пример (как у заказчика):** товары 500 000 ₽ + доставка 80 000 + таможня
40 000 = накладные 120 000 ₽ распределяются по строкам; итог партии 620 000 ₽.

## 4. Метод распределения по умолчанию и крайние случаи

- По умолчанию **`by_value`**: вес строки = `total_cost_rub`;
  `allocated = total_extra_cost × (total_cost_rub строки / Σ total_cost_rub)`.
- `by_quantity`: вес строки = `quantity`.
- **Если `Σ total_cost_rub = 0`** (нулевая стоимость товаров, но есть расходы):
  по `by_value` делить не на что → **автоматически переходим на `by_quantity`**.
  Если и `Σ quantity = 0` (нет строк/нулевые количества) → **фиксацию
  запрещаем** с понятным сообщением.
  *Обоснование:* по количеству — естественный запасной вес, когда стоимость
  нулевая (например, бесплатные образцы с оплаченной доставкой); блокируем только
  когда распределять буквально не на что.
- Если `total_extra_cost = 0` — `allocated = 0` для всех, `landed = базовая`.
- Фиксация требует **хотя бы одну строку**, иначе блок.

## 5. Округление

- Только **`Decimal`**, до копеек, `ROUND_HALF_UP`.
- **Сумма `allocated_overhead_rub` по строкам = `total_extra_cost` точно.**
- Копеечная разница после округления **добирается на крупнейшую строку**
  (по `total_cost_rub`), чтобы итог партии сходился до копейки.
- `landed_total_cost_rub` хранится точно (база + доля) и сходится с
  `Σ total_cost_rub + total_extra_cost`. `landed_unit_cost_rub` — производная
  цена за штуку (округление на единицу допустимо).

## 6. Переход статусов и фиксация

Действие **«Рассчитать себестоимость»** доступно только для партии в статусе
`accepted`. В одной транзакции:

1. Проверки (п.4): есть строки; есть на что распределять.
2. Расчёт `allocated_overhead_rub`, `landed_total_cost_rub`,
   `landed_unit_cost_rub` по строкам.
3. `status = cost_calculated`; `cost_finalized = true`;
   `cost_finalized_at = now`; `cost_finalized_by = текущий пользователь`.

После фиксации строки партии **нельзя свободно редактировать** (п.8).

## 7. Что разрешено после `cost_finalized`

- Просмотр карточки и landed cost по строкам.
- Печать/экспорт (в будущих слоях).
- Создание складских остатков (на слоях 8–12).

## 8. Что запрещено после `cost_finalized`

- Менять `quantity`, `unit_cost_currency`, `unit_cost_rub`, `exchange_rate`.
- Менять расходы партии (`goods_total`, доставка/таможня/комиссии/прочее).
- Удалять и добавлять строки.
- Повторно рассчитывать себестоимость.
- Отменять расчёт без отдельной административной операции — **на этом слое отмену
  не реализуем**.

Реализация: `BatchLine` уже не редактируется при статусе ≥ `accepted` (Слой 6);
добавляем явный запрет редактирования партии (`BatchForm`) и строк при
`cost_finalized`. Действие расчёта отклоняется, если статус ≠ `accepted`.

## 9. Правило доступности

`batch.is_available_for_sale` становится `true` только при `cost_finalized=true`.
Продаж и остатков всё ещё нет — это **только свойство/индикатор**.

## 10. Экраны

- **Карточка партии**: блок расходов, базовая сумма товаров (`Σ total_cost_rub`),
  накладные, **итоговая себестоимость**; кнопка **«Рассчитать себестоимость»**
  только для `accepted`-партии (и `can_manage`).
- **Экран предпросмотра расчёта** (`cost_preview`): показывает распределение и
  landed cost по строкам **без сохранения** + кнопка «Зафиксировать». Защищает от
  случайной фиксации.
- После фиксации — landed cost по строкам, `cost_finalized_at`/`by`, статус
  `Себестоимость рассчитана`.

## 11. Права

- Рассчитывать и фиксировать себестоимость — **Администратор и Руководитель**
  (`MANAGE_BATCHES`).
- Роли без `can_view_purchase_cost` **не видят суммы** (включая landed cost).
- **Кладовщик не фиксирует** себестоимость (нет `MANAGE_BATCHES`) и не видит сумм.
- **Продавец/Мастер** не видит партии (доступ закрыт, как на Слое 6).

## 12. Обязательные тесты

1. Landed cost считается пропорционально стоимости строк (`by_value`).
2. `total_extra_cost` распределяется **полностью** (Σ `allocated` = расходы).
3. Округление сходится до копейки (добор на крупнейшую строку).
4. `cost_finalized` становится `true` после расчёта.
5. Статус переходит `accepted → cost_calculated`.
6. Нельзя рассчитать себестоимость для не-`accepted` партии.
7. Нельзя редактировать строки после `cost_finalized`.
8. Нельзя удалить строки после `cost_finalized`.
9. `batch.is_available_for_sale` `true` только после `cost_finalized`.
10. Роли без права не видят закупочные/landed суммы.
11. Кладовщик не может фиксировать себестоимость (403).
12. Повторный расчёт запрещён (вторая фиксация отклоняется).
13. (доп.) `Σ total_cost_rub = 0` → распределение по количеству; нет строк → блок.

## 13. Ручная проверка

1. Войти администратором → партия со строками → перевести в `accepted`.
2. Открыть предпросмотр расчёта (видно распределение накладных и landed по
   строкам) → зафиксировать.
3. Проверить: статус `cost_calculated`, `cost_finalized=true`, итог партии =
   товары + накладные до копейки.
4. Попробовать изменить строку/расходы/удалить строку → запрещено.
5. Повторно нажать расчёт → отклонено.
6. Войти кладовщиком → сумм не видно, кнопки расчёта нет.
7. Убедиться, что складской остаток не появился.

## 14. Критерии готовности

1. Landed cost считается и фиксируется; Σ накладных сходится с `total_extra_cost`.
2. Округление до копейки; итог партии = товары + накладные.
3. `accepted → cost_calculated`, `cost_finalized=true`, `cost_finalized_at/by`.
4. После фиксации строки/расходы неизменяемы; повторный расчёт запрещён.
5. `is_available_for_sale` = `cost_finalized`.
6. Доступ: расчёт — `MANAGE_BATCHES`; суммы — `can_view_purchase_cost`.
7. Все обязательные тесты зелёные; `ruff`/`djlint` чисты; миграции с нуля;
   `makemigrations --check` чисто.
8. `StockLot`/`PartItem`/`StockMovement`/`StockBalance`/размещение/сканер/продажи
   **не реализованы**.
