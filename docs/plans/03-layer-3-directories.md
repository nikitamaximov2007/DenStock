# План реализации — Слой 3. Справочники

**Статус:** на согласовании (2026-06-24) · **Код не пишется до утверждения.**

## 1. Цель слоя

Простые и надёжные справочники, на которые опираются будущие слои (карточка
детали, партии, совместимость с техникой): категории-дерево, производители,
техника (вид → марка → модель), поставщики, единицы измерения. С экранами
просмотра/редактирования, ролевым доступом и навигацией. **Без склада, деталей,
партий, остатков, продаж и аналитики.**

## 2. Появляющиеся справочники

- **Категории деталей** — древовидные.
- **Производители** (деталей).
- **Виды техники** (авто, снегоход, квадроцикл, катер, яхта, …).
- **Марки техники**.
- **Модели техники** (с годами выпуска).
- **Поставщики**.
- **Единицы измерения** — включаю уже сейчас: дешёво и нужно карточке детали на
  Слое 5.

## 3. Django-приложения

- **`catalog`** — категории, производители, единицы, техника (вид/марка/модель).
- **`suppliers`** — поставщики.

Соответствует модульной карте из `01-architecture.md`. Создаём оба приложения на
этом слое (по вертикальному принципу — раньше их пустыми не заводили).

## 4. Модели, поля, ограничения

Общее для всех справочников: `is_active` (деактивация вместо удаления),
`created_at`/`updated_at`.

### catalog.Category — категория (дерево)
| Поле | Тип | Ограничения |
|---|---|---|
| name | char(150) | обязательное |
| parent | FK→Category ✓null | `on_delete=PROTECT`; `idx` |
| sort_order | int | по умолч. 0 |
| is_active | bool | по умолч. True |

- `unique_together(parent, name)` — нет дублей в одном узле.
- Запрет циклов — в `clean()` (см. п.5).

### catalog.Manufacturer — производитель детали
`name` char(150) **uniq**, `country` char(100) blank, `is_active`.

### catalog.Unit — единица измерения
`name` char(50) uniq (Штука), `short_name` char(20) (шт), `is_active`.
Стартовые значения (шт, компл, м, кг, л, упак) — data-миграцией.

### catalog.VehicleType — вид техники
`name` char(100) **uniq**, `sort_order` int, `is_active`.
Стартовые значения (Автомобиль, Снегоход, Квадроцикл, Катер, Яхта) —
data-миграцией; администратор может добавлять свои.

### catalog.VehicleMake — марка техники
| Поле | Тип | Ограничения |
|---|---|---|
| vehicle_type | FK→VehicleType | `on_delete=PROTECT`; `idx` |
| name | char(120) | |
| is_active | bool | |

`unique_together(vehicle_type, name)`.

### catalog.VehicleModel — модель техники
| Поле | Тип | Ограничения |
|---|---|---|
| vehicle_make | FK→VehicleMake | `on_delete=PROTECT`; `idx` |
| name | char(150) | |
| year_from / year_to | int ✓null | годы выпуска |
| is_active | bool | |

`unique_together(vehicle_make, name, year_from, year_to)`.

### suppliers.Supplier — поставщик
| Поле | Тип |
|---|---|
| name | char(200) **uniq** |
| country | char(100) blank |
| contact_person | char(150) blank |
| phone | char(50) blank |
| email | EmailField blank |
| website | URLField blank |
| default_currency | char(3), по умолч. `RUB` |
| comment | text blank |
| is_active | bool |

> **Соответствие `02-data-model.md`:** `VehicleType` ↔ бывш. `EquipmentType`,
> `VehicleMake` ↔ бывш. `Brand`. Принята строгая иерархия Тип → Марка → Модель
> (проще). Ссылки на эти сущности в модели данных приведу при сборке карточки
> детали (Слой 5).

## 5. Дерево категорий без лишнего усложнения

- **Только `parent`-ссылка** (без MPTT и сторонних зависимостей).
- **Запрет циклов** в `Category.clean()`: нельзя сделать родителем сам себя или
  любого своего потомка (проходим вверх по `parent` от предполагаемого родителя).
- **Сортировка** — `sort_order` (+ имя).
- **Активность** — `is_active`; деактивация вместо удаления.
- Отображение — рекурсивный отступ в списке (без перетаскиваний и конструкторов).

## 6. Связь техники

`VehicleType` → `VehicleMake` (FK на тип) → `VehicleModel` (FK на марку).
Вид техники получается у модели через марку. Формы используют зависимые
выпадающие списки на уровне выбора (марка фильтруется по типу — простой выбор,
без динамических JS-конструкторов; при необходимости перезагрузка списка).

## 7. Поля поставщика

Реализуются все перечисленные: название, страна, контактное лицо, телефон, email,
сайт, валюта по умолчанию (`RUB`), комментарий, активность. (См. модель в п.4.)

## 8. Экраны

Для каждого справочника — единый простой набор:
- **Список** с поиском по названию и фильтром «только активные»/«все»;
- **Создание**;
- **Редактирование**;
- **Активация/деактивация** (POST-переключатель, без удаления).

Плюс **страница-индекс «Справочники»** со ссылками на все списки.
Категории в списке показываются деревом (отступы). Реализация — типовые
class-based views (`ListView`/`CreateView`/`UpdateView` + toggle), общие шаблоны
списка/формы параметризуются. Без вложенных UI-конструкторов.

## 9. Права доступа

Добавляю в `apps/accounts/roles.py` **одну** возможность
`MANAGE_DIRECTORIES` (manage_directories):

| Возможность | Админ | Руковод. | Кладовщик | Продавец/Мастер | Наблюдатель |
|---|:--:|:--:|:--:|:--:|:--:|
| `manage_directories` (редактирование справочников) | ✓ | ✓ | — | — | — |
| просмотр справочников | ✓ | ✓ | ✓ | ✓ | ✓ |

- **Редактирование** (создание/изменение/переключение активности) — только
  `manage_directories` (Администратор, Руководитель). Иначе 403.
- **Просмотр** — любому авторизованному (кладовщик, продавец/мастер, наблюдатель).

Это единственное расширение модели прав; конструктора прав не вводим.

## 10. Навигация

В боковое меню добавляется пункт **«Справочники»** — виден всем авторизованным
(просмотр разрешён всем). Ведёт на индекс справочников. Кнопки
«Добавить»/«Изменить»/«Деактивировать» внутри экранов показываются только при
`manage_directories`; у остальных экраны открыты в режиме просмотра.

## 11. Обязательные тесты

1. Создание категории.
2. Запрет цикла в категориях (родитель = потомок → `ValidationError`).
3. Деактивация вместо удаления (после toggle объект существует, `is_active=False`).
4. Создание производителя.
5. Создание вида/марки/модели техники (цепочка связей).
6. Создание поставщика.
7. Не-админ/не-руководитель (кладовщик) **не может редактировать** (POST → 403).
8. Наблюдатель **не может редактировать** (POST → 403).
9. Справочники **доступны для просмотра** разрешённым ролям (кладовщик GET → 200).
10. Навигация показывает раздел «Справочники».

## 12. Ручная проверка

1. Войти администратором → создать дерево категорий (родитель/потомки),
   производителей, единицы; завести Тип → Марку → Модель; добавить поставщика.
2. Попробовать сделать категорию потомком самой себя → понятная ошибка.
3. Деактивировать запись → исчезает из «только активные», но остаётся в «все».
4. Войти кладовщиком → «Справочники» видны для просмотра, кнопок изменения нет;
   прямой POST на изменение → «Нет доступа».
5. Войти руководителем → может редактировать справочники.

## 13. Критерии готовности

1. Все справочники создаются/редактируются; деактивация вместо удаления работает.
2. Дерево категорий с запретом циклов и сортировкой.
3. Цепочка Тип → Марка → Модель связана и заполняема.
4. Поставщик со всеми полями.
5. Доступ: правка — Админ/Руководитель; просмотр — всем авторизованным; иначе 403.
6. «Справочники» в навигации; индекс со ссылками.
7. Все обязательные тесты зелёные; `ruff` и `djlint` чисты; миграции применяются с нуля.
8. Склад/детали/партии/остатки/продажи/аналитика **не реализованы**.

## Границы слоя (чего НЕ делаем)

- Не реализуем `Part`, OEM-номера, аналоги, штрихкоды.
- Не реализуем партии, остатки, движения, продажи, аналитику.
- Не реализуем физический склад `StorageLocation` — это Слой 4.
- Без сложных вложенных UI-конструкторов; справочники простые и надёжные.
