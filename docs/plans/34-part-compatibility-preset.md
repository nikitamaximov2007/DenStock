# План v1.1.6 — Preset совместимости деталей и техники (PartCompatibility)

**Статус:** ПЛАН (реализацию не начинаем без «go»). Продолжение
[плана 33](33-research-based-directory-preset.md) и слоя `import_preset` (commit `01af066`), где
`PartCompatibility` была **сознательно отложена**. В этом шаге — только план; код/JSON/команду/
модели/миграции/поиск не трогаем; остатки не создаём; `research_inputs/` не читаем и не коммитим.

**Главная мысль:** совместимость **опаснее** обычных справочников. Ложная связь «деталь подходит
к модели» может привести к неправильной продаже/ремонту. Поэтому добавляем совместимость **только**
если она **прямо подтверждена** research или **очевидно** следует из уже созданных сущностей —
без риска. Лучше не указать совместимость, чем указать ложную.

---

## 1. Почему PartCompatibility вынесена в отдельный слой

- Справочники брендов/моделей/категорий **безопасны**: они лишь описывают номенклатуру.
- Совместимость **влияет на действия**: по ней подбирают деталь под продажу/ремонт. Ошибка здесь
  = неправильная деталь у клиента/на аппарате.
- **Ложная совместимость опаснее отсутствующей**: пустая совместимость заставляет уточнить у
  мастера; ложная — создаёт ложную уверенность. Поэтому — отдельный, осторожный слой с явными
  уровнями доверия и вопросами к Денису.

---

## 2. Текущие модели и поля (факт из кода)

`apps/catalog/models.py`:

- **`PartCompatibility`**: `part` → `PartType` (CASCADE, related_name `compatibilities`);
  `vehicle_model` → `VehicleModel` (PROTECT); `year_from`/`year_to` (int, null); `note` (≤255).
  **UniqueConstraint** `(part, vehicle_model, year_from, year_to)` = `uniq_part_model_years`.
- **`PartType`**: `name`, `category`(FK), `manufacturer`(FK, null), `unit`(FK), `tracking_mode`.
  **Нет** DB-уникальности по name → сопоставление в импорте по `(name, category)`.
- **`PartNumber`**: `part`(FK), `value`, `normalized_value`(авто), `kind`(oem/article/analog/
  internal_ref), `is_primary`. **Нет** DB-уникальности.
- **`VehicleModel`**: `vehicle_make`(FK), `name`, `year_from`/`year_to`. Unique
  `(vehicle_make, name, year_from, year_to)`.
- Существуют форма/админка/тесты (`tests/test_parts.py`, `tests/test_search.py`) — модель уже в
  проде, менять её не нужно.

**Важно для дизайна:** у `PartCompatibility` **нет** полей `confidence`/`status`/`evidence`/
`source`. Значит эти атрибуты будущего `compatibility.json` — **метаданные data-файла**, а не
поля БД. Импорт использует `status` только чтобы **решить, создавать связь или нет**;
краткое `evidence` можно положить в `note`. **Новых полей/миграций не вводим.**

## 3. Какие связи можно считать безопасными (уровни)

**Уровень A — можно добавить в preset (status=active).** Только если связь **прямо подтверждена**
docs/research или **очевидна** по названию уже созданной сущности (конкретный двигатель ↔
конкретная модель), без обобщений.

**Уровень B — только candidate/deferred (status=deferred, НЕ импортировать).** Связь вероятна, но
не подтверждена точно: общая экосистема BRP/Rotax; узел без номера/спецификации; двигатель без
явно названной модели; Yamaha/Mercury без точной модели.

**Уровень C — не добавлять вообще.**
- Общие узлы/расходники (**подшипник, сальник, масло, клапан, прокладка, поршневая**) без точного
  **номера/размера/применимости** — они «универсальны» лишь на словах.
- Любые связи «по похожести бренда» (BRP → всё BRP).
- Любые неподтверждённые совместимости и любые fake part numbers.

## 4. Будущая структура `compatibility.json` (план, файл сейчас НЕ создаём)

```jsonc
[
  {
    "part_type": "Двигатель Rotax 1203",     // сопоставление с PartType по имени
    "part_number": null,                      // опц.; НЕ выдумывать OEM
    "vehicle_make": "Ski-Doo",                // для резолва VehicleModel
    "vehicle_model": "Renegade 1200",
    "evidence": "Telegram post 2296/2295: «Renegade 1200 с движком Rotax 1203»",
    "confidence": "high",                     // метаданные файла (в БД не пишется)
    "source_doc_section": "docs/research/01 §2/§4",
    "status": "active",                       // active → импортируем; deferred → только summary
    "note": "Двигатель модели (подтверждено research)"  // короткое → в PartCompatibility.note
  }
]
```

- `status` — единственное поле, управляющее импортом (active/deferred).
- `evidence`/`confidence`/`source_doc_section` — **не** поля БД; служат ревью и трассируемости;
  при импорте кратко попадают в `note` (≤255) или игнорируются.
- `part_number` — только если реально подтверждён; иначе `null` (fake OEM запрещены).

## 5. Как `import_preset` должен работать с compatibility (только план)

- Читать `data/presets/<preset>/compatibility.json`, если файл есть; нет файла → секция пропущена.
- **Резолв по существующим сущностям:** `PartType` по `(name, category?)`; `VehicleModel` по
  `(vehicle_make → VehicleMake.name, name)`. **Не создавать** PartType/VehicleModel «на лету».
- **Создавать только `status == "active"`.** Записи `deferred` — **не импортировать**, показывать
  в summary отдельной строкой.
- Если `PartType`/`VehicleModel` из active-записи **отсутствуют** → не создавать молча, а пометить
  `missing/deferred` с понятным сообщением (какой сущности не хватает).
- **Идемпотентность:** `get_or_create` по natural key `(part, vehicle_model, year_from, year_to)`
  → повторный запуск без дублей.
- **`--dry-run`:** показывает отдельно «active (будет создано)», «deferred (пропущено)»,
  «missing»; ничего не пишет (как в текущей команде — atomic + rollback).
- **Не создаёт остатки**, не трогает stock/inventory-services, не меняет поиск, не добавляет
  миграции/поля.

> Реализация — расширение существующей команды одной секцией (по образцу текущих секций), без
> изменения её архитектуры.

## 6. Тесты для будущей реализации

- `--dry-run` не пишет в БД (кол-во `PartCompatibility` не меняется).
- active-связь **создаётся** (напр. Rotax 1203 ↔ Renegade 1200).
- deferred-связь **не создаётся** (в summary отмечена как отложенная).
- повторный импорт **без дублей** (idempotent по natural key).
- если `PartType`/`VehicleModel` отсутствуют — команда **не создаёт их**, а пишет `missing/deferred`.
- **не создаются** `Batch`/`StockLot`/`PartItem`/`StockMovement`/`StockBalance`.
- `makemigrations --check` — **без миграций**; `manage.py check`/`ruff`/`pytest` зелёные.

## 8. Вопросы к Денису (перед реальной совместимостью)

1. Какие двигатели к каким моделям **точно** относятся (900 ACE, 1004, 1203, 849, 582, 552)?
2. Какие детали подходят **к нескольким** моделям (кросс-применимость)?
3. Какие расходники **действительно универсальны**, а какие — под конкретный двигатель?
4. Какие **номера деталей** (OEM/каталожные) использовать как источник совместимости?
5. Какие совместимости **опасно** указывать без проверки (частые ошибки)?
6. Как вы сами отвечаете клиенту «подойдёт или нет» — по номеру, по модели, по двигателю?

## 9. Что НЕ делать

- Не добавлять массовую совместимость «BRP ко всему BRP» / по общему бренду.
- Не связывать расходники/общие узлы без номера/спецификации.
- Не выдумывать part numbers; не импортировать `deferred`-связи.
- Не создавать PartType/VehicleModel «на лету» в импорте совместимости.
- Не менять поиск, модели, миграции; не создавать остатки; не трогать stock/inventory-services.

## 10. Критерии приёмки будущего слоя

1. Импортируются **только подтверждённые active** связи; `deferred` — не импортируются (только summary).
2. `--dry-run` безопасен (ничего не пишет; active/deferred/missing показаны отдельно).
3. Повторный импорт **без дублей** (natural key).
4. **Нет миграций**; модель `PartCompatibility` не менялась.
5. **Нет** изменений stock/inventory-services и остатков.
6. Отсутствующие PartType/VehicleModel → `missing/deferred`, а не тихое создание.
7. **Каждую созданную связь можно объяснить Денису** (evidence в data-файле/`note`).
8. `pytest`/`ruff`/`djlint --check`/`makemigrations --check`/`manage.py check` — зелёные.
