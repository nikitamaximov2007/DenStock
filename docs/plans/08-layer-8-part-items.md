# План реализации — Слой 8. Поштучный учёт (PartItem)

**Статус:** на согласовании (2026-06-24) · **Код не пишется до утверждения.**

## 1. Цель слоя

Научиться создавать и хранить **физические поштучные экземпляры** деталей
(`PartItem`) из уже **финансово закрытой партии** (`cost_finalized=true`):
внутренний номер и штрихкод, серийный номер, замороженная landed-себестоимость,
статус и (опционально) место хранения.

Экземпляры создаются **вручную из карточки партии/строки, без сканера**. Это ещё
**не складское движение**: `StockMovement`/`StockBalance` на этом слое **не
создаются** (Слой 10), количественный учёт `StockLot` — Слой 9, размещение
сканером — Слой 12. PartItem просто появляется как физический объект с
себестоимостью.

## 2. Где размещаем модели — новый app `inventory`

**Решение: создать новое приложение `apps/inventory`.**

*Обоснование.* PartItem — первая сущность «физического склада». За ней по
роадмапу идут `StockLot` (9), `StockMovement`/`StockBalance` (10),
`Reservation` (15), которые образуют единый домен «остатки и движения».
Этот домен отличается от:
- `procurement` — закупки/финансы партии (Batch/BatchLine, landed cost);
- `catalog` — справочные карточки видов деталей (PartType), без физики;
- `warehouse` — статичная структура мест хранения.

**Альтернативы и почему нет:**
- *PartItem в `procurement`* (раз создаётся из BatchLine) — отвергнуто: дальше
  инвентарь обрастёт движениями/балансами/резервами и раздует закупочный модуль;
  направление зависимости должно быть `inventory → procurement`, не наоборот.
- *PartItem в `catalog`* — отвергнуто: каталог остаётся справочным, без остатков.

Итог: `inventory` — естественный дом для PartItem и всех будущих слоёв склада.

## 3. Модель `PartItem`

| Поле | Тип | Назначение / правила |
|---|---|---|
| internal_number | CharField(20), unique, editable=False | `DS-000001`, генерируется (п.6) |
| internal_barcode | CharField(40), unique, editable=False | `ITEM:DS-000001`, производное (п.7) |
| part_type | FK→catalog.PartType (PROTECT) | только `tracking_mode=serial` (п.5) |
| batch | FK→procurement.Batch (PROTECT) | = `batch_line.batch`, проставляется сервисом |
| batch_line | FK→procurement.BatchLine (PROTECT) | источник экземпляра и себестоимости |
| serial_number | CharField(100), blank | опционален; уникальность — п.5 |
| landed_cost_rub | Decimal(12,2), editable=False | копия `batch_line.landed_unit_cost_rub` |
| status | CharField(20), choices | статусы — п.4, default `receiving` |
| current_location | FK→warehouse.StorageLocation (PROTECT), null/blank | если задано — `storage_allowed=true` (п.5) |
| note | CharField(255), blank | примечание |
| created_at | DateTimeField(auto_now_add) | |
| updated_at | DateTimeField(auto_now) | |

> **О режиме учёта.** В коде `PartType.TrackingMode` = `serial` («Поштучный») и
> `bulk` («Количественный»); отдельного `item` нет. Поскольку `serial_number`
> опционален, режим `serial` уже покрывает и «поштучно с серийником», и
> «поштучно без серийника». Поэтому **PartItem ↔ `tracking_mode=serial`**;
> `bulk` уходит в `StockLot` (Слой 9). Новый режим `item` **не вводим** (стабильность
> каталога), если вы явно не захотите различать «с серийником / без».

## 4. Статусы `PartItem`

Полный перечень (закладываем все, переходами этого слоя управляем частью):

`receiving` · `available` · `reserved` · `sold` · `installed` · `repair` ·
`written_off` · `returned` · `quarantine`.

- **Создаётся в статусе `receiving`** (получен из закрытой партии, но ещё не
  подтверждён как складской остаток — движений/баланса нет).
- **Переходы, разрешённые на этом слое (вручную, `MANAGE_INVENTORY`):**
  `receiving → available`, `receiving → quarantine`, `available ↔ quarantine`.
- **`reserved` / `sold` / `installed` / `repair` / `written_off` / `returned`**
  на этом слое **не выставляются вручную** — ими управляют свои слои
  (15/16/17/18/19) через движения. На Слое 8 такие переходы запрещены.

## 5. Правила (инварианты)

1. **Только поштучные.** PartItem создаётся лишь для `PartType.tracking_mode =
   serial`. Для `bulk` — отказ (`StockLot`, Слой 9).
2. **Только из закрытой партии.** Создание запрещено, если
   `batch_line.batch.cost_finalized=false`. (Гарантирует, что landed уже заморожен.)
3. **Себестоимость заморожена.** `landed_cost_rub = batch_line.landed_unit_cost_rub`
   на момент создания; далее не пересчитывается.
4. **`internal_number` уникален** (п.6); **`internal_barcode` уникален** (п.7).
5. **`serial_number`** — рекомендуем **уникальность в пределах `part_type`,
   игнорируя пустые** (partial `UniqueConstraint(fields=[part_type, serial_number]`,
   `condition=~Q(serial_number="")`).
   *Обоснование:* серийник выдаёт производитель и он осмыслен в рамках конкретного
   вида детали; одинаковая строка у разных видов деталей — допустимое совпадение,
   глобальная уникальность дала бы ложные конфликты. Пустой серийник разрешён
   (не у всех поштучных деталей он читаем).
   *Альтернатива (если хотите строже):* глобальная уникальность непустых серийников.
6. **`current_location`** — необязателен (на приёмке места ещё может не быть);
   если задан, обязан быть `storage_allowed=true` и `is_active=true`
   (через `StorageLocation.can_hold_stock()`), иначе ошибка валидации.
7. **Лимит по строке.** Число PartItem по `batch_line` не превышает
   `batch_line.quantity` (п.8). Для поштучной строки `quantity` должно быть целым.
8. **Без движений.** Создание PartItem **не пишет** `StockMovement`/`StockBalance`
   (их ещё нет). «Наличие» на этом слое — это факт существования PartItem и его
   статус, а не баланс.

## 6. Генерация `internal_number`

**Формат:** `DS-000001` (префикс `DS-`, 6 цифр, рост по 1).

**Рекомендация: ввести модель-счётчик `NumberSequence`** (в `inventory`) и брать
номер под блокировкой единственной строки счётчика:

```
key=part_item, prefix="DS-", last_value (PositiveInteger)
# в transaction.atomic:
row = NumberSequence.objects.select_for_update().get(key="part_item")
row.last_value += 1; row.save(); return f"{row.prefix}{row.last_value:06d}"
```

*Почему так, а не как у Batch.* Текущий `generate_batch_number` блокирует
**последнюю существующую строку** (`select_for_update().first()`), но на **пустой
таблице** блокировать нечего — два параллельных создания могут выдать `DS-000001`
дважды. Отдельная строка-счётчик всегда существует и сериализует выдачу даже при
нуле экземпляров. Строка счётчика создаётся **data-миграцией**.

> Это и есть «усиление атомарности» из ваших указаний — без «голого `max()+1`».
> Позже `NumberSequence` сможет обслуживать и нумерацию партий (консолидация);
> на этом слое трогаем только PartItem.

## 7. Генерация `internal_barcode`

`internal_barcode = f"ITEM:{internal_number}"` → `ITEM:DS-000001`. Уникален,
`editable=False`, проставляется в `save()`/сервисе. Префикс согласован с
`LOC:` у мест хранения и пригодится единому резолву сканера (Слой 11).

## 8. Связь с `BatchLine` и прогресс

- При создании: `count(PartItem по batch_line) + создаваемое ≤ batch_line.quantity`.
  Проверка — в сервисе под `transaction.atomic` с `select_for_update` на
  `batch_line` (защита от гонок при массовом создании).
- Для поштучной строки `quantity` обязана быть **целым** (дробные количества —
  это `bulk`/`StockLot`). При нецелом — отказ с понятным сообщением.
- **Прогресс** на карточке строки/партии: «Создано N из {quantity}». Когда
  `N = quantity` — кнопку создания скрываем/блокируем.

## 9. Экраны

- **Список экземпляров** (`item_list`): таблица (номер, деталь, статус, серийник,
  место, партия; себестоимость — только при праве). Фильтры: статус, деталь
  (part_type), партия, место (location).
- **Карточка экземпляра** (`item_detail`): все поля, штрихкод, источник
  (партия/строка), себестоимость (при праве).
- **Создание из строки** (`item_create` от `batch_line`): один экземпляр
  (серийник, место, примечание).
- **Массовое создание N** из строки (`item_bulk_create`): создать `N` экземпляров
  разом (в пределах остатка `quantity − создано`), серийники можно не вводить.
  **Без сканера.**
- **Редактирование** (`item_edit`): `serial_number`, `current_location`, `note`
  (и статус в рамках разрешённых переходов п.4). Номер/штрихкод/себестоимость/
  партия — только чтение.
- **На карточке партии/строки**: блок «Экземпляры» со списком созданных и
  прогрессом N/quantity; кнопка «Создать экземпляры» (при `MANAGE_INVENTORY` и
  `cost_finalized`).

## 10. Права

Вводим новую возможность **`MANAGE_INVENTORY`** (в `accounts/roles.py`).

| Роль | Доступ к экземплярам |
|---|---|
| Администратор | управление (создание/редактирование/статус) + себестоимость |
| Руководитель | управление + себестоимость |
| Кладовщик | **создание и правка** (серийник/место/статус) — приёмка; **без сумм** |
| Продавец/Мастер | на этом слое раздел **не виден** (наличие для продаж — слои 13/16) |
| Наблюдатель | только просмотр (включая суммы — есть `can_view_purchase_cost`) |

- `MANAGE_INVENTORY` выдаём ролям **Администратор, Руководитель, Кладовщик**.
- **Закупочная/landed себестоимость скрыта** без `can_view_purchase_cost`
  (у кладовщика её нет — создаёт экземпляры, но сумм не видит; копирование
  landed в `landed_cost_rub` идёт на сервере).
- Создание/правка под `MANAGE_INVENTORY`; иначе 403.

## 11. Обязательные тесты (`tests/test_part_items.py`)

1. PartItem создаётся для поштучного (`serial`) PartType.
2. Нельзя создать PartItem для количественного (`bulk`) PartType — отказ.
3. Нельзя создать PartItem из партии без `cost_finalized` — отказ.
4. `landed_cost_rub` копируется из `batch_line.landed_unit_cost_rub`.
5. `internal_number` генерируется и уникален (`DS-000001`, без дублей).
6. `internal_barcode` генерируется и уникален (`ITEM:DS-000001`).
7. Нельзя создать больше экземпляров, чем `quantity` в `batch_line`.
8. `current_location` должен быть `storage_allowed=true` (иначе ошибка).
9. `serial_number` сохраняется; дубликат непустого серийника в пределах
   `part_type` отклоняется; пустые серийники не конфликтуют.
10. Себестоимость скрыта без `can_view_purchase_cost` (кладовщик не видит сумм).
11. Права ролей работают (кладовщик создаёт → 200/302; продавец → 403/нет раздела;
    наблюдатель — только чтение).
12. Раздел «Экземпляры» в навигации появляется только нужным ролям.
13. (доп.) Прогресс «создано N из quantity» корректен; при `N=quantity` создание
    блокируется.

## 12. Ручная проверка

1. Войти администратором → партия со строками → **зафиксировать себестоимость**
   (Слой 7).
2. На строке поштучной детали нажать «Создать экземпляры» → создать 1 и затем
   N штук; увидеть номера `DS-…`, штрихкоды `ITEM:DS-…`, прогресс N/quantity.
3. Указать серийник и место (только `storage_allowed`) → сохранить; проверить
   карточку экземпляра.
4. Попробовать создать больше, чем `quantity` → отказ. Попробовать выбрать место
   с `storage_allowed=false` → отказ.
5. Войти кладовщиком → может создавать/править, **сумм не видит**.
6. Войти продавцом → раздела экземпляров нет.
7. Убедиться, что **складского движения/баланса не появилось** (их ещё нет).

## 13. Критерии готовности

1. Создаются физические PartItem из закрытой (`cost_finalized`) партии; для
   `bulk` и незакрытой партии — отказ.
2. `internal_number`/`internal_barcode` уникальны и атомарны (через
   `NumberSequence`, без `max()+1`).
3. `landed_cost_rub` заморожен из `batch_line.landed_unit_cost_rub`.
4. Лимит по `quantity` соблюдается; прогресс N/quantity отображается.
5. `current_location` — только `storage_allowed=true`.
6. Права: управление — `MANAGE_INVENTORY` (Админ/Руководитель/Кладовщик); суммы —
   `can_view_purchase_cost`; продавец раздела не видит.
7. `StockLot`/`StockMovement`/`StockBalance`/сканер/размещение/продажи/резерв/
   аналитика/PDF-этикетки **не реализованы**.
8. Все тесты зелёные; `ruff`/`djlint` чисты; миграции с нуля;
   `makemigrations --check` чисто.

## Границы слоя (чего НЕ делаем)

- Не реализуем `StockLot` и количественный учёт.
- Не реализуем `StockMovement` и `StockBalance`.
- Не реализуем единый резолв сканера и размещение по ячейкам.
- Не реализуем продажу, резерв, установку, возврат, списание, инвентаризацию.
- Не реализуем аналитику и PDF-этикетки.
- PartItem появляется как **физический объект с себестоимостью**, но **движения
  остатков ещё нет**.
