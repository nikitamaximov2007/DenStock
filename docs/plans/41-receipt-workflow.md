# План 41 (Layer 28) - «Поступление»: документ прихода с созданием детали на лету

**Статус:** реализовано в Layer 28. Активирована последняя заглушка меню
«Поступление»; серых пунктов в навигации больше нет.

---

## 1. Чем «Поступление» отличается от «Партии» и «Приёмки сканером»

| | Партии (`/batches/`) | Приёмка сканером (`/scanner/receiving/`) | Поступление (`/receipts/`) |
|---|---|---|---|
| Что это | техническая основа прихода: строки, валюта, доп. расходы, landed cost | постановка конкретной физической детали в ячейку сканером | человеческий документ поставки: несколько позиций, себестоимость и ячейки в одном экране |
| Кто пользуется | закупки/руководитель | кладовщик у стеллажа | кладовщик/руководитель при разборе коробки от поставщика |
| Результат | партия и себестоимость | экземпляр/лот становится доступным | партия + себестоимость + остатки одним нажатием |

«Поступление» НЕ вторая система склада: при проведении оно просто вызывает
существующую цепочку (партия -> фиксация себестоимости -> создание и приёмка
экземпляров/лотов). Партии и приёмка сканером продолжают работать как раньше.

## 2. Модели и миграции

Добавлено новое приложение `apps.receipts` с двумя минимальными моделями:

- **Receipt**: number (ПОС-, счётчик NumberSequence), supplier (nullable в черновике),
  received_at, status (draft/posted), comment, batch (ссылка на созданную партию),
  created_by/created_at, posted_by/posted_at.
- **ReceiptLine**: receipt, part_type, quantity (>0, CheckConstraint), unit_cost_rub
  (>=0, CheckConstraint), location, comment, batch_line (ссылка после проведения).

Чистая обёртка над `Batch` без новых моделей невозможна: у `BatchLine` нет ячейки
на строку, а черновик партии не умеет «провести всё атомарно до остатков». Никакой
бухгалтерии (оплата/долги/НДС/счета) не добавлено.

**Миграции (2, обе необходимы):** `receipts.0001_initial` (две таблицы) и
`receipts.0002_seed_receipt_sequence` (счётчик номеров, тот же паттерн, что у
списаний). Чужие модели не менялись.

## 3. Черновик

Черновик - просто строки документа. Можно создавать, править шапку
(поставщик/дата/комментарий), добавлять/менять/удалять позиции, сохранять и
открывать позже. Черновик НЕ создаёт партий, лотов, экземпляров, движений и
остатков - зафиксировано тестами (`test_draft_does_not_touch_stock`,
`test_create_and_add_line_via_views`: снимок склада до/после идентичен).

## 4. Проведение

Кнопка «Провести поступление» -> `services.post_receipt` в `transaction.atomic`
под `select_for_update`:

1. валидация: есть позиции; выбран поставщик (обязателен, т.к. `Batch.supplier`
   NOT NULL); каждая строка корректна;
2. создаётся `Batch` (RUB, курс 1, без доп. расходов) + `BatchLine` по позициям;
3. `procurement.finalize_cost` фиксирует себестоимость (landed cost за единицу
   равен введённой цене, т.к. накладных нет);
4. поштучные детали: `create_part_items` (нужное число экземпляров) +
   `receive_part_item` в ячейку строки; количественные: `create_stock_lot` +
   `receive_stock_lot`. Движения и остатки создают эти сервисы, не receipts;
5. документ получает статус «Проведено», posted_by/posted_at и ссылку на партию.

Гарантии: повторное проведение запрещено (тест: снимок склада не меняется);
ошибка любой строки откатывает всё (тест с дробным количеством поштучной детали:
ни партии, ни движений); проведённый документ read-only (сервисы бросают
ReceiptError, экраны редактирования уводят на просмотр).

## 5. «+ Новая деталь» из поступления (вариант A)

Рядом с формой позиции - ссылка «+ Новая деталь» на существующую страницу
создания детали с параметром `?next=<черновик>`. `PartTypeCreateView` после
сохранения возвращает на `next` (только безопасные relative-URL,
`url_has_allowed_host_and_scheme`; внешние адреса игнорируются - тест) и передаёт
`?new_part=<pk>`: черновик подставляет созданную деталь в форму позиции.
Создание детали НЕ создаёт остатков (тест); отмена (уход со страницы) ничего не
ломает - черновик живёт отдельно.

Примечание: создание деталей требует `can_manage_parts` (Админ/Руководитель) -
это существующее правило каталога, не менялось. У кладовщика кнопка не
показывается; он добавляет только существующие детали.

## 6. Типы деталей

Оба существующих режима: поштучный (`serial`) - создаётся нужное число
экземпляров, количество обязано быть целым (валидация); количественный (`bulk`) -
создаётся лот на количество. Новых режимов учёта не выдумано.

## 7. Проверки (валидация)

Нельзя провести: без позиций; без поставщика; строку без детали/ячейки;
количество <= 0; себестоимость < 0; в ячейку без хранения
(`can_hold_stock`); дробное количество поштучной детали. Формы дублируют
дружелюбные проверки, сервисы - обязательные.

## 8. Права

- Создание/правка/проведение/просмотр: `can_manage_inventory`
  (Администратор, Руководитель, Кладовщик) - как у приёмки сканером.
- Продавец: 403 (тест). Наблюдатель: раздел не показывается (складской документ).
- Денежная колонка «Сумма» в списке скрывается без `can_view_purchase_cost`
  (кладовщик суммы итогов не видит, но цену при вводе с накладной заполняет).
- «+ Новая деталь»: `can_manage_parts` (существующее правило каталога).

## 9. Навигация

Пункт «Поступление» перенесён из заглушки в группу «Склад» (рядом с «Приёмкой
сканером») и стал активной ссылкой на `/receipts/`. Заглушек «скоро» в меню не
осталось (тест `test_no_stub_items_left`).
