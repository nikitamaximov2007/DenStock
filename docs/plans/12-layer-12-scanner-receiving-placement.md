# План реализации — Слой 12. Поступление и размещение через сканер

**Статус:** на согласовании (2026-06-25) · **Код не пишется до утверждения.**

---

## 1. Цель слоя

Первый слой, где **скан приводит к складскому действию** — но только к
**приёмке/размещению** уже созданных `PartItem`/`StockLot` из финансово закрытой
партии. Оператор: сканирует экземпляр (или выбирает лот), сканирует ячейку,
подтверждает — объект становится `available`, создаётся `StockMovement`,
обновляется `StockBalance`.

**Главный архитектурный контроль:** все изменения идут **только через сервисы
Слоя 10** (`receive_part_item`/`receive_stock_lot`/`move_part_item`/
`move_stock_lot`). **View не создаёт `StockMovement` напрямую** и не дублирует
ledger-логику. View — это оркестратор: *резолв → проверка → вызов сервиса*. Это
закрепляется тестом (см. §11).

**Чего слой НЕ делает:** продажи, резервы, установки, возвраты, списания,
инвентаризация, аналитика, PDF; не вводит barcode для `StockLot`; не меняет
scanner resolve так, чтобы он сам выполнял действия (резолвер остаётся чистым).

---

## 2. Сценарии слоя

| Сценарий | Как | Сервис Слоя 10 |
|---|---|---|
| Приёмка `PartItem` (с размещением) | скан экземпляра → скан ячейки → подтвердить | `receive_part_item(item, to_location=ячейка)` |
| Перемещение/перерасмещение `PartItem` | скан размещённого экземпляра → скан другой ячейки → подтвердить | `move_part_item(item, ячейка)` |
| Приёмка `StockLot` | выбрать лот (UI) → скан его ячейки → подтвердить | `receive_stock_lot(lot)` |
| Перемещение `StockLot` | выбрать размещённый лот → скан другой ячейки → подтвердить | `move_stock_lot(lot, ячейка)` |
| Подтверждение действия | отдельный шаг «подтвердите» перед вызовом сервиса | — |
| Обработка ошибок скана | unknown/ambiguous/не тот тип → понятный статус-блок, **без действия** | — |

«Приёмка» и «размещение» для `PartItem` — это **одно** действие
`receive_part_item(item, to_location)`: сервис и ставит `available`, и проставляет
`current_location` отсканированной ячейкой. Для уже `available` объекта тот же
экран делает **перемещение** (`move_*`). Диспетчеризация — по статусу (см. §8/§9).

## 3. Как используем scanner resolve (Слой 11)

Каждый скан на экране приёмки проходит через **чистый** `resolve_scan(code)` —
тот же сервис, что и Слой 11; **он не выполняет действий**, только возвращает тип
и объект. Экран приёмки решает, годится ли распознанное для текущего шага.

| Скан | Шаг «деталь» | Шаг «ячейка» |
|---|---|---|
| `ITEM:DS-…` / `DS-…` → `PartItem` | ✅ берём экземпляр | ❌ «ожидается ячейка» |
| `LOC:…` / код ячейки → `StorageLocation` | ❌ «сначала отсканируйте деталь» | ✅ берём ячейку |
| `PartBarcode`/`OEM` → `PartType` | ⚠️ «это вид детали, а не конкретный экземпляр; отсканируйте экземпляр (ITEM:/DS-) или серийник» | ❌ «ожидается ячейка» |
| `ambiguous` (напр. серийник у нескольких) | показываем `candidates`, оператор **выбирает** (не авто-выбор) | показываем `candidates` |
| `unknown` | понятная ошибка + запись `UnresolvedScan` (как в endpoint) | то же |

- **`unknown`** на экране приёмки журналируется так же, как в endpoint Слоя 11 —
  переиспользуем `core.views._record_unresolved` (журнал из server-side вьюхи, а
  не дубль логики).
- **`ambiguous`** — рендерим список `candidates` как выбираемые; клик по варианту
  кладёт его в состояние как выбранный объект. **Никакого авто-выбора.**
- На любом «не том» результате (`unknown`/`ambiguous`/`PartType`/неверный тип
  для шага) **сервис не вызывается** → движение/остаток не меняются (тест §11).

## 4. Важное по `StockLot` (нет своего barcode)

`StockLot` **не имеет** собственного штрихкода/номера (Слой 9 их не вводил, Слой
11 подтвердил решение). Поэтому **скан лота не делаем** и **barcode для `StockLot`
не вводим**. Лот **выбирается из UI**:

- на экране приёмки — **список лотов в статусе `receiving`** (с указанием детали,
  партии, количества и **назначенной ячейки** из Слоя 9), у каждого кнопка
  «Разместить»;
- клик кладёт лот в состояние операции и переводит на шаг «скан ячейки».

**Подбор лота сканом `BatchLine`/`PartType` — НЕ вводим** (усложняет: одна строка
партии может иметь несколько лотов в разных ячейках → снова неоднозначность).
Простой вариант: **UI-выбор лота + скан ячейки** (рекомендация). Скан `PartType`
на шаге выбора лота можно использовать как **фильтр списка лотов** (необязательно,
вынесено в открытые вопросы), но не как прямой выбор лота.

## 5. Сервисы — переиспользуем Слой 10, без дублирования

Используем **существующие** сервисы (точные сигнатуры — `apps/inventory/services.py`):

```python
receive_part_item(item, *, to_location=None, by=None, comment="")  # receiving→available, ставит current_location
receive_stock_lot(lot,  *, by=None, comment="")                    # receiving→available; ячейка = lot.location
move_part_item(item, to_location, *, by=None, comment="")          # перемещение размещённого экземпляра
move_stock_lot(lot,  to_location, *, by=None, comment="")          # перемещение целого лота
```

- **`receive_part_item` принимает `to_location`** — отсканированная ячейка
  становится `current_location` и статусом `available` за один вызов.
- **`receive_stock_lot` НЕ принимает `to_location`** — ячейка лота задана при
  создании (Слой 9). Скан ячейки на приёмке лота — это **подтверждение**: ячейка
  должна совпасть с `lot.location` (см. §9).
- **View вызывает только эти сервисы.** Никаких `StockMovement.objects.create`,
  никакого `_refresh_balance`, никакой записи в `StockBalance` из вьюхи. Вся
  транзакционность/блокировки/журнал/баланс — внутри сервисов (Слой 10). View
  ловит `InventoryError` и показывает сообщение.

> **Контроль архитектуры (главное в слое):** если действие потребует логики,
> которой нет в сервисах, — её добавляем **в `services.py`**, а не во вьюху.
> На Слое 12 такой необходимости нет: все четыре операции уже реализованы.

## 6. Правила

| Правило | Кто гарантирует |
|---|---|
| Принимать можно только объект в статусе `receiving` | `receive_*` отклоняет не-`receiving` (`InventoryError`) |
| После приёмки статус → `available` | `receive_*` |
| Ячейка должна `can_hold_stock()` (active + storage_allowed) | `receive_part_item`/`move_*` (проверка внутри сервиса) |
| Операция создаёт `StockMovement` | сервис (`_record_movement`) |
| Операция обновляет `StockBalance` | сервис (`_refresh_balance`) |
| Повторная приёмка не дублирует движение | `receive_*` отклоняет уже `available` (нет второго движения) |
| Нельзя принять в `inactive`/`storage_allowed=false` | `can_hold_stock()` в сервисе → `InventoryError` |
| Нельзя принять объект из партии без `cost_finalized` | **структурно невозможно**: Слой 8/9 не создают объекты из незакрытой партии. Дополнительно сервис замораживает `landed`-стоимость при создании. View эту инвариантность не дублирует, только опирается на неё |

Все проверки уже живут в сервисах Слоя 10 — Слой 12 их **не переписывает**, лишь
показывает понятные сообщения об ошибках, которые сервисы бросают.

## 7. UI — экран «Приёмка сканером»

Отдельный экран `/scanner/receiving/` (`name="scanner_receiving"`), крупное поле
скана. **Состояние — в скрытых полях формы (stateless, без session)**: шаг
выводится из заполненности полей.

**Состояния-шаги** (выводятся из скрытых `object_kind`/`object_id`/`location_id`):

1. **«Сканируйте деталь»** — нет выбранного объекта. Поле скана активно; ниже —
   список лотов `receiving` с кнопками «Разместить».
2. **«Сканируйте ячейку»** — объект выбран (экземпляр отсканирован / лот выбран),
   ячейка ещё нет.
3. **«Подтвердите»** — объект и ячейка есть; показываем сводку (что, куда, какое
   действие — приёмка/перемещение) и кнопку **«Подтвердить»**.

- **Статус-блоки** `partials/_status_block.html`: «Успешно» (после подтверждения),
  «Ошибка» (InventoryError/неверный тип/unknown), «Требуется действие» (ambiguous —
  список вариантов).
- **Кнопка «Сбросить»** — очищает текущую операцию (возврат к шагу 1).
- **История последних успешных операций** — последние ~10 движений
  `receive_*`/`move_*` **этого пользователя** (`created_by=request.user`) из
  ledger: дата, тип, объект, ячейка. Суммы — только при `can_view_purchase_cost`.
  (Читаем из `StockMovement`, без отдельного хранилища.)
- **No-JS fallback:** экран полностью **server-rendered**. Поле скана — обычный
  `<input>` в `<form>`: **Enter отправляет форму нативно** (без кастомного JS).
  Каждый скан — POST, сервер резолвит и перерисовывает следующий шаг. Опционально
  `scanner.js` возвращает фокус в поле после перезагрузки (мелкое улучшение,
  переиспользуем существующий приём автофокуса).

**Поток (stateless, скрытые поля):**

```
GET            → шаг 1 (пустая форма)
POST scan      → resolve(code); по шагу:
                   нет объекта → ожидаем PartItem → кладём object_kind/id, шаг 2
                   есть объект, нет ячейки → ожидаем StorageLocation → location_id, шаг 3
POST select_lot→ object_kind=stock_lot, object_id=<lot>, шаг 2
POST confirm   → вызвать сервис (диспетчер §8/§9); успех → история + сброс; ошибка → сообщение
POST reset     → пустая форма
```

Скрытые `object_id`/`location_id` управляемы пользователем, но вьюха **всегда
перечитывает** объект (`get_object_or_404`) и доверяет проверкам сервиса —
подмена id ничего не ломает (сервис проверит статус/ячейку/права контекстом).

## 8. Для `PartItem`

1. Скан экземпляра (`ITEM:DS-…`/`DS-…`/серийник) → `resolve_scan` → `PartItem`.
2. Скан ячейки (`LOC:…`/код) → `resolve_scan` → `StorageLocation`.
3. Подтвердить → **диспетчер по статусу**:
   - `status == receiving` → `receive_part_item(item, to_location=ячейка, by=request.user)`
     → `available`, `current_location` = отсканированная ячейка, движение
     `receive_item`, баланс обновлён;
   - `status == available` → `move_part_item(item, ячейка, by=request.user)`
     → `current_location` обновлён, движение `move_item`;
   - иначе (терминальный/прочий) → ошибка «нельзя разместить в текущем статусе».

`current_location` всегда становится **отсканированной** ячейкой (скан решает
размещение). Повторная приёмка уже `available`-экземпляра в ту же ячейку →
`move_part_item` отклонит «уже в этой ячейке» (без дубля движения).

---

## 9. Для `StockLot`

1. Выбрать лот из списка `receiving`-лотов (кнопка «Разместить») — **без скана**
   (у лота нет barcode).
2. Скан ячейки (`LOC:…`/код) → `resolve_scan` → `StorageLocation`.
3. Подтвердить → **диспетчер по статусу**:
   - `status == receiving`:
     - если отсканированная ячейка **совпадает** с `lot.location` →
       `receive_stock_lot(lot, by=request.user)` → `available`, движение
       `receive_lot`, баланс обновлён;
     - если **не совпадает** → ошибка «лот создан для ячейки {lot.location.code};
       отсканирована {ячейка.code}» (без неявного перемещения — рекомендация §
       откр. вопросов);
   - `status == available` → `move_stock_lot(lot, ячейка, by=request.user)` →
     `location` обновлён, движение `move_lot` (с проверкой уникальности ячейки из
     Слоя 10).

**barcode для `StockLot` не вводим.** Скан ячейки на приёмке лота служит
**физическим подтверждением** правильной ячейки (ячейка задана ещё в Слое 9).

## 10. Права

| Роль | Экран приёмки | Действия | Суммы в истории |
|---|---|---|---|
| Администратор / Руководитель | да | да | да |
| Кладовщик | да | да | **нет** |
| Наблюдатель | **нет** (только журнал/остатки Слоя 10) | нет | — |
| Продавец/Мастер | **раздел не виден** | нет | — |

- Экран приёмки — под **`MANAGE_INVENTORY`** (проверка
  `request.user.can_manage_inventory`, как в action-вьюхах Слоёв 8–10). Наблюдатель
  сюда **не входит** (это action-экран; просмотр остатков/журнала ему уже доступен
  отдельно).
- **Суммы** в истории операций (`unit_cost_rub`/`total_cost_rub`) скрыты без
  `can_view_purchase_cost` (контекст `show_costs`).
- Навигация: пункт **«Приёмка сканером»** виден только при `can_manage_inventory`.
  Продавец/Мастер/Наблюдатель его не видят.

9. `StockLot` (выбран из списка) + скан его ячейки + подтверждение → принят.
10. После приёмки `StockLot.status == available`.
11. `StockBalance` обновлён после приёмки `StockLot`.
12. Нельзя принять в `storage_allowed=false` (→ ошибка, нет движения).
13. Нельзя принять в `is_active=false` (→ ошибка, нет движения).
14. `unknown`-скан **не создаёт** движения (и пишет `UnresolvedScan`).
15. `ambiguous`-скан **не создаёт** движения (показывает `candidates`).
16. **View не пишет `StockMovement` напрямую, а вызывает сервис** —
    `unittest.mock.patch("apps.core.views.receive_part_item")`: при подтверждении
    сервис вызван **ровно один раз**, а при замоканном сервисе вьюха **сама не

17. Себестоимость скрыта без `can_view_purchase_cost` (кладовщик не видит сумм в
    истории; админ — видит).

Доп.: скан ячейки на шаге «деталь» → ошибка «ожидается деталь», без действия;
несовпадение ячейки лота с `lot.location` → ошибка, без движения; перемещение
уже `available`-экземпляра в другую ячейку (`move_item`) обновляет два баланса.

## 12. Ручная проверка

1. Войти кладовщиком → «Приёмка сканером».
2. Отсканировать `receiving`-экземпляр (`DS-…`) → шаг «сканируйте ячейку».
3. Отсканировать годную ячейку (`LOC:…`/код) → шаг «подтвердите» со сводкой.
4. Подтвердить → «Успешно»; экземпляр `available`, в «Движениях» — `receive_item`,
   в «Остатках» — строка ячейки; в истории операций — запись.
5. Отсканировать ту же деталь снова + ту же ячейку → «уже в этой ячейке»/«уже
   принят», движение не задвоилось.
6. Выбрать лот из списка `receiving` → отсканировать **его** ячейку → подтвердить →
   лот `available`, движение `receive_lot`, баланс обновлён.
7. Выбрать лот → отсканировать **другую** ячейку → понятная ошибка, без движения.
8. Отсканировать ячейку со `storage_allowed=false` → ошибка, без движения.
9. Отсканировать мусор → «не распознан», запись в «Нераспознанные», без движения.
10. Войти кладовщиком → суммы в истории не видны; админом → видны.
11. Войти продавцом → пункта «Приёмка сканером» нет; прямой URL → 403.
12. Убедиться (в «Движениях»), что все движения имеют автора (`created_by`) —
    доказательство, что прошли через сервис.

## 13. Критерии готовности

1. Скан экземпляра + ячейки + подтверждение принимает/размещает объект **только
   через сервисы Слоя 10**; во вьюхе нет `StockMovement.objects.create` и записи в
   `StockBalance` (доказано тестом-моком §11.16).
2. `PartItem`: `receiving` → `receive_part_item`, `available` → `move_part_item`;
   статус/ячейка/движение/баланс корректны.
3. `StockLot`: выбор из UI + скан ячейки → `receive_stock_lot` (ячейка совпадает)
   или `move_stock_lot` (перемещение); barcode для лота **не введён**.
4. Все правила §6 соблюдены сервисами; повторная приёмка не дублирует движение;
   негодная ячейка отклоняется.
5. `unknown`/`ambiguous`/неверный тип скана **не вызывают** действий (нет движений)
   и понятно сообщаются; `unknown` журналируется в `UnresolvedScan`.
6. Права: экран — `MANAGE_INVENTORY`; продавец/мастер/наблюдатель не входят; суммы
   скрыты без `can_view_purchase_cost`.
7. Границы соблюдены: нет продаж/резервов/установок/возвратов/списаний/
   инвентаризации/аналитики/PDF; scanner resolve остался чистым; ledger-логика
   Слоя 10 не изменена и не продублирована.
8. Все тесты зелёные; `ruff`/`djlint` чисты; `makemigrations --check` чисто (новых
   моделей нет → миграций нет); `manage.py check` без проблем.
