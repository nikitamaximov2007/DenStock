# Раздел 1. Архитектура и развёртывание

**Статус:** утверждён (2026-06-24)

## Форма приложения

Монолитное Django-приложение с серверным рендерингом шаблонов + HTMX для
динамики (живой поиск, сканер, частичное обновление экрана) и Tailwind для
вёрстки. Один процесс, один язык — оптимально для сопровождения нетехническим
владельцем.

## Стек

| Слой | Технология |
|---|---|
| Язык / фреймворк | Python, Django |
| Интерактивность | HTMX + Alpine.js (точечно) |
| Вёрстка | Tailwind CSS |
| База данных | PostgreSQL |
| Запуск | Docker Compose |
| Веб-сервер | Gunicorn за reverse-proxy (отдаёт статику, готов к HTTPS) |

## Принцип целостности

Источник истины об изменениях склада — **неизменяемый журнал движений**
(`StockMovement`). Любое изменение остатка = новая запись движения. Текущие
остатки (`StockLot`, `PartItem`) обновляются **в той же транзакции** с
блокировкой строки (`select_for_update`) и всегда пересобираемы из журнала.

## Развёртывание

- **Docker Compose**: сервисы `web` (Django+Gunicorn), `db` (PostgreSQL),
  `proxy`. Один и тот же compose-файл работает локально и на VPS.
- **Запуск в один клик**: ярлык на рабочем столе → скрипт поднимает контейнеры
  и открывает браузер. Первичная настройка делается один раз.
- **Локальная сеть**: другие устройства заходят по адресу складского ПК.
- **Переезд в облако**: тот же compose + домен + автоматический HTTPS +
  восстановление базы из бэкапа. Код не меняется.

## Резервное копирование

- Ежедневный `pg_dump` в папку бэкапов, хранение нескольких последних копий,
  опционально копия в облачную папку (Яндекс.Диск/Google Drive).
- Кнопка «Сделать копию сейчас» и журнал успешных/неуспешных копий (`BackupLog`).
- Фото и документы партий хранятся в томе и входят в бэкап.
