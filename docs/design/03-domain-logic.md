# Раздел 3. Доменная логика (переработанная)

**Статус:** на согласовании (2026-06-24)

Обозначения: 🔒 — операция выполняется в **транзакции БД с блокировкой**
(`select_for_update`); 📒 — пишется запись в `StockMovement`; 📊 — обновляется
кэш `StockBalance` в той же транзакции; 📝 — пишется `ActivityLog`.

---

## 3.1. Жизненный цикл партии

```
DRAFT ──► ORDERED ──► IN_TRANSIT ──► ARRIVED ──► RECEIVING ──► ACCEPTED
создана    заказана     в пути       прибыла     принимается    принята
                                                                   │
                                                                   ▼
                                                          COST_CALCULATED ──► CLOSED
                                                        себестоимость рассч.   закрыта
            (из ранних статусов, пока нет физ. остатка)
                          └──────────────► CANCELLED отменена
```

| Статус | Что происходит | Остаток создан? | Доступно к продаже? |
|---|---|---|---|
| `DRAFT` | Создана карточка партии, поставщик, плановые расходы. Полностью редактируема | нет | нет |
| `ORDERED` | Заказ размещён, номера заказа/счёта | нет | нет |
| `IN_TRANSIT` | Отгружена, `ship_date` | нет | нет |
| `ARRIVED` | Физически на складе, `arrival_date`, ещё не обработана | нет | нет |
| `RECEIVING` | Идёт приёмка: создаются `BatchLine`, `PartItem`/`StockLot` (зона приёмки или сразу ячейка) с **базовой** ценой | да (база) | **нет** |
| `ACCEPTED` | Приёмка завершена, количества подтверждены | да | **нет** |
| `COST_CALCULATED` | Распределены накладные, посчитан и **заморожен** landed cost → `cost_finalized=true` | да (landed) | **да** |
| `CLOSED` | Партия закрыта, расходы больше не меняются | да | да |
| `CANCELLED` | Отмена. Разрешена только пока **нет** физического остатка (до `RECEIVING`). После приёмки — только через списание/обратные движения |

**Ключ:** физический остаток появляется на `RECEIVING`, но **продавать нельзя**,
пока не наступит `COST_CALCULATED`. Это защищает прибыль от расчёта по неполной
себестоимости.

---

## 3.2. Когда партия становится доступной к продаже

Единственное условие: `batch.cost_finalized = true` (статусы `COST_CALCULATED`
или `CLOSED`).

Сервисы продажи, установки и резерва **в начале** проверяют этот флаг. Если
`false` — операция отклоняется с сообщением:

> «Партия ещё не закрыта. Продажа невозможна — себестоимость не зафиксирована.»

Инвариант действует и для поштучных (`PartItem.batch`), и для количественных
(`StockLot.batch`).

---

## 3.3. Расчёт landed cost (переход `ACCEPTED → COST_CALCULATED`) 🔒📝

Одна транзакция:

1. `overhead_rub = (shipping + customs + fees + extra) × exchange_rate` —
   накладные партии в рублях.
2. База распределения (`allocation_base`):
   - `BY_VALUE` (по умолчанию): `вес строки = base_unit_cost_rub × quantity_received`;
   - `BY_QUANTITY`: `вес строки = quantity_received`.
3. Для каждой строки `BatchLine`:
   `allocated_overhead_rub = overhead_rub × (вес строки / Σ весов)`.
4. `landed_unit_cost_rub = base_unit_cost_rub + allocated_overhead_rub / quantity_received`.
5. **Округление до копеек.** Остаток от округления (чтобы `Σ allocated = overhead_rub`
   точно) добавляется к строке с наибольшим весом.
6. `landed_unit_cost_rub` замораживается в `BatchLine`, затем копируется в
   `unit_cost_rub` каждого связанного `PartItem` и `StockLot`.
7. `status = COST_CALCULATED`, `cost_finalized = true`.

После этого `BatchLine` неизменяема.

**Пример (как у заказчика):** товары 500 000 ₽, доставка 80 000 ₽, таможня
40 000 ₽ → накладные 120 000 ₽ распределяются по стоимости; реальная база партии
620 000 ₽.

## 3.4. FIFO и ручной выбор партии (продажа/установка количественной детали)

- При добавлении позиции система показывает **доступные лоты** этой детали
  (только `cost_finalized=true`), отсортированные по дате прихода партии —
  **старые сверху (FIFO)**.
- По умолчанию подставляется старший лот. Если запрошенное количество больше
  одного лота — FIFO разбивает по нескольким лотам (каждый лот = своя строка
  движения и своя себестоимость).
- **Ручной выбор:** продавец может выбрать другой лот/партию (старый лежит
  глубоко, новый ближе, лучшее состояние и т.п.).
- **Поштучные детали** FIFO не используют: выбирается конкретный `PartItem`
  (скан/поиск), у него своя себестоимость.

---

## 3.5. Продажа 🔒📒📊📝

1. Создаётся черновик `Sale` (клиент, сотрудник). Черновик без проведения можно
   удалить — остаток он не трогает.
2. Добавление позиций (поиск/сканер):
   - поштучно → конкретный `PartItem` (статус `IN_STOCK`, партия `cost_finalized`);
   - количественно → лот по FIFO или вручную, количество, цена
     (по умолчанию `recommended_price`; если ниже `min_price` — предупреждение),
     скидка.
3. **Проведение** — одна транзакция:
   - 🔒 `select_for_update` на затрагиваемых `StockLot`/`PartItem` и строке `StockBalance`;
   - проверка `batch.cost_finalized = true`;
   - проверка доступного количества (`StockLot.quantity` / статус `PartItem`);
   - заморозка `unit_cost_rub` в `SaleLine` (копия себестоимости лота/экземпляра);
   - 📒 движение `SALE`;
   - расход: количественно — `StockLot.quantity -= qty` (CHECK ≥ 0); поштучно —
     `PartItem.status = SOLD`, `location = null`;
   - 📊 `StockBalance.physical_qty -= qty`, пересчёт `available_qty`;
   - `profit = line_total − qty × unit_cost_rub`;
   - если продажа по резерву — `Reservation → COMPLETED_BY_SALE`, резерв снят;
   - 📝 запись в журнал.
4. `Sale.status = COMPLETED`. **Удалить нельзя.**

---

## 3.6. Установка в ремонтируемую технику 🔒📒📊📝

Отдельный документ `Installation` (мастер, ремонтируемая техника,
`price_charged`). Механика проведения идентична продаже, но:

- движение `INSTALL`;
- в аналитике разделяется: **чистые продажи** vs **установки в ремонт** vs
  **общий оборот по запчастям**.

> Дополнительно поддерживается промежуточная **выдача в ремонт**
> (`ISSUE_TO_REPAIR`): уменьшает доступный остаток (`issued_qty`), деталь
> числится «в ремонте» до установки (`INSTALL`, окончательный расход) или
> возврата (`RETURN`). `Доступно = физический − резервы − выдано в ремонт`.

---

## 3.7. Резервирование 🔒📒📊📝

1. Создание `Reservation` (клиент, телефон, деталь, кол-во, опц. конкретный
   `PartItem`/`Batch`, срок `expires_at`).
2. Проверки: партия `cost_finalized=true`; доступно ≥ кол-во.
3. Транзакция: 🔒 блокировка остатка → 📒 движение `RESERVE` →
   📊 `StockBalance.reserved_qty += qty`, пересчёт `available_qty`;
   поштучно — `PartItem.status = RESERVED`. Физический остаток **не уменьшается**.
4. Статусы: `ACTIVE` → `COMPLETED_BY_SALE` (преобразован) /
   `CANCELLED` (вручную: 📒 `UNRESERVE`, снятие) /
   `EXPIRED` (по сроку: фоновая задача 📒 `UNRESERVE`, снятие).
5. Зарезервированное **не показывается доступным** другим клиентам.
6. «Преобразовать в продажу» — создаёт `Sale` из резерва; при проведении резерв
   закрывается, физический остаток уменьшается.

---

## 3.8. Отмена продажи 🔒📒📊📝

Проведённую продажу **нельзя удалить** — только отменить обратной операцией.

- Транзакция: для каждой `SaleLine` 🔒 блокировка → 📒 обратное движение
  (восстановление в исходный лот/ячейку) → восстановление остатка:
  количественно — `StockLot.quantity += qty` (вернуть в исходный лот/ячейку);
  поштучно — `PartItem.status = IN_STOCK`, `location =` исходная ячейка;
- 📊 `StockBalance.physical_qty += qty`, пересчёт;
- `Sale.status = CANCELLED`; исходные продажа и движения **сохраняются**,
  отмена добавляет новые движения;
- 📝 журнал (`SALE_CANCEL`, комментарий/причина);
- в аналитике отменённые продажи исключаются из выручки и прибыли.

---

## 3.9. Возврат 🔒📒📊📝

Привязан к конкретной продаже (`Return.sale`, `ReturnLine.sale_line`).

- Система определяет: к какой продаже относится деталь; можно ли вернуть на
  склад; в какую ячейку (по умолчанию исходная); состояние.
- Транзакция: 📒 движение `RETURN` → восстановление остатка с **исходной
  партией и себестоимостью** (количественно — лот в `to_location`; поштучно —
  `PartItem.status = RETURNED → IN_STOCK`, `location = to_location`);
  📊 обновление `StockBalance`.
- Если деталь повреждена и не подлежит возврату на склад — вместо восстановления
  оформляется **списание** (маршрут на `WriteOff`).
- В аналитике возврат уменьшает выручку/прибыль периода.

Отличие от отмены: отмена аннулирует всю продажу (обычно сразу); возврат —
частичный физический возврат позже, с проверкой состояния.

---

## 3.10. Списание 🔒📒📊📝

- Причина из списка (`DEFECT`/`DAMAGE`/`LOSS`/`USED`/`UNUSABLE`/`INV_SHORTAGE`/
  `OTHER`); **комментарий обязателен**.
- Подтверждение ролью «Руководитель»/«Администратор»:
  `WriteOff.status: PENDING → CONFIRMED`. На первом этапе при совмещении ролей —
  мягкий шаг-подтверждение.
- При подтверждении — транзакция: 🔒 блокировка → проверка доступного →
  📒 движение `WRITEOFF` → расход (количественно `StockLot.quantity -= qty`;
  поштучно `PartItem.status = WRITTEN_OFF`, `location=null`) →
  📊 `StockBalance.physical_qty -= qty`, пересчёт; стоимость потери = снимок
  `unit_cost_rub`.
- **Удалить нельзя.** Ошибочное списание исправляется обратной корректировкой.

---

## 3.11. Инвентаризация 🔒📒📊📝

1. `InventoryCount` с областью (`scope_location`: склад/зона/ячейка), статус `DRAFT`.
2. `COUNTING`: сканирование/ввод `counted_qty`. Система фиксирует `expected_qty`
   (из `StockLot`/`PartItem` в области).
3. `REVIEW`: `diff = counted − expected` по каждой строке; видны недостачи и излишки.
4. `APPLIED` (подтверждение, роль «Руководитель») — транзакция: для каждой
   строки с расхождением 🔒 → 📒 движение `INV_SURPLUS` / `INV_SHORTAGE` →
   корректировка `StockLot`/`PartItem` → 📊 обновление `StockBalance`; 📝 журнал.
5. Применённую инвентаризацию **удалить нельзя**; дальнейшие правки — новыми
   корректировками. Отчёт сохраняется.

---

## 3.12. Обновление `StockBalance` (read-optimized кэш)

`StockBalance` — **не источник истины**. Источники: `StockMovement`, `PartItem`,
`StockLot`, `Reservation`, документы операций. Кэш полностью пересобираем.

Правила (условия заказчика):

1. **Обновляется только внутри той же транзакции**, где создаётся движение.
2. На расходных операциях — `select_for_update` на строке баланса/лота/экземпляра,
   чтобы остаток **не ушёл в минус**.
3. **Ручное редактирование из интерфейса запрещено** (read-only, нет форм правки).
4. Management-команда **`recalc_stock_balance`** — полный пересчёт из первичных
   данных.
5. Management-команда **`check_stock_balance`** — проверка целостности: сравнение
   `StockBalance` с остатком, рассчитанным по движениям/лотам/экземплярам;
   отчёт о расхождениях.
6. **Тест обязателен:** одновременная продажа последней детали двумя
   пользователями → ровно один успех, второй — честная ошибка, минуса нет,
   баланс корректен.

Поддерживаемые поля: `physical_qty`, `reserved_qty`, `issued_qty`,
`available_qty = physical − reserved − issued`.

---

## 3.13. Запрет отрицательных остатков (три рубежа защиты)

1. **БД:** `CHECK (quantity >= 0)` на `StockLot`; статусные переходы `PartItem`.
2. **Сервисный слой:** транзакция + `select_for_update` + явная проверка
   доступного **перед** записью расхода.
3. **Бизнес-правило:** нельзя продать/списать/установить экземпляр не в статусе
   `IN_STOCK`/`RESERVED` (для своего резерва).

При гонке двух расходов на последний остаток второй получает ошибку
«Недостаточно остатка», а не отрицательное число.

---

## 3.14. Операции, требующие транзакции и блокировки

| Операция | Что блокируется (`select_for_update`) |
|---|---|
| Поступление (приёмка) | `NumberSequence`, создаваемые `StockLot`/`PartItem` |
| Расчёт landed cost | строки `BatchLine`, связанные `StockLot`/`PartItem` |
| Продажа / Установка | `StockLot`/`PartItem` позиций, `StockBalance` детали |
| Выдача в ремонт | `StockLot`/`PartItem`, `StockBalance` |
| Резерв (создание/снятие/истечение/преобразование) | `StockBalance`, при поштучном — `PartItem` |
| Отмена продажи | `StockLot`/`PartItem`, `StockBalance` |
| Возврат | `StockLot`/`PartItem`, `StockBalance` |
| Подтверждение списания | `StockLot`/`PartItem`, `StockBalance` |
| Применение инвентаризации | затрагиваемые `StockLot`/`PartItem`, `StockBalance` |
| Выдача любого внутреннего номера | `NumberSequence` |

Общий принцип: **любое изменение остатка** = одна транзакция, содержащая
движение + изменение первичной сущности + обновление `StockBalance`.

---

## 3.15. Что нельзя удалять (только отмена обратной операцией)

| Объект | Правило |
|---|---|
| `StockMovement` | Только добавление; никогда не редактируется и не удаляется |
| `Sale`, `Installation` | После проведения — только `CANCELLED` + обратные движения |
| `WriteOff` (подтверждён) | Только обратная корректировка |
| `Return` | Сохраняется; ошибки — новыми операциями |
| `InventoryCount` (применён) | Только новые корректировки |
| `Reservation` | Сохраняется как история; статусы `CANCELLED`/`EXPIRED` |
| `BatchLine` | Неизменяема после `cost_finalized` |
| Черновики (`DRAFT`) | Непроведённые `Sale`/`Installation`/`InventoryCount` и `Batch` в `DRAFT` без остатка **можно** удалить — они не затронули склад |
