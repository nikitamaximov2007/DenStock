# Дизайн системы DenStock

Складской учёт запчастей для ремонта автомобилей, снегоходов, квадроциклов,
катеров и яхт. Веб-приложение для быстрого ответа клиенту, приёмки партий,
поиска, перемещения, продажи и инвентаризации деталей.

> Главная цель: во время звонка клиента за несколько секунд найти деталь и
> увидеть её количество, стоимость и точное местоположение на складе.

## Ключевые архитектурные решения

| Решение | Суть |
|---|---|
| Стек | Django + HTMX + Tailwind + PostgreSQL, запуск Docker Compose |
| Монолит | Один проект с модулями; без микросервисов и отдельного фронтенда |
| Развёртывание | Локально на складе сейчас, переносимо на VPS позже без изменений кода |
| Сопровождение | Нетехнический владелец — приоритет «просто работает» |
| Источник истины | Неизменяемый журнал `StockMovement`; остатки пересобираемы из него |
| `StockBalance` | Read-optimized кэш доступности, **не** источник истины |
| Себестоимость | landed cost (с доставкой/таможней), замораживается до допуска к продаже |
| Допуск к продаже | Только при `cost_finalized=true` (партия с зафиксированной себестоимостью) |
| Списание остатка | Только через движения, в транзакции с `select_for_update`; минус запрещён |
| Документы | Не удаляются — отменяются обратной операцией |
| Учёт | Поштучный (`PartItem`) и количественный (`StockLot`) — раздельные правила |
| Интерфейс | Складской операционный: крупные кнопки, режим сканера, статусы цвет+текст+иконка |

## Порядок сборки

Система строится **вертикальными слоями** — см.
[05-roadmap.md](05-roadmap.md). Каждый слой запускается, проверяется и
коммитится отдельно. Денежный контур (продажа) появляется только после склада,
партий и остатков. Финальный слой проверяет 16 критериев готовности ТЗ.
