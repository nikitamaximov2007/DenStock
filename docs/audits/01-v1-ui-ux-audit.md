# Аудит v1 — UI/UX системы DenStock

**Дата:** 2026-06-30 · после `v1.0-local` и UI-polish галереи (`bc0bbb8`, `29a5888`).
**Статус:** только анализ. Код не менялся. Реализацию ведём отдельными маленькими этапами (§F).

**Метод аудита:** статический разбор шаблонов (`templates/`), стилей (`static/css/app.css`),
view-контекста и навигации (grep + чтение). Браузерные скриншоты не делались: Docker в
окружении недоступен, Playwright не ставился (не добавляем зависимость ради аудита). Поведение
подтверждено существующими тестами (476 зелёных). Выводы привязаны к конкретным файлам/строкам.

## A. Executive summary

**Общее состояние.** Функционально продукт полный (склад→продажи→отчёты→фото→backup), но
**визуальный слой остался на уровне «скелета Слоя 1»**: `static/css/app.css` сам себя описывает
как «минимальный скелетный стиль» (строка 1) и почти не вырос — ~147 строк, **0 `@media`-запросов**,
один цвет-акцент, один тип бейджа. README заявляет стек «Django + HTMX + **Tailwind**», но Tailwind
в проекте нет — всё на ручном CSS. В итоге экраны читаемы, но выглядят черновыми и местами
**вводят в заблуждение** (мёртвый дашборд, неотличимые статусы, недостижимые записи за 50-й).

**Главные проблемы (суть).**
1. **Дашборд — мёртвый плейсхолдер**: 4 плитки показывают «—» ([core/dashboard.html](templates/core/dashboard.html)). Первый экран после входа выглядит сломанным.
2. **Нет пагинации** ни на одном списке, хотя `ListView.paginate_by = 50` → записи за 50-й **недостижимы** (поиск/фильтр это не компенсируют).
3. **Статусы — плоский текст** в 24 местах (`{{ obj.get_status_display }}`), единого визуального языка нет: «Черновик» и «Проведена» выглядят одинаково.
4. **Флагманский поиск не стилизован**: [core/search.html](templates/core/search.html) ссылается на классы (`.search-result`, `.scanner-page__input`, `.search-result__avail`), которых **нет в CSS** → результаты рендерятся как голый текст.
5. **Опасные действия** («Удалить», «Снять», «Отменить», «Деактивировать», «Списать») — без отдельного цвета и **без подтверждения** (`confirm()` — 0 на весь проект).
6. **Деньги/числа** — сырой вывод, выровнены влево, без разрядов; в таблицах «съезжают» визуально.

**Главные быстрые улучшения (high value / low risk).**
- Добавить в CSS **дизайн-фундамент** (токены + компоненты: `.pill`, `.btn--danger/secondary`, `.empty`, `.card`, `.num`) — чисто аддитивно. Это сразу «оживляет» поиск (классы уже используются в шаблоне) — см. §G.
- Заменить плитки дашборда на **рабочие KPI/быстрые ссылки** (данные уже есть в `apps/reports.services`).
- Ввести **status-pill** и применить в списках/деталях.
- Добавить **общий партиал пагинации** и подключить в `*_list.html`.

## C. Проблемы по приоритетам

Формат: **где** · **почему** · **как** · **файлы** · **риск** · **тест**.

### P0 — мешает использовать / вводит в заблуждение

**P0-1. Дашборд — мёртвые плитки.**
- Где: [core/dashboard.html](templates/core/dashboard.html:7-24) — 4 `.tile__value` = «—».
- Почему: первый экран после входа выглядит сломанным/незаполненным; не даёт ни данных, ни навигации в сценарии.
- Как: наполнить реальными KPI из `apps/reports.services` (всего деталей, доступно, продажи сегодня — под `can_view_*`) **или** заменить на блок «быстрые действия» (Поиск, Приёмка, Продажа, Отчёты). Денежную плитку — только под `can_view_purchase_cost`.
- Файлы: `core/dashboard.html`, `apps/core/views.py` (`dashboard`).
- Риск: низкий (view только читает существующие сервисы; без новых моделей).
- Тест: render-тест дашборда (200 + наличие чисел/ссылок); проверка, что денежный KPI скрыт без права.

**P0-2. Нет пагинации в списках.**
- Где: все `*_list.html` (напр. [inventory/item_list.html](templates/inventory/item_list.html), [sales/sale_list.html](templates/sales/sale_list.html)); во view стоит `paginate_by = 50`, но в шаблонах **нет** `page_obj`/`is_paginated` (grep — 0 совпадений).
- Почему: записи за 50-й **недостижимы** — выглядит как «больше ничего нет». Прямая потеря данных для пользователя.
- Как: общий партиал `partials/_pagination.html` (предыдущая/следующая/№ страницы, сохраняя query) + подключить в списках; контексты уже содержат `page_obj` (ListView).
- Файлы: новый `partials/_pagination.html`; `*_list.html` (item/lot/movement/balance/sale/reservation/repair/return/writeoff/stocktaking/part/batch/user).
- Риск: средний (много файлов, но правка однотипная и аддитивная).
- Тест: создать >50 объектов → список показывает пагинатор и вторую страницу.

**P0-3. Пустой список справочника без empty-state.**
- Где: [directories/list.html](templates/directories/list.html:24-54) — цикл `{% for row in rows %}` **без** `{% empty %}` → при отсутствии данных видна таблица из одних заголовков (выглядит сломанной).
- Почему: непонятно — «пусто» или «ошибка»; нет CTA «Добавить».
- Как: добавить `{% empty %}`-строку с понятным текстом и (для `can_manage`) ссылкой «Добавить».
- Файлы: `directories/list.html` (+ единый компонент `.empty`, см. §E).
- Риск: низкий.
- Тест: рендер пустого справочника содержит текст empty-state.

### P1 — сильно портит UX/визуал

**P1-1. Флагманский поиск не стилизован.**
- Где: [core/search.html](templates/core/search.html) использует `.search-form`, `.scanner-page__input`, `.search-result`, `.search-result__avail` — **ни одного нет** в `app.css`.
- Почему: главный сценарий («за секунды найти деталь») выглядит как неоформленный текст; поле ввода — браузерный дефолт.
- Как: добавить эти классы в CSS (карточка результата, акцент на «Доступно», крупное поле). Шаблон менять не нужно — классы уже проставлены.
- Файлы: `static/css/app.css` (только добавление).
- Риск: минимальный (аддитивный CSS).
- Тест: существующие тесты поиска зелёные; (опц.) ассерт наличия классов.

**P1-2. Статусы — единый плоский текст.**
- Где: 24× `get_status_display` (item/lot/batch/sale/reservation/repair/return/writeoff/stocktaking), напр. [sale_list.html:41](templates/sales/sale_list.html), [item_list.html:53](templates/inventory/item_list.html), [returns/return_detail.html](templates/returns/return_detail.html). На весь проект — 1 `.badge` и 1 `.status--`.
- Почему: «Черновик», «Проведена», «Списан», «Карантин» визуально неотличимы → оператор не считывает состояние с одного взгляда.
- Как: компонент `.pill` + модификаторы по семантике (нейтральный/в работе/успех/опасность/карантин); маппинг «статус→класс» через мелкий шаблонный include или inclusion-tag (без изменения моделей/сервисов).
- Файлы: CSS + `partials/_status_pill.html`; затем точечно списки/детали.
- Риск: средний (много мест, но чисто презентационно).
- Тест: рендер списка содержит `pill pill--…` для известного статуса.

**P1-3. Деньги/числа без формата и выравнивания.**
- Где: [sale_list.html:43-44](templates/sales/sale_list.html), [sale_detail.html](templates/sales/sale_detail.html), reports — `{{ value }}`/`{{ value }} ₽` сырьём, влево.
- Почему: суммы и количества трудно сравнивать; «1000» и «999» прыгают по ширине; нет разрядов.
- Как: класс `.num` (right-align, табличные цифры) для числовых `td`; единый формат денег (разряды + ₽) — фильтром/`floatformat`/`intcomma` (`django.contrib.humanize`, это stdlib-приложение Django, не внешняя зависимость).
- Файлы: CSS + числовые `td` в таблицах (sales/reports/inventory/…); опц. `humanize` в `INSTALLED_APPS`.
- Риск: низкий-средний.
- Тест: рендер не падает; (опц.) числовая ячейка имеет класс `.num`.

**P1-4. Опасные действия без цвета и подтверждения.**
- Где: «Снять» ([sale_detail.html:96](templates/sales/sale_detail.html), returns), «Деактивировать» ([directories/list.html:44](templates/directories/list.html)), «Отменить» (reservation), «Списать», «Удалить» (фото). `confirm()` — 0 на проект.
- Почему: разрушающее действие выглядит как обычная кнопка; случайный клик не страхуется.
- Как: `.btn--danger` (приглушённо-красная) для разрушающих; ненавязчивое подтверждение — нативный `confirm` на форме или маленький `data-confirm` хэндлер в существующем JS-стиле (vanilla, без зависимостей).
- Файлы: CSS + точечно кнопки; опц. крошечный `static/js/confirm.js`.
- Риск: низкий (презентация + опц. подтверждение; бизнес-логику не трогаем).
- Тест: POST по-прежнему работает (подтверждение — только client-side); существующие тесты зелёные.

**P1-5. Detail-страницы — стопка таблиц без иерархии.**
- Где: [sale_detail.html](templates/sales/sale_detail.html), [returns/return_detail.html](templates/returns/return_detail.html), item/lot/repair/writeoff/stocktaking detail — 2-4 `.table` подряд (свойства/позиции/итоги) + формы; статус спрятан строкой первой таблицы.
- Почему: нет визуальных «зон» (главное/свойства/позиции/итоги/действия); статус не виден сразу; CTA и опасные действия в одном ряду.
- Как: ввести `.card`/`.section` обёртки и единый `page-header` (заголовок + статус-pill + основные действия справа); итоги выделить.
- Файлы: CSS + detail-шаблоны (поэтапно, по одному модулю).
- Риск: средний (вёрстка), но по одному экрану — безопасно.
- Тест: render-тесты деталей зелёные (контент тот же).

**P1-6. Навигация плоская + мёртвые ссылки.**
- Где: [context_processors.py](apps/accounts/context_processors.py) — ~20 пунктов одним списком; «Поступление» и «Статистика» — `stub=True` (рендерятся как серый `.sidebar__soon`, ведут в никуда).
- Почему: длинный неструктурированный список тяжело сканировать; видимые «заглушки будущего» выглядят недоделанностью.
- Как: сгруппировать в секции (Склад / Операции / Документы / Отчёты / Администрирование) с заголовками; **убрать stub-пункты** из меню до их реализации.
- Файлы: `context_processors.py` (структура), `base.html` (рендер групп), CSS (заголовки секций).
- Риск: средний (меню — сквозной элемент; покрыто `test_roles` навигации).
- Тест: `test_roles` навигации зелёные; стабы не отображаются.

## D. Компонентные проблемы

- **Buttons.** Два мира: `.btn/.btn--primary` (52×) и **голые** `<button type="submit">` (55×). Нет `--secondary`/`--danger`/`--ghost`, нет единых размеров (`.btn` 12-18px против inline-form 6-10px). → Ввести вариативную систему кнопок; голые сабмиты привести к классам.
- **Cards.** Карточки как сущности нет — есть только `.tile` (дашборд) и `.table`. Detail-страницы лепят таблицы встык. → `.card`/`.section` + `.page-header`.
- **Forms.** `as_p` + единственный сабмит; нет required-маркеров, help-текста, группировки, «Отмена». `.form input` стилизован только для `text/password` (не для `select`, `number`, `date`, `file`). → Единый стиль полей + cancel.
- **Tables.** База ок (`.table`), но: числа влево, длинные `name`/OEM/серийник не контролируются (нет `word-break`/ширин), `colspan` захардкожен, нет zebra/hover строки, нет «sticky» заголовка. → `.num`, контроль переноса, hover-строки.
- **Badges/Statuses.** Один зелёный `.badge`; статусы сущностей — текст. → `.pill` + семантические модификаторы; маппинг статусов.
- **Empty states.** Разнобой: где-то `<td colspan class="muted">X нет.</td>`, где-то (directories) — вообще нет. → Компонент `.empty` (иконка + текст + опц. CTA), единый текст.
- **Detail pages.** Нет иерархии «главное→свойства→позиции→итоги→фото→опасное». Статус не выделен у заголовка. → `page-header` со status-pill + зоны.
- **Gallery.** После polish ок: основное фото ≤400×320, превью 96×96 приглушены, выбранное подсвечено, primary помечен badge. Замечания: (1) badge «Основное» виден **только** под `can_manage_images` (наблюдатель не видит, какое фото основное); (2) у PartItem с fallback-фото вида показывается «Фотографий нет.» при наличии крупного типового фото — формулировку смягчить («Свои фото не загружены; показано типовое фото вида»).
- **Reports.** KPI как key-value таблицы — читаемо, но денежные суммы сырьём; «CSV» неприметен; активный пресет `muted`. Разделение по `show_costs` работает корректно (финансы скрыты) — хорошо.
- **Responsive.** **0 `@media`.** Сайдбар фиксирован 220px, `.layout` — flex без переноса; на 390px контент сожмётся, длинные таблицы дадут горизонтальный скролл/разъезд. Desktop/tablet терпимо, mobile — деградирует.

## E. Рекомендуемая дизайн-система

- размеры шрифта: `--fs-sm/base/lg` (13/15/20).

**Компоненты (унифицировать как переиспользуемые классы/партиалы):**
