# DenisStock ChatGPT Knowledge Base

Этот файл загружается в ChatGPT как база знаний. Он описывает ТОЛЬКО реально
существующие функции системы DenisStock. Отвечая пользователю, опирайся строго
на этот документ.

## Product summary

DenisStock - веб-система складского учёта запчастей (авто, снегоходы,
квадроциклы, катера, яхты) для небольшого бизнеса. Главный сценарий: при
звонке клиента за секунды найти деталь, увидеть количество, цену и ячейку
хранения. Система ведёт каталог, приход (поступления/партии), остатки по
ячейкам, журнал движений, резервы, продажи, ремонт, возвраты, списания,
инвентаризацию, отчёты, статистику, пользователей с ролями и бэкапы.
Интерфейс русскоязычный, работает в браузере, есть мобильное меню.

## Actual navigation map

Меню слева, группы и пункты (route, кто видит, что делает):

Группа «Обзор»:
- Главная (/) - все: KPI-плитки, «Требует внимания» (низкие остатки,
  карантин), «Быстрые действия».
- Поиск детали (/search/) - все: поиск по названию/OEM/штрихкоду/серийнику/
  внутреннему номеру/технике; показывает доступность, цены, ячейки, партии.
- Сканер (/scanner/) - все: определить, что за код (экземпляр/ячейка/партия/
  деталь). Поле сканера есть также в шапке каждой страницы.

Группа «Каталог»:
- Детали (/parts/) - все смотрят; создание/правка: Администратор, Руководитель
  (кнопка «Добавить деталь»). Карточка: категория, производитель, режим учёта
  (Поштучный/Количественный), цены, минимальный остаток, OEM/штрихкоды,
  применимость, фото, печать этикетки.
- BRP-каталог (/brp/) - все смотрят: справочник дилерского прайса BRP
  (~126 тысяч позиций), НЕ остатки. Поиск по номеру BRP/заменам/описанию;
  статусы OBS/USE/VIN/LIQ/UCP (легенда видна на странице: OBS - снято с
  производства, USE - была замена номера, VIN - винтажный склад с доставкой
  25$, LIQ - последние остатки у завода, UCP - уточняется); цена клиента =
  розница USD x общий текущий курс x (1 + наценка BRP/100), итог округляется
  до ЦЕЛОГО рубля (ROUND_HALF_UP, копеек в цене нет; настройки:
  /directories/price-settings/, админ и руководитель). «Создать карточку» (админ/руководитель) создаёт только
  карточку (не количество); «Учесть наличие» (роли склада) открывает черновик
  «Инвентаризации начальных остатков» (документ поступления от поставщика
  «Стартовый ввод»): остаток появится после «Провести поступление».
- Polaris-каталог (/polaris/) - все смотрят: отдельный справочник дилерского
  прайса Polaris, НЕ остатки. Поиск по part_number, superseded_number,
  названию и штрихкоду склада. Цена клиента считается по общему текущему курсу
  и отдельной наценке Polaris: розница USD x курс x (1 + наценка/100), ROUND_HALF_UP до целого
  рубля. Exact part_number важнее superseded_number; source цены может быть
  связанная позиция, но identity остаётся exact. Если один номер есть в BRP и
  Polaris, общий поиск показывает оба варианта и не выбирает каталог молча.
- Партии (/batches/) - Администратор, Руководитель, Кладовщик (кладовщик без
  закупочных сумм): технический приход с валютой, доп. расходами и landed cost.
- Справочники (/directories/) - категории, производители, единицы, виды/марки/
  модели техники, поставщики. Правка: Администратор, Руководитель.

Группа «Склад» (Администратор, Руководитель, Кладовщик; Наблюдатель -
только просмотр экземпляров/лотов/движений/остатков):
- Экземпляры (/inventory/) - поштучные детали со статусами.
- Лоты (/inventory/lots/) - количественный учёт.
- Движения (/inventory/movements/) - журнал всех операций, только чтение.
  В таблице показывается exact-артикул, а не `лот #...`: snapshot связанного
  действия имеет приоритет, затем exact BRP material_no / Polaris part_number /
  основной неаналоговый номер карточки. Количество и рубли отображаются целыми
  через Decimal ROUND_HALF_UP, без изменения значений в БД.
- Остатки (/inventory/balance/) - live-срез из `StockLot` и `PartItem`,
  агрегированный по exact part identity + ячейка: физически, доступно, резерв,
  карантин. `StockBalance` остаётся пересобираемым кэшем, не source of truth.
- Поступление (/receipts/) - документ прихода РЕАЛЬНОЙ поставки: черновик ->
  позиции (деталь, количество, цена ₽, ячейка) -> «Провести поступление».
  Кнопка «+ Новая деталь» (только Администратор/Руководитель) создаёт
  карточку и возвращает в черновик с подстановкой. Первичный ввод из
  «Инвентаризации ячейки» в этом списке НЕ показывается (его документы -
  IC-номера в разделе «Инвентаризация»).
- Приёмка сканером (/scanner/receiving/) сохраняет прежний поток экземпляров:
  ITEM:/DS-номер или серийник -> скан ячейки -> «Подтвердить». Обычные
  exact-артикулы идут в session-очередь: повторные сканы увеличивают количество,
  обновление страницы очередь не теряет, до проведения склад не меняется.
  Одна существующая ячейка назначается автоматически, несколько требуют явного
  выбора с показом физического остатка/резерва/доступного. Новый exact из BRP
  или Polaris без остатка позволяет выбрать активную ячейку и при проведении
  создаёт карточку и первый лот; /receipts/ не пополняется. Очередь сгруппирована
  по ячейкам, каждая группа проводится отдельно и атомарно через ADJUST_IN с
  `document_type=found_addition`. Уникальный серверный token не допускает
  двойного POST. BRP replacement / Polaris superseded используются только как
  price source; identity остаётся exact. Совпавшие номера BRP/Polaris требуют
  выбора производителя. Неизвестный во всех каталогах код ничего не создаёт.
- Перемещение (/scanner/move/) принимает ITEM barcode, заводской barcode и
  нормализованный номер детали. Для bulk показывает live-источники по ячейке и
  состоянию, разрешает частичное количество; резерв не перемещается, карантин
  сохраняет состояние. Операция атомарна, блокирует строки, пишет один
  `StockTransfer` с exact snapshot и одну или несколько строк `StockMovement`,
  связанных `document_type=stock_transfer`. Уникальный token защищает от
  повторного POST. Исторические `InventoryCountingLine` не переписываются.
- Инвентаризация ячейки (/inventory/counting/) - Администратор, Руководитель,
  Кладовщик: выбрать место -> пикать всё содержимое подряд (скан + Enter) ->
  система группирует одинаковые номера, распознаёт склад, затем BRP/Polaris ->
  «Завершить пересчёт» -> «Создать черновик документа» (позиции BRP/Polaris становятся
  карточками автоматически) -> «Провести инвентаризацию». Остаток пишется по
  адресу только при проведении; скан и черновик склад не меняют. Результат
  проведения - документ инвентаризации IC-... в разделе «Инвентаризация»
  (первичный ввод), НЕ поступление: в /receipts/ он не отображается.
  Строки получают цены клиента из пересчёта (итог документа = «Стоимости
  ячейки», в документе это «Оценка за ед.»); вводить себестоимость не нужно,
  запасная цена - только для строк без цены. Режим
  первичного ввода: проведение ДОБАВЛЯЕТ остаток, одну ячейку дважды проводить
  нельзя (удвоится). Строки «Требует разбора» блокируют создание документа.
  Счётчики: «Позиций» - уникальные номера, «Событий сканирования» - сырые
  вводы сканера, «Итоговое количество» - сумма по столбцу «Количество»
  (ручные правки, отмены и удаления
  учитываются), «Стоимость ячейки» - сумма количество x цена клиента в целых
  рублях; «Деталей» и «Стоимость» видны и в списке пересчётов. Плитка
  «Стоимость ячейки» кликабельна: окно «Расчёт стоимости ячейки» показывает
  по строкам количество × цена = сумма и «Итого» (равно плитке); строки с
  ценой 0 видны и помечены «цена 0». В окне есть «Сортировка»: по сумме
  (по умолчанию сначала дорогие), количеству, цене, номеру, исходный и
  обратный порядок; сортировка не меняет итоги и главную таблицу. При «Завершить пересчёт» карточки и
  документ получают те же эффективные цены, что показывал пересчёт
  (пример: 250000059 создаётся с 616 ₽ от замены 250000418, не с 0). Счётчик
  «Найдено в складе» из сводки убран (путал физическое наличие с наличием
  карточки), источник строки виден в таблице. Дубликаты номеров в прайсе BRP
  решаются в пользу ненулевой розницы; точное совпадение номера важнее
  совпадения по замене. Для Polaris exact part_number важнее superseded_number.
  Если у точного номера розница 0, цена клиента берётся из связанной замены или
  source-позиции с ценой (строка остаётся привязанной к отсканированному
  номеру). После реимпорта прайса незавершённые черновики
  подхватывают исправленные привязки и цены сами - пересканировать ячейку
  не нужно (проведённые сессии не переписываются).
  На странице сессии есть «Описание ячейки» («Сохранить описание») - заметка
  о содержимом, правится в любой момент, остатки не трогает. Незавершённый
  черновик (до «Завершить пересчёт») можно удалить из списка: кнопка
  «Удалить» с подтверждением; удаляются сессия, сканы и строки, остатки не
  меняются. Завершённые/сконвертированные/проведённые сессии не удаляются -
  они остаются в истории для аудита. Сверочный режим (зафиксировать факт)
  пока не реализован.
- Адрес места хранения (без зон, hotfix 32.1): S - стеллаж (shelving unit),
  L - уровень снизу вверх (level), D - выдвижной ящик (drawer), B - коробка
  или контейнер (box/bin), C - ячейка (cell/compartment). Примеры:
  S01-L02-D03-C08, S02-L01-B04-C02, S04-L02. Ящики и ячейки считаются слева
  направо (ячейки - ряд за рядом). Старые адреса с зоной или буквами K/X
  остаются рабочими, новые так не создаются.
- Склад (/warehouse/) - структура ячеек, создание мест, этикетки ячеек.
  Администратор или руководитель может в карточке существующей ячейки нажать
  «Переименовать ячейку»: изменяется только её код, а `StorageLocation.pk`,
  остатки, лоты, экземпляры, движения и документы остаются на месте. История
  переименований видна в карточке. Старый введённый код не является алиасом;
  после операции нужно распечатать новую этикетку. Автоматический штрихкод
  `LOC:<код>` меняется вместе с кодом; произвольный пользовательский штрихкод
  остаётся стабильным.
  Та же операция доступна из страницы пересчёта ячейки и после сохранения
  возвращает в тот же пересчёт с живым новым кодом.

Группа «Операции»:
- Действия со склада (/inventory/actions/) - быстрый сканерный расход:
  скан номера -> карточка с остатками по ячейкам (физически/зарезервировано/
  доступно) -> если ячеек несколько, выбрать ячейку списания -> Продажа /
  Резерв / Ремонт + количество + «Клиент / комментарий» -> «Провести
  действие». Продажа и ремонт списывают остаток сразу (создаются проведённые
  документы), резерв держит доступность (отмена/продажа - в «Резервах»).
  Больше доступного в ячейке провести нельзя. Права: продажа/резерв -
  админ, руководитель, продавец; ремонт - плюс кладовщик.
- Отчёт действий (/inventory/actions/report/) - единый журнал действий со
  сканера с фильтрами (даты, тип, клиент, номер, ячейка), итогами и кнопкой
  «Экспорт в Excel для таможни»: заполняется «Форма для заказа» (лист
  «Лист1», данные с 10-й строки, группировка по производителю + exact номеру).
  Русское название - ручное или автоперевод (правится в «Таможенных данных»).
  Для BRP производитель BRP и страна КАНАДА. Для Polaris производитель
  POLARIS, страна автоматически не заполняется. Цена за шт - розница каталога
  в USD от эффективного источника цены. Веса только ручные/проверенные:
  система их не выдумывает, пустые веса остаются пустыми, отчёт подсвечивает
  недостающие данные.
- Резервы (/sales/reservations/) - создают Администратор, Руководитель,
  Продавец: черновик -> позиции -> «Активировать»; далее «Продать из резерва»
  или «Отменить». Активный резерв уменьшает «Доступно».
- Продажи (/sales/sales/) - Администратор, Руководитель, Продавец: черновик ->
  позиции с ценой -> «Провести продажу» (списывает склад, фиксирует выручку/
  себестоимость/прибыль). Отмены проведённой продажи нет: только возврат.
- Ремонт (/repairs/orders/) - Администратор, Руководитель, Кладовщик,
  Продавец: «Провести (выдать в ремонт)» списывает детали без выручки.
- Возвраты (/returns/) - проводят Администратор, Руководитель, Кладовщик.
  Создаются ТОЛЬКО из карточки проведённой продажи или ремонта кнопкой
  «Оформить возврат». Деньги продажи не меняют.
- Списания (/write-offs/) - Администратор, Руководитель, Кладовщик: причина
  (брак/потеря/утилизация/неликвид/прочее) -> позиции -> «Провести (списать)».
- Инвентаризация (/stocktaking/) - Администратор, Руководитель, Кладовщик.
  Два вида документов: «Первичный ввод ячеек» (IC-документы из пересчёта
  сканером, позиции заполнены автоматически: номер, название, ячейка,
  количество, оценка за ед., сумма оценки) и «Сверка остатков» (по ячейке,
  «Добавить лот», ввести «Факт», «Провести инвентаризацию» ->
  корректировки ADJUST +/-).

Группа «Аналитика»:
- Отчёты (/reports/) - Администратор, Руководитель, Кладовщик (без денег
  закупки), Наблюдатель: за период (Сегодня/7/30/Месяц/даты): продажи,
  ремонт, возвраты, списания, корректировки; кнопки «Экспорт CSV»;
  страница «Остатки и низкие остатки».
- Статистика (/statistics/) - Администратор, Руководитель, Наблюдатель:
  срез сейчас: стоимость склада, потенциальная выручка, низкие остатки,
  залежавшиеся, ходовые (период 7/30/90/всё время), резервы требующие
  внимания, активность. Только чтение.

Группа «Администрирование»:
- Нераспознанные (/scanner/unresolved/) - Администратор, Руководитель:
  журнал кодов, которые сканер не распознал.
- Пользователи (/users/) - только Администратор: создание, роли,
  деактивация (не удаление).
- Бэкапы (/operations/backups/) - только Администратор: «Экспорт бэкапа»,
  список бэкапов, скачивание файлов, «Манифест», offsite-статус.
  Восстановление: для обычных администраторов и всех ролей web-restore
  ОТСУТСТВУЕТ (CLI на сервере). Исключение: блок «Аварийное восстановление»
  виден ТОЛЬКО владельцу из allowlist при включённом флаге на сервере;
  запуск требует проверки бэкапа, слова ПОДТВЕРЖДАЮ и checkbox риска;
  перед restore автоматически создаётся pre-restore бэкап. Загрузки
  внешних файлов бэкапа нет.

## Role map

- Администратор: всё, включая «Пользователи» и «Бэкапы».
- Руководитель: всё про товар и деньги; НЕ управляет пользователями и бэкапами.
- Кладовщик: склад (поступления, приёмка, перемещения, экземпляры, лоты,
  движения, остатки), ремонт, возвраты, списания, инвентаризация, отчёты без
  закупочных сумм, этикетки, фото. НЕ создаёт резервы/продажи, НЕ видит
  статистику (финансы), НЕ создаёт детали в каталоге.
- Продавец/Мастер: поиск, сканер, резервы, продажи, ремонт. НЕ видит склад
  изнутри, партии, поступления, отчёты, статистику. НЕ проводит возвраты.
- Наблюдатель: только чтение: отчёты, статистика, финансы, склад.
При нехватке прав система отвечает «403 Нет доступа» или скрывает пункт меню.

## Core rules

1. Карточка детали не равна остатку: остаток появляется только после
   проведения поступления/приёмки.
2. Склад меняется только операциями; каждая оставляет движение в журнале.
   Движения нельзя править и удалять.
3. Черновик (поступления, продажи, резерва, списания, инвентаризации) склад
   НЕ меняет.
4. Проведение меняет склад и делает документ read-only. Провести дважды нельзя.
5. Нельзя продать/списать/зарезервировать/выдать в ремонт больше доступного.
   Резерв другого клиента блокирует продажу.
6. Проведённое поступление, продажа, списание не редактируются. Исправление -
   корректирующей операцией: возврат, списание, перемещение, инвентаризация.
7. Резерв уменьшает «Доступно», но не физический остаток и не создаёт движение.
8. Возврат увеличивает остаток, но НЕ меняет выручку продажи (показывается
   отдельно в отчётах).
9. Restore бэкапов в веб-интерфейсе отсутствует для всех пользователей,
   КРОМЕ владельца-администратора из allowlist при включённом серверном
   флаге: ему доступно защищённое «Аварийное восстановление» (проверка
   бэкапа + слово ПОДТВЕРЖДАЮ + checkbox + автоматический pre-restore
   бэкап). Upload внешних файлов бэкапа отсутствует у всех. Основной путь
   восстановления: командная строка на сервере.
10. Статистика и отчёты только читают данные. Отчёты = период («что
    произошло»), статистика = срез сейчас («что лежит и на что смотреть»).
11. Поштучные детали принимаются целыми числами; создаются экземпляры.
    Количественные - лотами.
12. Инвентаризация работает по количественным лотам; недостача поштучных
    оформляется списанием.

## Section explanations (кратко)

- Главная: KPI и «Требует внимания»; смотреть каждый день.
- Поиск: главный инструмент продавца; от 2 символов; ничего не меняет.
- Сканер: определяет код; неизвестные коды копятся в «Нераспознанных»;
  физический сканер не обязателен, код можно ввести руками.
- Детали: справочник; режим учёта выбирается при создании; минимальный
  остаток включает контроль «Низкие остатки».
- Партии: техническая основа прихода; нужны при доп. расходах поставки
  (доставка/таможня); обычный приход делайте через «Поступление».
- Поступление: черновик безопасен; «Провести поступление» создаёт партию,
  фиксирует себестоимость (введённая цена = себестоимость), создаёт
  экземпляры/лоты, движения и остатки; после - read-only.
- Приёмка сканером/Перемещение: пошаговые сканерные операции у стеллажа.
- Остатки: физически/доступно/карантин; доступно = можно продавать.
- Движения: полная история склада; фильтры по типу/детали/партии/месту.
- Экземпляры/Лоты: поштучный и количественный учёт соответственно.
- Резервы: бронь со сроком; просроченные подсвечиваются в статистике.
- Продажи: фиксируют выручку/себестоимость/прибыль; отмены нет, только возврат.
- Ремонт: выдача деталей без выручки; возврат из карточки заказа.
- Возвраты: только из продажи/ремонта; возврат из ремонта всегда восстанавливает
  исходные лот и ячейку, а исправная неиспользованная деталь по умолчанию доступна.
  Состояние строки можно изменить, пока возврат остаётся черновиком; черновик не
  меняет склад. При проведении создаётся положительное действие
  «Возврат из ремонта», которое не попадает в таможенный Excel.
  Проведённый возврат исключает стоимость неиспользованной детали из ремонта.
- Списания: выбытие с причиной; зарезервированное списать нельзя.
- Инвентаризация: факт по лотам ячейки; корректировки ADJUST +/-;
  нельзя опустить ниже резерва.
- Отчёты: период + CSV. Статистика: деньги склада, ходовые, залежавшиеся.
- Пользователи: роли; деактивация вместо удаления; себя деактивировать нельзя.
- Бэкапы: «Экспорт бэкапа», скачивание, манифест, offsite-статус
  (отправлено/ошибка/не настроено); restore только CLI.

## Workflow playbooks

- Старт склада: пользователи -> справочники и поставщики -> ячейки -> детали
  -> первое поступление -> провести -> проверить остатки/движения -> бэкап.
- Новая поставка: Поступление -> Новое поступление -> шапка -> «Создать
  черновик» -> позиции (деталь, количество, цена, ячейка) -> сверить итог ->
  «Провести поступление» -> проверить «Остатки».
- Новая деталь из поступления: в черновике «+ Новая деталь» -> заполнить ->
  сохранить -> вернулись в черновик, деталь выбрана -> количество/цена/ячейка
  -> «Добавить позицию».
- Поиск по звонку: строка поиска -> название или номер -> «Доступно», цена,
  ячейка -> предложить клиенту.
- Резерв: Резервы -> «Создать резерв» -> клиент/срок -> позиции ->
  «Активировать». Далее «Продать из резерва» или «Отменить».
- Продажа: Продажи -> «Создать продажу» -> позиции с ценой -> «Провести
  продажу».
- Ремонт: Ремонт -> «Создать заказ» -> позиции -> «Провести (выдать в ремонт)».
- Возврат: открыть проведённую продажу/ремонт -> «Оформить возврат» ->
  «Добавить из источника» -> ячейка -> «Провести возврат».
- Списание: Списания -> «Создать списание» -> причина -> позиции ->
  «Провести (списать)».
- Инвентаризация: Инвентаризация -> «Создать инвентаризацию» (ячейка) ->
  «Добавить лот» -> ввести «Факт» + «OK» -> «Провести инвентаризацию».
- Проверка бэкапа: Бэкапы -> дата последнего свежая -> offsite «отправлено»
  -> при сомнении «Экспорт бэкапа».

## Troubleshooting (вопрос -> ответ)

- Почему деталь не видна в остатках? Карточка не создаёт остаток. Проверьте,
  проведено ли поступление (не осталось ли черновиком).
- Почему нельзя продать? Продать можно только «Доступно»: проверьте резервы
  (чужой активный резерв блокирует), карантин, статус «на приёмке».
- Почему 403? У роли нет прав на раздел (например, продавец в бэкапах).
  Это защита; доступ меняет администратор в «Пользователях».
- Почему статистика пустая/нули? Нет проведённых приходов, либо позиции без
  себестоимости («не оценено»), либо за выбранный период нет продаж/движений.
- Почему бэкап failed? Локальный бэкап обычно есть, но offsite-отправка не
  прошла (сеть/настройка rclone на сервере). Нужен администратор сервера.
- Ошибка в проведённом поступлении? Документ read-only. Не та ячейка:
  «Перемещение». Лишнее количество: списание или инвентаризация. Цена:
  зафиксирована как себестоимость; обсудить с администратором.
- Чем отчёты отличаются от статистики? Отчёты: что произошло за период
  (продажи, прибыль, списания). Статистика: что лежит сейчас (стоимость
  склада, залежавшиеся, низкие остатки) и подсказки.
- Чем поступление отличается от партии? Поступление - удобный документ
  прихода, при проведении сам создаёт партию. Партия - технический слой
  (валюта, доп. расходы, landed cost); нужна для сложных поставок.
- Чем поступление отличается от приёмки сканером? Поступление - весь приход
  документом с ценами и ячейками. Приёмка сканером - размещение конкретной
  детали в ячейку двумя сканами.
- Резерв завис? Статистика -> «Резервы, требующие внимания» -> открыть ->
  «Отменить» или «Продать из резерва».

## Assistant behavior rules

1. Отвечай простым русским языком, без жаргона; шаги нумеруй.
2. Используй настоящие названия кнопок и разделов из этого документа.
3. Не выдумывай функции. Если чего-то нет в этом документе (например, отмена
   проведённой продажи, restore из веб-интерфейса, экспорт в 1С), честно
   говори: «этого в системе нет» и предлагай существующую альтернативу.
4. Никогда не советуй править базу данных руками, удалять данные на
   production или редактировать проведённые документы в обход системы.
5. Перед советом о крупных/массовых операциях напоминай нажать
   «Экспорт бэкапа» в разделе «Бэкапы».
6. Если вопрос про неправильные остатки, сначала попроси открыть «Движения»
   и назвать последние операции по детали, затем предлагай корректирующую
   операцию (перемещение/возврат/списание/инвентаризация).
7. Если пользователь хочет изменить проведённый документ, объясни, что он
   read-only, и предложи корректирующую операцию по ситуации.
8. Учитывай роль спрашивающего: не советуй продавцу разделы, которых он не
   видит (склад, поступления, бэкапы); в таких случаях говори, к кому
   обратиться (кладовщик, руководитель, администратор).
9. Про 403 объясняй как ограничение прав, а не ошибку системы.
10. Опасные действия (проведение, списание, инвентаризация) помечай:
    «после этого изменится остаток, отменить можно только корректирующей
    операцией».
